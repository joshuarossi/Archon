The Draft Coach Panel brings a private, AI-assisted drafting workspace into JointChatView as a collapsible side panel. It lets a party iterate on a message with AI help before deciding whether to send it — and nothing reaches the joint chat until the user clicks Send. This task exists in Phase 2 because it depends on the joint chat foundation (T16) being stable and is the primary surface through which the "Draft Coach" user story (US-12, FR-08) is delivered. It also contributes one of the five privacy-security Playwright tests (PRD §7.1 launch criterion #2, Test 3) that must pass before the Phase 2 completion gate closes.

## Acceptance Criteria

- JointChatView has a 'Draft with Coach' button that opens DraftCoachPanel as a side panel without navigating away.
- DraftCoachPanel shows its own ChatWindow for the draft conversation; messages are invisible to the other party and to the joint-chat Coach.
- Sending a message in the Draft Coach panel calls `draftCoach/sendMessage` and schedules `draftCoach/generateResponse`.
- A 'Generate Draft' button posts a canonical user message ('Generate a draft for me') to the draft conversation.
- When `draftSessions.finalDraft` is set, `DraftReadyCard` replaces the message input, showing: the polished draft text, Send, Edit, Continue Refining, and Discard buttons.
- Clicking Send calls `draftCoach/sendFinalDraft` which internally calls `jointChat/sendUserMessage`, marks the session SENT, closes the panel, and appends the message to the joint chat.
- HARD RULE: No message is ever sent to the joint chat without an explicit Send click; generating a draft does not auto-send.
- Clicking Discard calls `draftCoach/discardSession`, marks session DISCARDED, closes the panel; the draft never appears in joint chat.
- Draft Coach panel is private: its messages are NOT returned by `jointChat/messages` and NOT visible in any query accessible to the other party.
- Playwright `draft-coach.spec.ts`: open panel, iterate 2 turns, click Generate Draft, verify joint chat is NOT updated yet, click Send, verify message appears in joint chat.
- Playwright `draft-coach.spec.ts`: click Discard, verify draft does not appear in joint chat.
- Playwright (`privacy.spec.ts`, PRD §7.1 launch criterion #2, Test 3 — draft message isolation): user B calls `jointChat/messages` for a case where user A has an active draft session; asserts that draftMessages table content is absent from the result (only jointMessages are returned, confirming draft isolation from the joint channel).

## Implementation Notes

**Panel layout and shell.** On desktop the panel slides in from the right at 420px wide, full viewport height, with `--shadow-3`. On mobile (< 768px) it becomes a full-screen bottom sheet using shadcn `Sheet` with `side="bottom"`. The panel is an `<aside>` landmark. The first visual element is a persistent privacy header: Lucide `Lock` icon + "Jordan can't see what you're discussing here." using `--private-tint` background. This header is never dismissable and must not be `aria-hidden`.

**Session lifecycle.** `draftCoach/startSession` is called when the panel first opens (idempotent — if a PENDING session already exists for this user+case it returns its ID rather than creating a duplicate). The session transitions: `PENDING → SENT` via `sendFinalDraft` or `PENDING → DISCARDED` via `discardSession`. Both mutations close the panel client-side and are idempotent (calling twice on an already-terminal session is a no-op, not an error).

**DraftReadyCard state.** When the reactive query for `draftSessions.finalDraft` is non-null, `DraftReadyCard` replaces the `MessageInput`. The four buttons have these exact outcomes: **Send** → `sendFinalDraft` → inserts into joint chat → closes panel; **Edit** → clears `finalDraft` (client-side, restores input with draft text pre-filled); **Continue Refining** → clears `finalDraft` (client-side, keeps session PENDING); **Discard** → `discardSession` → closes panel, nothing sent. Only **Send** ever touches `jointChat/sendUserMessage`.

**Privacy enforcement at the query layer.** `draftCoach/messages` query requires `requireAuth` and verifies the caller owns the session's `userId`. The `jointChat/messages` query is already scoped to `jointMessages` rows only; it must never join or include `draftMessages`. The Playwright privacy test (Test 3 in `privacy.spec.ts`) verifies this by having user B call `jointChat/messages` while user A has an active draft session — the result must contain zero `draftMessages` rows.

**Scheduling generateResponse.** `draftCoach/sendMessage` inserts the user message row and calls `ctx.scheduler.runAfter(0, internal.draftCoach.generateResponse, { sessionId, caseId })`. The AI action (T19) is a separate task; T18 only needs to schedule it correctly and ensure the session/message schema fields it writes are correct.

**Design tokens.** Panel background uses `--bg-surface`. Privacy header uses `--private-tint`. Draft Coach AI messages use `--accent-subtle` (not `--coach-subtle` — Draft Coach is not the joint-chat Coach). Buttons in `DraftReadyCard`: Send is `variant="primary"`, Discard is `variant="danger"`, Edit and Continue Refining are `variant="secondary"`.

**Focus management.** When the panel opens, focus moves to the textarea inside the panel. When the panel closes (Send or Discard), focus returns to the 'Draft with Coach' button in JointChatView. Esc closes the panel without sending.

**Queries used**

| Symbol | Called from |
|--------|-------------|
| `api.draftCoach.startSession` | DraftCoachPanel (on open) |
| `api.draftCoach.sendMessage` | DraftCoachPanel MessageInput submit handler |
| `api.draftCoach.sendFinalDraft` | DraftReadyCard Send button |
| `api.draftCoach.discardSession` | DraftReadyCard Discard button + panel Esc handler |
| `api.draftCoach.messages` | DraftCoachPanel ChatWindow (reactive useQuery) |
| `internal.jointChat.sendUserMessage` | Called internally inside `sendFinalDraft` mutation |

**Invariants**

- `draftCoach/messages` query never returns rows to a caller who is not the session owner; throws `FORBIDDEN`.
- `jointChat/messages` query never returns rows from the `draftMessages` table; only `jointMessages` rows are in scope.
- `sendFinalDraft` is idempotent on already-SENT sessions (no error, no duplicate joint message).
- `discardSession` is idempotent on already-DISCARDED sessions.
- No code path inserts a row into `jointMessages` as a side effect of draft generation or the 'Generate Draft' button click — only `sendFinalDraft` may call `jointChat/sendUserMessage`.
- `startSession` creates at most one PENDING session per user per case; a second call returns the existing session ID.

**Non-goals**

- T18 does NOT implement the AI generation logic inside `generateResponse` (that is T19).
- T18 does NOT implement cost-cap handling inside `generateResponse`.
- T18 does NOT implement streaming tokens or STREAMING/COMPLETE status transitions on `draftMessages` rows — T19 owns that.
- T18 does NOT add new columns to `draftSessions` or `draftMessages` beyond what the schema (T1) already defines.
- T18 does NOT implement edit-in-place of the draft text inline (Edit button clears `finalDraft` and pre-fills the textarea; it does not render an editable field over the card).
