This task delivers the admin template management system for Conflict Coach, covering the full lifecycle of coaching templates: creation, immutable versioning, archival, and audit logging. Phase 3's goal is to make the app production-ready for an admin operator, and templates are the primary lever an admin uses to tune Coach and Draft Coach behavior per conflict category. Without this surface, the system can only run on the seed defaults planted in T4. The task also enforces the invariant that cases pin to a specific template version at creation time and are never affected by later edits or archival — which is critical for auditability in a mediation context.

## Acceptance Criteria

- All `/admin/*` routes return FORBIDDEN to any user whose `users.role != 'ADMIN'`; the server-side check in every admin mutation is present (no client-side flag trusted).
- TemplatesListPage lists all templates including archived, shows current version number, and links to TemplateEditPage.
- TemplateEditPage shows: category, name, all versions with their content; has a form to publish a new version with globalGuidance, coachInstructions, draftCoachInstructions fields.
- Publishing a new version creates a new `templateVersions` row with `version` incremented, `publishedAt` and `publishedByUserId` set; the previous version row is unchanged.
- Archiving a template sets `archivedAt`; archived templates no longer appear in the category picker on NewCasePage.
- A case that was created when a template was at v1 continues to use v1 even after v2 is published and the template is archived.
- Every `create`, `publishNewVersion`, and `archive` action writes a `auditLog` row with `actorUserId`, `action` string (e.g., 'TEMPLATE_PUBLISHED'), `targetType: 'templateVersion'`, `targetId`, and `createdAt`.
- Playwright `admin-templates.spec.ts`: admin creates template, publishes v2, verifies pinned case still resolves v1; archives template, verifies it disappears from category picker.

## Implementation Notes

**Admin role guard pattern.** Every admin Convex function must start with `requireAuth(ctx)` (from `convex/lib/errors.ts`) to obtain the user row, then check `user.role === 'ADMIN'` and throw `FORBIDDEN` if not. This is the same pattern as all other functions — the role check is additive. Never trust a React prop or query param to gate admin access; the client is untrusted.

**Template versioning is append-only.** The `templateVersions` table is never mutated after row creation — no `UPDATE` on existing rows, no soft-deletes of versions. When an admin publishes a new version, a new row is inserted with `version: prevMax + 1`. Cases carry a `templateVersionId` foreign key written at creation time and sealed thereafter. This means `archive` touches only the `templates` row (sets `archivedAt`), not any version rows, and not any `cases` rows. The case's coach behavior is determined forever by whatever version was current at case creation.

**Determining "current version" for display.** TemplatesListPage needs to show the highest-numbered published version for each template. Index on `templateVersions.templateId` + `version` or use a query that sorts by `version` descending and takes the first result. Convex requires explicit indexes: ensure `by_template_version` (compound: `templateId`, `version`) is declared in `convex/schema.ts` (T1's schema).

**Category picker filtering.** `NewCasePage` queries active (non-archived) templates to populate the category select. This query must filter out templates where `archivedAt` is set. Add a `by_category_active` index (or query the `by_archivedAt` index) to avoid full-table scans at runtime.

**Audit log writes.** The `auditLog` table stores `actorUserId`, `action`, `targetType`, `targetId`, and `createdAt` per the schema from T1. Each admin mutation writes the audit row as part of its own transaction (mutations are transactional in Convex). Use a shared helper `convex/lib/auditLog.ts` → `writeAuditEntry(ctx, ...)` to avoid duplicating the insert across three mutations.

**UI conventions.** Both admin pages are behind `/admin/*` routes already scaffolded in T3. They use shadcn `Card`, `Table` (or a simple `<table>` styled with Tailwind), `Dialog` for the publish-new-version form, and `Button` variant `danger` for Archive actions. Archive requires a confirmation `Dialog` per the `states-and-feedback` design rule — never `window.confirm()`. Use the `--text-secondary` token for version metadata (published date, author) and `--text-primary` for template names. Archived templates in the list get a muted `opacity-50` treatment and an "Archived" badge using `--bg-surface-subtle` background.

**Tone.** Archive CTA reads "Archive template" (not "Delete"). Confirmation dialog reads: "This will hide the template from the category picker. Existing cases using this template are unaffected." The admin surface follows the same voice rules as the rest of the app — no exclamation marks, no "Are you sure?".

**Queries used**

- `api.admin.templates.create` — called from `TemplateEditPage` "Create" form
- `api.admin.templates.publishNewVersion` — called from `TemplateEditPage` "Publish new version" form
- `api.admin.templates.archive` — called from `TemplateEditPage` "Archive template" button
- `api.admin.templates.list` — subscribed by `TemplatesListPage` and `TemplateEditPage` via `useQuery`
- `api.templates.activeForCategory` — subscribed by `NewCasePage` category picker
- `internal.lib.auditLog.writeAuditEntry` — called internally from each admin mutation

**Invariants**

- A user with `role !== 'ADMIN'` calling any `admin.templates.*` mutation or query receives a `FORBIDDEN` `ConvexError`; the check happens inside the Convex function handler, not on the client.
- `templateVersions` rows are never mutated after creation — `publishNewVersion` only inserts; it does not patch existing rows.
- `cases.templateVersionId` is written once at case creation (`cases/create`, T6) and never updated by any template operation.
- `archive` only sets `archivedAt` on the `templates` row; no cascade to `templateVersions` or `cases`.
- `auditLog` rows are inserted within the same mutation transaction as the state change they record — they cannot exist without the corresponding state change having committed.
- `version` on a new `templateVersions` row is strictly `MAX(existing versions for templateId) + 1`; there are no gaps and no reuse.

**Non-goals**

- No admin UI for promoting a user to ADMIN role (manual Convex Dashboard action per planning assumptions).
- No deletion of `templateVersions` rows (append-only; archival is the only soft-state change).
- No template import/export.
- No bulk-archival or bulk-version operations.
- Email notifications on template publish are P2 deferred features (US-16 through US-19).
