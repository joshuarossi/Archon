# Convex Auth configuration (magic link + Google OAuth)

This task wires the authentication layer that every subsequent Phase 0 and Phase 1 task depends on. The project uses Convex Auth to manage sessions and identity; this task configures two providers (email magic link and Google OAuth), establishes the `users` table upsert on first login, and documents all required environment variables in `.env.example`. Getting this right in Phase 0 ensures that T3's route guards, T5's login UI, and every Convex function's `requireAuth(ctx)` call have a stable identity source to build against.

## Acceptance Criteria

- Magic link auth provider is configured in `convex/auth.ts` and can deliver a sign-in link (tested locally with a real email or the Convex Auth test helper).
- Google OAuth provider is configured; `GOOGLE_OAUTH_CLIENT_ID` and `GOOGLE_OAUTH_CLIENT_SECRET` env vars are required and documented.
- On first login (either provider), a `users` row is upserted keyed by email with `role: 'USER'` and `createdAt` set.
- Subsequent logins with the same email return the existing `users` row without creating a duplicate.
- `.env.example` lists all required env vars: `ANTHROPIC_API_KEY`, `CLAUDE_MOCK`, `GOOGLE_OAUTH_CLIENT_ID`, `GOOGLE_OAUTH_CLIENT_SECRET`, `SITE_URL`, `MAGIC_LINK_EMAIL_FROM`, `VITE_CONVEX_URL`, `VITE_SITE_URL`.
- `ctx.auth.getUserIdentity()` returns a non-null identity in a Convex function when called from an authenticated session.

## Implementation Notes

**Convex Auth setup.** Install `@convex-dev/auth` and its peer `@auth/core`. The entry point is `convex/auth.ts`, which calls `convexAuth({ providers: [...] })` and re-exports `{ auth, signIn, signOut, store }`. These exports are consumed by Convex's internal plumbing — they are not called directly from application code.

**Magic link provider.** Use `@convex-dev/auth/providers/Email` (or the equivalent `ResendOTP` / `Resend` provider depending on the package version). The `MAGIC_LINK_EMAIL_FROM` and `SITE_URL` env vars control the from address and the redirect base URL. In local dev the Convex Auth test helper can intercept the link without sending a real email.

**Google OAuth provider.** Use `@convex-dev/auth/providers/Google`. The provider requires `GOOGLE_OAUTH_CLIENT_ID` and `GOOGLE_OAUTH_CLIENT_SECRET` set in the Convex deployment environment (`npx convex env set ...`), not in `.env.local` (they are server-side only).

**`users` upsert on login.** Convex Auth fires a `createOrUpdateUser` callback (inside `convex/auth.ts`) every time a session is established. Inside this callback, query the `users` table by email (using the `by_email` index defined in T1's schema). If no row exists, insert with `role: 'USER'` and `createdAt: Date.now()`. If a row exists, return its `_id` unchanged — do not update fields. This guarantees idempotence and prevents duplicates.

**Key Convex Auth caveat (beta).** Pin the `@convex-dev/auth` version. The OAuth redirect flow navigates away from the app and back; any state in memory or `location.state` is lost. Downstream invite-token handling (T7) must store the token in `localStorage` before triggering `signIn("google")` — note this for T5/T7 implementers.

**`requireAuth` interaction.** The `requireAuth(ctx)` helper (defined in T1's `convex/lib/errors.ts`) calls `ctx.auth.getUserIdentity()` and throws `UNAUTHENTICATED` if null. After this task, every Convex function should be able to call `requireAuth` successfully when invoked from an authenticated client.

**Environment variables.** All server-side secrets belong in the Convex deployment environment (not `.env.local`). The `VITE_CONVEX_URL` and `VITE_SITE_URL` vars are client-visible (prefixed with `VITE_`) and go in `.env.local`. Document all of them in `.env.example` with placeholder values and one-line comments.

## Test-Gen Brief

### Contract

**Files**
- `convex/auth.ts` — provider configuration, `createOrUpdateUser` callback, `{ auth, signIn, signOut, store }` exports
- `convex/schema.ts` — `users` table definition (owned by T1; do not modify)
- `convex/lib/errors.ts` — `requireAuth(ctx)` helper (owned by T1; consumed here for verification)
- `.env.example` — documentation of all required env vars
- `convex/auth.test.ts` — Vitest unit tests for the upsert logic

**Exports**
- `convex/auth.ts`: named exports `auth`, `signIn`, `signOut`, `store` — values (Convex Auth framework internals)
- `convex/auth.ts`: `createOrUpdateUser` callback — not independently exported; exercised via `convex-test`

**Signatures**
```typescript
// convex/auth.ts — conceptual shape; exact API matches @convex-dev/auth version
export const { auth, signIn, signOut, store } = convexAuth({
  providers: [
    ResendOTP({ /* magic link config */ }),
    Google,
  ],
  callbacks: {
    async createOrUpdateUser(
      ctx: MutationCtx,
      args: { existingUserId: Id<"users"> | null; profile: { email: string; name?: string } }
    ): Promise<Id<"users">>,
  },
});
```

**Queries used**
- `ctx.db.query("users").withIndex("by_email", q => q.eq("email", email)).first()` — find existing user by email
- `ctx.db.insert("users", { email, role: "USER", createdAt: Date.now() })` — create new user row
- `ctx.auth.getUserIdentity()` — called in `requireAuth` and in test assertions

**Invariants**
- For any given email address, the `users` table contains at most one row at all times.
- Every row created by this callback has `role: 'USER'` (never `'ADMIN'`).
- `createdAt` is set exactly once, at insert time; subsequent logins do not overwrite it.
- `ctx.auth.getUserIdentity()` is always non-null inside any Convex function invoked from an authenticated session.

**Non-goals**
- Admin role assignment — admin promotion is a manual Convex Dashboard action in v1.
- Password-based authentication — no password field, no credential hashing.
- Clerk integration — swap plan exists but is out of scope unless Convex Auth blocks Phase 1.
- Session expiry enforcement beyond Convex Auth defaults — no custom TTL logic.
- Client-side role gating — role checks belong inside Convex mutations/queries (enforced by `requireAuth` + explicit `role` checks in later tasks).
