# 'Mark private coaching complete' flow + BOTH_PRIVATE_COACHING/READY_FOR_JOINT transitions

This task wires the gate that moves both parties out of private coaching and into the synthesis pipeline. Once a user finishes their private conversations with the AI coach, they signal readiness by clicking a button; when the second party does the same, the system automatically schedules AI synthesis and eventually transitions the case to `READY_FOR_JOINT`. The feature lives at the intersection of the `PrivateCoachingView` UI, the `privateCoaching/markComplete` mutation, and the state machine built in T1 — it is the bridge between Phase 1's coaching work and Phase 2's joint session.

## Acceptance Criteria

- PrivateCoachingView shows a 'I'm ready for the joint session' button that is enabled only when the user has sent at least one message in private coaching.
- Calling `markComplete` when already completed is idempotent (no error, no second timestamp set).
- After one party marks complete, the other party sees a status indicator: 'Your partner has completed their private coaching' (without any content revelation).
- After both parties mark complete, `synthesis/generate` is scheduled via `ctx.scheduler.runAfter(0, ...)` from within the mutation.
- The case status transitions from `BOTH_PRIVATE_COACHING` to `READY_FOR_JOINT` only after synthesis is written (within the synthesis action, not the markComplete mutation).
- State machine enforces that markComplete can only be called when the case is in `DRAFT_PRIVATE_COACHING` or `BOTH_PRIVATE_COACHING`; calling it in any other state throws `CONFLICT`.
- Playwright: create solo case, complete private coaching as both parties (toggling between them), verify case status changes to READY_FOR_JOINT on dashboard and both parties' case detail shows awaiting-synthesis state.

## Implementation Notes

The mutation `privateCoaching/markComplete` sets `partyStates.privateCoachingCompletedAt` for the calling user after verifying via `requireAuth`. It must check the case status via `stateMachine.ts` before writing — only `DRAFT_PRIVATE_COACHING` and `BOTH_PRIVATE_COACHING` are valid entry states, throwing `CONFLICT` otherwise. After setting the timestamp, it checks whether both parties now have `privateCoachingCompletedAt` set; if so, it fires `ctx.scheduler.runAfter(0, internal.synthesis.generate, { caseId })`. This pattern is the prescribed Convex idiom: keep the mutation fast and transactional, hand expensive work off to a scheduled action.

The `READY_FOR_JOINT` transition must NOT happen inside `markComplete`. It belongs inside `synthesis/generate` (T14), which calls `stateMachine.transitionCase` after writing both `partyStates.synthesisText` rows. This keeps the state machine consistent: the case is only marked ready once synthesis actually exists.

The button in `PrivateCoachingView` is enabled when `privateMessages.length > 0` for the current user — this can be derived from the existing reactive `privateCoaching/messages` query already wired in T10b. The other-party status indicator requires a lightweight reactive query (e.g., `privateCoaching/partnerStatus`) that returns only `{ partnerHasCompleted: boolean }`, nothing from the partner's private content. This satisfies the privacy requirement from AC3.

Idempotency: if `partyStates.privateCoachingCompletedAt` is already set, `markComplete` returns void without re-setting it or re-scheduling synthesis. Use a Convex `patch` only if the field is null/undefined.

Scheduling is not transactional in Convex: `ctx.scheduler.runAfter()` fires even if a later error rolls back the mutation. Design `synthesis/generate` to be idempotent — checking at start whether synthesis is already written before calling Claude.

Button copy must follow the voice rules: "I'm ready for the joint session" (outcome-oriented). The status indicator for the other party reads "Your partner has completed their private coaching" — never exposes which party or any content. Wrap the status glyph in `aria-label` per the accessibility rules.

**Signatures**
```typescript
// convex/privateCoaching/markComplete.ts
export const markComplete = mutation({
  args: { caseId: v.id('cases') },
  handler: async (ctx: MutationCtx, { caseId }: { caseId: Id<'cases'> }): Promise<void>
});

// convex/privateCoaching/partnerStatus.ts
export const partnerStatus = query({
  args: { caseId: v.id('cases') },
  handler: async (ctx: QueryCtx, { caseId }: { caseId: Id<'cases'> }): Promise<{ partnerHasCompleted: boolean }>
});
```

**Queries used**
- `internal.synthesis.generate` — scheduled via `ctx.scheduler.runAfter(0, internal.synthesis.generate, { caseId })`
- `api.privateCoaching.partnerStatus` — called reactively by `PrivateCoachingView` to show partner completion badge
- `api.privateCoaching.markComplete` — called via `useMutation` in `PrivateCoachingView`

**Invariants**
- `markComplete` is only callable when `case.status` is `DRAFT_PRIVATE_COACHING` or `BOTH_PRIVATE_COACHING`; any other status throws `CONFLICT`.
- `partyStates.privateCoachingCompletedAt` is set at most once; calling `markComplete` when it is already set is a no-op.
- `synthesis/generate` is scheduled exactly once — when both `partyStates.privateCoachingCompletedAt` fields are set; never scheduled a second time if called again.
- `partnerStatus` returns only `{ partnerHasCompleted: boolean }` — no content, no timestamps, no identifying information from the partner's private coaching session.
- The case status remains `BOTH_PRIVATE_COACHING` (or `DRAFT_PRIVATE_COACHING`) after `markComplete` runs; it advances to `READY_FOR_JOINT` only from inside `synthesis/generate`.

**Non-goals**
- This ticket does NOT implement `synthesis/generate` (that is T14).
- This ticket does NOT transition the case to `READY_FOR_JOINT` — that transition is owned by T14.
- This ticket does NOT modify the state machine in `convex/stateMachine.ts` beyond what T1 already delivered; it only calls `transitionCase` as a guard check.
- This ticket does NOT add or change any private message queries.
- This ticket does NOT implement the `ReadyForJointView` (that is T15).
