# Dashboard + case detail routing (US-05, US-15)

Every party needs a reliable entry point into their cases and a routing hub that surfaces the correct phase-specific view without exposing data from other parties. T8 builds these two surfaces for Phase 1: `Dashboard.tsx`, which lists all the user's active and closed cases with a privacy-safe peek at the other party's phase-level status, and `CaseDetail.tsx`, which reads case status and delegates to `PrivateCoachingView`, `ReadyForJointView`, `JointChatView`, or `ClosedCaseView`. Both surfaces depend on T5 (auth) and T6 (case creation) being in place; downstream tasks (T10b, T11, T12, T15, T16) all rely on this routing layer being stable before they can wire their own views.

## Acceptance Criteria

- Dashboard lists all cases where the logged-in user is initiator or invitee, grouped into 'Active' and 'Closed' sections.
- Each case row displays: other party name (or 'Pending' if invitee hasn't joined), category, created date, current status label, and last activity date.
- The `cases/partyStates` query returns `{ self: PartyState, otherPhaseOnly: { hasCompletedPC: boolean } }` and nothing else about the other party.
- Clicking a case row navigates to `/cases/:caseId` which renders the correct subview: PrivateCoachingView for private coaching statuses, ReadyForJointView for READY_FOR_JOINT, JointChatView for JOINT_ACTIVE, ClosedCaseView for closed statuses.
- If a case's status requires the other party to have joined and they haven't, the case detail shows a waiting state with the invite link still accessible.
- Dashboard first meaningful paint occurs in under 1 second (measured via Playwright performance API in CI).
- Attempting to read another user's case detail via direct URL returns NOT_FOUND (or 404-equivalent redirect).

## Implementation Notes

**Convex queries.** `cases/list` is a reactive `query()` that runs two index scans — one on `by_initiator` and one on `by_invitee` — and merges the results. Both indexes must exist in `convex/schema.ts` (T1). Never full-table-scan. `cases/partyStates` is a second reactive query gated by case membership; it deliberately returns only `{ self: PartyState, otherPhaseOnly: { hasCompletedPC: boolean } }` to enforce privacy at the query layer, not the UI layer. `cases/get` is used by `CaseDetail` and must throw `NOT_FOUND` (via `errors.ts`) when the caller is neither initiator nor invitee. All three functions begin with `requireAuth(ctx)`.

**Reactive subscription pattern.** `Dashboard.tsx` calls `useQuery(api.cases.list)` — no `useEffect` sync into `useState`. Convex pushes updates automatically; the component just re-renders. `CaseDetail` subscribes to `useQuery(api.cases.get, { caseId })` and uses the returned status to branch sub-view rendering. While the query is loading (`undefined` return), render the case detail skeleton.

**Sub-view routing in CaseDetail.** The routing table is straightforward:
- `DRAFT_PRIVATE_COACHING` → `PrivateCoachingView` (invitee not yet joined — show waiting banner with invite link)
- `BOTH_PRIVATE_COACHING` → `PrivateCoachingView`
- `READY_FOR_JOINT` → `ReadyForJointView`
- `JOINT_ACTIVE` → `JointChatView`
- `CLOSED_RESOLVED | CLOSED_UNRESOLVED | CLOSED_ABANDONED` → `ClosedCaseView`

In Phase 1 these sub-view components are stubs (T9, T10b, T11, T15, T16 fill them in); `CaseDetail` still must route correctly so downstream tasks can integrate cleanly.

**Status labels and glyphs.** The dashboard case row maps each status to one of the four status glyphs defined in the design system (states-and-feedback rule). Use:
- `●` (green, `--accent`) — "Your turn" (case is in a status where the viewer must act)
- `○` (gray, `--text-tertiary`) — "Waiting on the other party"
- `◐` (amber, `--warning`) — "Ready for joint session" (`READY_FOR_JOINT`)
- `◼` (neutral, `--text-secondary`) — Closed

Each glyph is wrapped in `<span aria-label="[status description]">` — glyphs alone are inaccessible per the accessibility design rule.

**Dashboard layout.** `max-w-[720px] mx-auto`, `px-4 md:px-8`. Two sections: "Active" and "Closed", each with a section heading. Case rows render as shadcn `Card` components with `--shadow-1` at rest. Load state: three skeleton rows using `animate-pulse` on `bg-[color:var(--bg-surface-subtle)]` blocks that match the real row shape. Empty state copy (design system tone-and-copy rule): `"No cases yet. When you're ready to work through something, start a new case."`

**Performance gate (AC6).** Playwright measures `performance.getEntriesByName('first-contentful-paint')` and asserts `< 1000ms`. The query must resolve fast enough to clear this bar — ensure the Convex index scans are tight (no full-table merges, no N+1 lookups per row).

**Auth guard.** `CaseDetail` and `Dashboard` are inside the authenticated route tree established in T3. The Convex `cases/get` server-side NOT_FOUND guard is the hard enforcement; the client-side redirect is a UX shortcut only.

## Test-Gen Brief

### Contract

**Files**
- `src/routes/Dashboard.tsx` — Dashboard page component
- `src/routes/CaseDetail.tsx` — Case detail sub-view router
- `convex/cases/list.ts` — `api.cases.list` reactive query
- `convex/cases/get.ts` — `api.cases.get` query (auth + NOT_FOUND enforcement)
- `convex/cases/partyStates.ts` — `api.cases.partyStates` privacy-constrained query
- `playwright/dashboard.spec.ts` — Playwright specs for AC2, AC5, AC6, AC7
- `playwright/case-routing.spec.ts` — Playwright spec for AC4 (or inline in `solo-full-flow.spec.ts`)

**Exports**
- `api.cases.list` — named Convex query export (via `convex/_generated/api`)
- `api.cases.get` — named Convex query export
- `api.cases.partyStates` — named Convex query export
- `Dashboard` — default React component export from `src/routes/Dashboard.tsx`
- `CaseDetail` — default React component export from `src/routes/CaseDetail.tsx`

**Signatures**
```ts
// convex/cases/list.ts
export const list = query({
  args: {},
  handler: async (ctx): Promise<Doc<"cases">[]>
});

// convex/cases/get.ts
export const get = query({
  args: { caseId: v.id("cases") },
  handler: async (ctx, { caseId }): Promise<Doc<"cases">>
  // throws NOT_FOUND if caller is neither initiatorUserId nor inviteeUserId
});

// convex/cases/partyStates.ts
export const partyStates = query({
  args: { caseId: v.id("cases") },
  handler: async (
    ctx,
    { caseId }
  ): Promise<{
    self: Doc<"partyStates">;
    otherPhaseOnly: { hasCompletedPC: boolean };
  }>
  // throws FORBIDDEN if caller is not a party; NOT_FOUND if case not found
});

// src/routes/CaseDetail.tsx
// Props: none (reads caseId from useParams())
// Renders sub-view based on case.status
export default function CaseDetail(): JSX.Element;
```

**Queries used**
- `api.cases.list` — called in `Dashboard.tsx` via `useQuery`
- `api.cases.get` — called in `CaseDetail.tsx` via `useQuery`
- `api.cases.partyStates` — called wherever per-party phase status is needed

**Invariants**
- `cases/list` never returns cases where the calling user is neither `initiatorUserId` nor `inviteeUserId`.
- `cases/get` throws `NOT_FOUND` (not `FORBIDDEN`) when the caller has no membership — callers must not be able to distinguish whether a case exists from whether they can access it.
- `cases/partyStates.otherPhaseOnly` never contains raw message content, form field values, or synthesisText.
- `CaseDetail` renders the correct sub-view for every possible `case.status` value — an unhandled status value is a compile-time exhaustiveness error, not a runtime blank.

**Non-goals**
- Does not implement `PrivateCoachingView`, `ReadyForJointView`, `JointChatView`, or `ClosedCaseView` beyond stub placeholders (those are T9, T10b, T11, T15, T16).
- Does not implement the solo mode `PartyToggle` (T12).
- Does not build the case creation form (T6).
- Does not paginate the dashboard case list (v1 scope: all cases rendered reactively).
