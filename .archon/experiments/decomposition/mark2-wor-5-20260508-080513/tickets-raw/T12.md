Solo mode lets a single user experience the full Conflict Coach flow from both parties' perspectives — useful for self-reflection, facilitation practice, or developer testing. This task adds the `isSolo` creation path to the `cases/create` mutation, the `PartyToggle` nav component that switches which party's context is active, and the `useSoloActingParty` hook that all phase views use to resolve the correct acting userId from the `?as=` URL param. Without solo mode the E2E `solo-full-flow.spec.ts` suite — used to validate synthesis, joint chat, and Draft Coach in CI — cannot run, making it a prerequisite for Phase 2 integration testing.

## Acceptance Criteria

- Dashboard has a 'New Solo Case' button distinct from 'New Case'; solo cases are visibly flagged with a 'SOLO' badge in the case list and case detail.
- `cases/create` with `isSolo: true` creates both `partyStates` rows bound to the same userId in one transaction; no invite token is created.
- Solo cases show `PartyToggle` in the top nav with 'Viewing as Initiator' and 'Viewing as Invitee' options.
- Toggling switches `?as=` in the URL; refreshing the page preserves the toggle state.
- All queries in PrivateCoachingView, ReadyForJointView, and JointChatView use `useSoloActingParty()` to resolve the acting userId; each party's private data remains isolated even in solo mode.
- AI responses in private coaching are generated independently per acting party (separate message rows, separate context).
- Solo cases are flagged in audit log entries where applicable.
- Playwright `solo-full-flow.spec.ts`: create solo case, toggle to initiator, complete private coaching (mark complete), toggle to invitee, complete private coaching (mark complete), verify both parties' dashboard shows READY_FOR_JOINT status.

## Implementation Notes

**Mutation changes (`convex/cases/create.ts`):** Add `isSolo: v.optional(v.boolean())` to the validator args. When `isSolo: true`, set both `initiatorUserId` and `inviteeUserId` to the calling user, create two `partyStates` rows (one with role `INITIATOR`, one with role `INVITEE`) in the same mutation, and skip invite token generation entirely. The case starts in `BOTH_PRIVATE_COACHING` status rather than `DRAFT_PRIVATE_COACHING`, since both parties are present immediately. Use `v.id("users")` for all user ID fields — never raw strings — so the generated `Id<"users">` type enforces correctness at compile time.

**`useSoloActingParty` hook (`src/hooks/useSoloActingParty.ts`):** This hook reads `?as=initiator` or `?as=invitee` from the URL via React Router's `useSearchParams()`. It maps the role string to the appropriate userId from the case document. The hook should default to `'initiator'` when `?as=` is absent or carries an unrecognised value. In non-solo mode the hook is still called but returns the logged-in user's ID regardless of the param, keeping callers uniform. All phase views (PrivateCoachingView, ReadyForJointView, JointChatView) must call this hook and pass its `actingUserId` to every Convex query rather than reading from session directly — this is the key isolation boundary.

**`PartyToggle` component (`src/components/PartyToggle.tsx`):** Per the accessibility rules, this toggle must use `role="radiogroup"` on the container with individual `role="radio"` children, consistent with how the spec calls out the party toggle in the `accessibility` design rule. It reads the current value from `useSearchParams()` and writes back via `setSearchParams`. Keep it out of the global nav for non-solo cases — render it only when `case.isSolo === true`.

**Dashboard badge:** The SOLO badge is a small pill using `--accent-subtle` background and `--text-secondary` text — no custom color; do not introduce a new token for this. The `cases/list` query already returns `isSolo` from the schema, so no query changes are needed beyond reading the field.

**Audit log entries:** When `isSolo: true`, include `{ isSolo: true }` in the `metadata` field of any `auditLog` rows written during case creation. The `auditLog` schema uses `v.any()` for `metadata`, so this requires no schema change.

**Private data isolation in solo mode:** Even though both `partyStates` rows share the same `userId`, the `privateCoaching/messages` query must scope results to the specific `partyState._id` (not just `userId`). Callers pass the partyState ID derived from the active role, not the user ID. The existing privacy enforcement at the query layer (introduced in T10b) must remain intact — a direct cross-role query must still return FORBIDDEN.

- Any changes to the `privateCoaching/messages` query beyond ensuring `partyStateId` is the scope key (the FORBIDDEN enforcement from T10b is unchanged).
