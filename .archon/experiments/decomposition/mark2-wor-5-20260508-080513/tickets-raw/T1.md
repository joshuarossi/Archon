This task lays the type-safe data foundation every other Phase 0 and Phase 1 task depends on. It defines the complete Convex database schema (all 11 tables), the single state machine module that controls all case status transitions, the normalized error factory used by every Convex function, and the `requireAuth` helper that enforces authentication at the boundary of every callable. Because the schema is Convex's source of truth for generated TypeScript types, every downstream query, mutation, and UI component will break at compile time if these definitions are incomplete or incorrect â€” getting them right in Phase 0 means type errors surface as close to their origin as possible.

## Implementation Notes

**Schema design (convex/schema.ts):** Use Convex's `defineSchema` + `defineTable` API throughout. All foreign-key fields must use `v.id("tableName")` â€” never raw strings â€” so Convex codegen produces the branded `Id<"tableName">` type that creates compile-time cross-table safety. Every field that will be queried by a non-`_id` predicate needs an explicit index declaration (`index("by_caseId", ["caseId"])`); missing indexes cause runtime errors, not silently slow queries. The `cases` table's three cost fields (`costUsd`, `costTokens`, `costLimited`) are optional because they are only populated after the first AI call and cap enforcement happens in Phase 2 (T10d stub, T17c real logic). Adding them to the schema now avoids a future schema migration that would require redeployment.

**State machine design (convex/stateMachine.ts):** The module enforces the DAG of valid transitions so that no mutation anywhere in the codebase can directly write a new `status` onto a case â€” they all go through `transitionCase`. The valid transition edges are: `DRAFT_PRIVATE_COACHING â†’ BOTH_PRIVATE_COACHING`, `BOTH_PRIVATE_COACHING â†’ READY_FOR_JOINT`, `READY_FOR_JOINT â†’ JOINT_ACTIVE`, and from `JOINT_ACTIVE` to any of three terminal statuses (`CLOSED_RESOLVED`, `CLOSED_UNRESOLVED`, `CLOSED_ABANDONED`). Re-entry idempotence means calling `transitionCase` with the case's current status should be a no-op rather than an error â€” this prevents double-execution from scheduled actions from throwing spuriously. Any other transition (e.g., `DRAFT_PRIVATE_COACHING â†’ JOINT_ACTIVE`, or attempting to leave any CLOSED status) must throw `createError("CONFLICT", ...)`.

**Error normalization (convex/lib/errors.ts):** `ConvexError` is the Convex-native error class that propagates cleanly to the React client. The `createError` factory should wrap it with a typed discriminated union shape `{ code: ErrorCode; message: string }` so the client can switch on `err.data.code` without string matching. The `requireAuth` helper calls `ctx.auth.getUserIdentity()` and performs a users table lookup by the identity's `tokenIdentifier` or email field; it must throw `UNAUTHENTICATED` rather than return null so callers never need to null-check the result.

**Convex function types:** This task only creates schema and library helpers â€” no callable functions (`query`, `mutation`, `action`) yet. `transitionCase` is a plain async TypeScript function that accepts a `MutationCtx` because it will be called from mutations. It is not itself wrapped in `mutation()`. Following the Convex architecture rules: never call external APIs (Claude, email) from mutations; `transitionCase` is safe because it only reads/writes the DB.

**Testing approach:** Use `convex-test` for any test that exercises Convex schema or the in-memory runtime. The Vitest spec imports `transitionCase` directly and seeds cases rows at each status before exercising each transition group. Do NOT mock Convex internals â€” `convex-test` provides a real in-memory runtime that validates index usage and schema types.
