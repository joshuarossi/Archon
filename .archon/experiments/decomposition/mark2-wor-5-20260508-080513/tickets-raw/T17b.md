# Coach Facilitation Action — Sonnet Streaming, Soft-Cap Degradation, Privacy Filter Integration

T17b implements `convex/jointChat/generateCoachResponse.ts`, the Convex action that drives real-time AI facilitation in the joint chat session. In Phase 2, once both parties have completed private coaching and their syntheses are available, this action is the mechanism by which the Coach responds to the live conversation — streaming its reply into `jointMessages` row-by-row. It integrates three cross-cutting concerns built in parallel: the soft/hard cost cap enforcement from T17c, the privacy leak filter from T13, and the multi-gate short-circuit logic from T17a's `coachShouldRespond` check.

## Acceptance Criteria

- `generateCoachResponse` action calls `coachShouldRespond` first; if `shouldRespond: false`, returns immediately without calling Sonnet.
- Before calling Sonnet, calls `checkCostLimit(ctx, caseId)` from `convex/lib/costTracking.ts`; if `{ limited: true }` (hard cap), returns immediately without calling Claude.
- If `checkCostLimit` returns `{ softCapReached: true, limited: false }`, the action skips calling Sonnet and instead inserts a `jointMessages` row (authorType: COACH, status: COMPLETE) with the boilerplate soft-cap text from tech spec §6.6 (e.g., 'I'll keep my facilitation brief to stay within this case's AI budget — consider summarizing where you are and whether you've reached agreement.'); subsequent invocations that still return `softCapReached: true` repeat the same boilerplate without calling Sonnet.
- When generating a full Coach response, calls `assemblePrompt(role=COACH)` which includes joint chat history + both parties' synthesisText; does NOT include any raw privateMessages.
- Coach response is passed through `privacyFilter.checkPrivacyLeak` against both parties' raw private messages; retries up to 2 times on failure; falls back to generic facilitation message ('Let's take a moment to reflect on what's been shared.') on third failure.
- Streaming pattern: insert `jointMessages` row with `status: STREAMING`, batch token updates every ~50ms, set `status: COMPLETE` on finish.
- After streaming completes, calls `recordSpend(ctx, caseId, tokensUsed)` from `convex/lib/costTracking.ts`.
- All AI error cases (429, timeout, content filter, invalid key) surface an error state to the user with a Retry button; 401 errors log to audit log per tech spec §6.5.
- Vitest unit test: privacy filter is applied to Coach output; adversarial input containing verbatim other-party text triggers retry flow.
- Vitest unit test: given `checkCostLimit` returning `{ softCapReached: true, limited: false }`, `generateCoachResponse` does not call the Anthropic SDK and inserts a jointMessages row containing the boilerplate soft-cap text.

## Implementation Notes

This action follows the same structure as T10b's `privateCoaching/generateResponse.ts` and T14's synthesis action — but with two additional guard layers before reaching the Anthropic SDK.

**Short-circuit order matters:** The three guards must fire in this exact sequence to avoid unnecessary DB writes or API calls: (1) `coachShouldRespond` (no-op if not the Coach's turn), (2) `checkCostLimit` hard cap (no-op if budget exhausted), (3) `checkCostLimit` soft cap (insert boilerplate, skip Claude). Only after all three pass does the action call `assemblePrompt(role=COACH)` and open the Anthropic stream.

**Prompt assembly:** `assemblePrompt` (from `convex/lib/prompts.ts`, T10a) must be called with `role: "COACH"`. The Coach prompt context includes joint chat history and both parties' `synthesisText` fields. It must never include `privateMessages` — the privacy guarantee is architectural: `assemblePrompt(role=COACH)` is defined to exclude private data, but `generateCoachResponse` also validates this by running `privacyFilter.checkPrivacyLeak` after the response is generated.

**Privacy filter retry loop:** After each Sonnet response, call `privacyFilter.checkPrivacyLeak(otherPartyMessages, response)` for both parties' private message arrays. If `passed: false`, discard the response and retry (up to 2 retries). On the third failure, insert the fallback string rather than retrying. This is the same pattern as T14's synthesis action. The `checkPrivacyLeak` function is a pure function from `convex/lib/privacyFilter.ts` (T13) — no Convex dependency, no mock needed in unit tests.

**Streaming persistence:** Insert a `jointMessages` row with `status: STREAMING` before the first token arrives. Batch-patch `body` content every ~50ms via `ctx.runMutation()`. On the final stream event, capture `usage.output_tokens` and call `completeMessage` (sets `status: COMPLETE`, stores token count) and `recordSpend` (accumulates USD cost on the `cases` row). Do not record spend on retry paths that ultimately fall back to the privacy fallback message — only record real Anthropic token usage.

**Error handling per tech spec §6.5:** 429 → retry once after 2s; second 429 → surface error state with Retry button. 401 → do NOT retry; write to audit log via `internal.lib.auditLog.logEvent`. Timeout and content-filter errors → surface error state with Retry button. Use `ConvexError` with typed `{ code, message }` shapes for all thrown errors. The UI (T16/T17a's calling surface) renders errors as `MessageBubble` with `--warning-subtle` background and a Retry button.

**Model:** Use `claude-sonnet-4-5` (defined in `convex/lib/models.ts`). The Coach uses Sonnet (not Haiku) because facilitation quality matters more than cost at this stage; Haiku is reserved for cheap classification tasks like `coachShouldRespond`.

**Invariants**

- `coachShouldRespond` is always the first check; if it returns false, the action exits with zero DB mutations and zero API calls.
- `checkCostLimit` is always called before any Anthropic API call. A `{ limited: true }` result exits with zero DB mutations.
- A `{ softCapReached: true, limited: false }` result inserts exactly one `jointMessages` row (COMPLETE, COACH, boilerplate body) and never calls the Anthropic SDK.
- `privacyFilter.checkPrivacyLeak` is applied after every full Sonnet response (not during streaming); it is never skipped.
- `recordSpend` is called exactly once per successful full-generation path (i.e., after `completeMessage`), using the `usage.output_tokens` value from the final stream event. It is not called on soft-cap, hard-cap, or privacy-fallback paths.
- The STREAMING row is inserted before the first token is emitted; the action never emits tokens to a non-existent row.

**Non-goals**

- Does NOT implement the `coachShouldRespond` gate logic (T17a).
- Does NOT implement `checkCostLimit` or `recordSpend` enforcement thresholds (T17c).
- Does NOT implement `privacyFilter.checkPrivacyLeak` (T13).
- Does NOT implement `assemblePrompt` (T10a).
- Does NOT build or modify any joint-chat UI components (T16).
- Does NOT handle scheduling `generateCoachResponse` — the caller (T17a's trigger path or the joint-chat message mutation) schedules it via `ctx.scheduler.runAfter`.
