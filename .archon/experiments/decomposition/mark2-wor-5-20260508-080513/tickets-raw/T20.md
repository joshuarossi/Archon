Case closure is the final step of the joint session in Conflict Coach. When both parties feel they have reached agreement, either party can propose a formal closure with an agreed-outcome summary; the other party confirms it and the case is marked `CLOSED_RESOLVED`. If one party needs to leave the process without mutual agreement, they can walk away unilaterally, producing a `CLOSED_UNRESOLVED` state. This task, in Phase 2 after the joint chat is functional (T16), implements all three closure mutations, the closure UI within `JointChatView`, and the read-only `ClosedCaseView` that both parties see once the case is closed.

## Acceptance Criteria

- JointChatView shows a 'Propose Closure' button when case is JOINT_ACTIVE; clicking opens a modal for entering agreed-outcome summary text.
- Calling `proposeClosure` sets `partyStates.closureProposed = true` and stores the summary on the case; the other party sees a closure proposal banner in JointChatView with Confirm and Reject options.
- Calling `confirmClosure` when the other party has proposed transitions the case to CLOSED_RESOLVED (via stateMachine); both parties are redirected to ClosedCaseView.
- Rejecting a closure proposal clears `closureProposed` and the banner disappears.
- JointChatView shows a separate 'Walk Away' option; calling `unilateralClose` transitions the case to CLOSED_UNRESOLVED immediately.
- The other party's dashboard shows a badge/indicator when their case has been unilaterally closed.
- ClosedCaseView shows: closure status label, closure summary (if CLOSED_RESOLVED), full joint chat message history (read-only), closed-at date.
- Any mutation that would modify a CLOSED case (sendUserMessage, proposeClosure, etc.) throws `CONFLICT`.
- Closed cases appear in the 'Closed' section of the Dashboard and clicking them navigates to ClosedCaseView.
- Playwright: solo mode — propose closure as initiator, confirm as invitee, verify both views show ClosedCaseView.

## Implementation Notes

All four closure mutations are Convex mutations (not actions — no external I/O). Each must begin with `requireAuth(ctx)` from `convex/lib/errors.ts` and verify the caller is a party to the case before touching any data. All state transitions go through `stateMachine.transitionCase` (from T1) to guarantee invariants: only `JOINT_ACTIVE` → `CLOSED_RESOLVED` and `JOINT_ACTIVE` → `CLOSED_UNRESOLVED` are legal from this path; any other source status throws `CONFLICT` at the state machine level.

`proposeClosure` writes `partyStates.closureProposed = true` for the calling party's row and sets an `agreementSummary` field on the `cases` row. The other party's `JointChatView` subscribes reactively via `useQuery`; the proposal banner appears automatically when the query returns the proposal state — no polling needed. `confirmClosure` validates that the caller is the party whose `closureProposed` field is false (i.e., they did not propose — they are confirming the other's proposal), then calls `stateMachine.transitionCase` to `CLOSED_RESOLVED` and sets `cases.closedAt`. `rejectClosure` simply clears `partyStates.closureProposed` and the summary. `unilateralClose` calls `stateMachine.transitionCase` to `CLOSED_UNRESOLVED` and sets `cases.closedAt`.

The "Walk Away" flow is destructive in the mediation sense. Per the design system `tone-and-copy` rules, the confirmation dialog must explain the consequence in plain language referencing the other party by name — never expose the `CLOSED_UNRESOLVED` code directly. Per `states-and-feedback`, all state-changing modals must use Radix `Dialog` (not `window.confirm()`).

`ClosedCaseView.tsx` is a read-only shell. It reuses `ChatWindow` (T9) in a non-interactive mode: no `MessageInput`, no send button, no retry buttons. The `cases/get` query and `jointChat/messages` query already enforce party membership; the state machine enforces that no write mutations can proceed once a case is in any `CLOSED_*` status. The Dashboard `cases/list` query already groups by Active/Closed (T8); this task does not change the query but does ensure `CaseDetail.tsx` routes to `ClosedCaseView` for all three closed statuses.

For the unilateral-close badge on the other party's dashboard: the `cases/list` query reactive update carries the new `cases.status` to the dashboard immediately; add a badge style to case rows whose status is `CLOSED_UNRESOLVED`. The badge copy should be understated per tone-and-copy: "Closed without resolution" not "Jordan walked away."

**Invariants**

- `proposeClosure`, `confirmClosure`, `rejectClosure`, `unilateralClose` all throw `FORBIDDEN` if the caller is not a party to the case.
- `proposeClosure`, `confirmClosure`, `rejectClosure` throw `CONFLICT` if `cases.status !== 'JOINT_ACTIVE'`.
- `unilateralClose` throws `CONFLICT` if `cases.status !== 'JOINT_ACTIVE'`.
- `confirmClosure` throws `CONFLICT` if the calling user is the one who proposed (only the non-proposing party may confirm).
- `sendUserMessage` (T16) throws `CONFLICT` on any `CLOSED_*` status — this is enforced by its existing guard and remains unchanged.
- `cases.closedAt` is set exactly once, in the same mutation that sets the terminal status.
- `ClosedCaseView` renders only `jointMessages`; it never renders `privateMessages` or `draftMessages`.

**Non-goals**

- Email or push notifications to the other party on unilateral close — deferred.
- Re-opening or appealing a closed case — not in v1.
- T21 (abandoned case cron, `CLOSED_ABANDONED`) — separate task.
- PDF transcript export — deferred P2 user story.
- Editing the `agreementSummary` after proposing but before confirmation.
- Admin-level forced case closure.
