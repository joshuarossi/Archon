This task completes the UI polish layer for Conflict Coach v1 by filling in every missing feedback surface the user can encounter during normal and degraded operation. Phase 3 is the final hardening pass before launch: by the time Phase 2 AI features are wired, every loading state, error state, empty state, cost-cap UI, crisis safety gate, and synthesis fallback needs to exist so that no interaction path leaves the user staring at a blank screen or a broken layout. T26 delivers all of those surfaces in a single coordinated pass.

## Acceptance Criteria

- AI messages with `status: ERROR` show an error description and a Retry button; clicking Retry re-schedules the AI action for that message.
- When a case reaches the $2 soft cap, the Coach AI sends the boilerplate degraded message from tech spec §6.6 (implemented in T17b's soft-cap branch); subsequent Coach invocations continue sending boilerplate without calling Sonnet.
- When a case reaches the $10 hard cap (`COST_LIMITED`), JointChatView shows a banner 'AI features are limited for this case' and the Coach/Draft Coach/synthesis action buttons are disabled; parties can still send manual messages.
- Dashboard shows an empty state ('No active cases yet — start a new one') when the user has no cases.
- Joint chat shows an empty state message ('The conversation will appear here once you begin') when no messages exist yet.
- Case detail shows loading skeletons while Convex reactive queries are pending.
- NewCasePage shows a crisis safety card: 'Conflict Coach is not for situations involving abuse, threats, or immediate safety concerns. If you are in danger, please contact emergency services.' This appears above the form and must be acknowledged (checkbox or dismissal) before the form is interactive.
- Synthesis fallback UI: when synthesis used the generic fallback, the ReadyForJointView shows 'Personalized guidance is unavailable for this case — here is general joint-session advice' with a generic text block.
- All error states are screen-reader-friendly: error messages use `role='alert'`.
- Playwright: in CLAUDE_MOCK mode, configure the mock to return an error response; verify the Retry button appears and clicking it retriggers the AI action.

## Implementation Notes

**Error bubble (AC1, AC9):** The `MessageBubble` component already handles `status: "STREAMING"` and `status: "COMPLETE"`. Extend it with a third render branch for `status: "ERROR"`: a bubble using `--danger-subtle` background (`.cc-bubble-error` from the design system), a Lucide `AlertTriangle` icon at 16×16 in `--danger`, the error description text, and a secondary `[Retry]` button flush right. The Retry button calls a Convex mutation — `api.privateCoaching.retryMessage` or `api.jointChat.retryMessage` depending on context — which sets the message row back to `status: "STREAMING"` and re-schedules the AI action via `ctx.scheduler.runAfter(0, ...)`. Wrap the error container in a `<div role="alert">` so screen readers announce it on render (AC9). Do not use a toast — inline placement in the message thread is required by the design rules.

**Soft-cap degraded message (AC2):** This AC tests T17b's soft-cap branch, not new UI code. The soft-cap logic lives in `convex/lib/costTracking.ts` and the Coach mutation. T26's responsibility is confirming that when `costUsd >= 2`, the existing boilerplate path fires and the boilerplate text appears in the chat thread as a normal `COMPLETE` Coach bubble — no new component needed. Verify via Playwright with a seeded case whose `costUsd` is $1.99, then send one more message.

**COST_LIMITED banner (AC3):** In `JointChatView.tsx`, read `caseData.costLimited` from the reactive `cases/get` query. When truthy, render a dismissible (but persistent — it re-renders on page load) `<div role="alert">` banner at the top of the chat body: "AI features are limited for this case." The Coach intervention button, the Draft Coach button, and the synthesis CTA in `ReadyForJointView` must each check this flag and render as `disabled` + `opacity-50 cursor-not-allowed`. Manual `MessageInput` remains fully enabled — only AI-triggering surfaces are gated.

**Loading skeletons (AC6):** Use Tailwind `animate-pulse` on `bg-[color:var(--bg-surface-subtle)]` blocks matching the real layout. Dashboard skeleton: three rows, each shaped like a case row (avatar circle, two text lines, status glyph at right). Case detail skeleton: varies by phase — private coaching shows a chat skeleton (alternating left/right bubbles); ready-for-joint shows a card skeleton. Render skeletons when the `useQuery` return is `undefined` (loading) rather than `null` (not found) or an array (loaded). Convex reactive queries return `undefined` only during the initial load; downstream renders are synchronous updates.

**Empty states (AC4, AC5):** These are simple conditional renders when the query returns an empty array. Dashboard: `cases.length === 0` → centered text "No cases yet. When you're ready to work through something, start a new case." with a primary `[+ New Case]` button. Joint chat: `messages.length === 0` → centered muted text "The conversation will appear here once you begin." Note: the design rule says this state "shouldn't happen" because Coach always sends an opener, but defensive rendering is still required.

**Crisis safety card (AC7):** Render a `Card` component above the new-case form in `NewCasePage.tsx` with the required copy. Gate form interactivity with a controlled boolean state: a shadcn `Checkbox` labeled "I understand" (or a `[Dismiss]` button) sets `acknowledged = true`, after which the form fields and submit button become enabled. Before acknowledgement, inputs render with `opacity-50 pointer-events-none` and the submit button is `disabled`. This is purely client-side state — no persistence needed.

**Synthesis fallback (AC8):** T14 already writes a `isFallback: boolean` field (or equivalent) onto the synthesis record. In `ReadyForJointView.tsx`, when `synthesis.isFallback === true`, replace the synthesis card header with "Personalized guidance is unavailable for this case — here is general joint-session advice" and render a static generic text block (can be a constant string in the file). The rest of the synthesis card layout stays the same.

**Playwright (AC10):** In `CLAUDE_MOCK=true` mode, extend the mock to support an `ERROR` response variant — e.g., when the message content matches a sentinel string like `__trigger_error__`, the mock action writes `status: "ERROR"` instead of `COMPLETE`. Author the spec in `playwright/error-states.spec.ts`. Send the sentinel message, wait for the Retry button to appear (`page.getByRole('button', { name: 'Retry' }).waitFor()`), click it, and assert that a second Coach message begins streaming (i.e., the prior `ERROR` bubble is replaced or followed by a `STREAMING` bubble).

**Invariants**
- Error bubble (`role="alert"`) must render inline at the position the AI response would have appeared — never as a toast.
- COST_LIMITED banner re-renders on every page load from the reactive query; it must not be client-dismissible in a way that persists across sessions.
- Crisis safety card acknowledgement is purely transient client state; it resets on page reload (intentional — the card must be visible every time the user opens NewCasePage).
- Loading skeleton renders only when `useQuery` returns `undefined`; it must not flash when the query returns an empty array.
- Retry button is absent on `status: "STREAMING"` and `status: "COMPLETE"` messages — it appears exclusively on `status: "ERROR"`.
- Disabled AI buttons under `COST_LIMITED` must remain visible at `opacity-50` — never hidden.
