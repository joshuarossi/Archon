# Solo mode — isSolo case creation, PartyToggle, useSoloActingParty hook (US-13, FR-09)

Solo mode lets a single user experience the full Conflict Coach flow from both parties' perspectives — useful for self-reflection, facilitation practice, or developer testing. This task adds the `isSolo` creation path to the `cases/create` mutation, the `PartyToggle` nav component that switches which party's context is active, and the `useSoloActingParty` hook that all phase views use to resolve the correct acting userId and role from the `?as=` URL param. Without solo mode the E2E `solo-full-flow.spec.ts` suite — used to validate synthesis, joint chat, and Draft Coach in CI — cannot run, making it a prerequisite for Phase 2 integration testing.

## Acceptance Criteria

- Dashboard has a 'New Solo Case' button distinct from 'New Case'; solo cases are visibly flagged with a 'SOLO' badge in the case list and case detail.
- `cases/create` with `isSolo: true` creates both `partyStates` rows bound to the same userId in one transaction; no invite token is created.
- Solo cases show `PartyToggle` in the top nav with 'Viewing as Initiator' and 'Viewing as Invitee' options.
- Toggling switches `?as=` in the URL; refreshing the page preserves the toggle state.
- All queries in PrivateCoachingView, ReadyForJointView, and JointChatView use `useSoloActingParty()` to resolve the acting userId; each party's private data remains isolated even in solo mode.
- AI responses in private coaching are generated independently per acting party (separate message rows, separate context).
- Solo cases are flagged in audit log entries where applicable.
- Playwright `solo-full-flow.spec.ts`: create solo case, toggle to initiator, complete private coaching (mark complete), toggle to invitee, complete private coaching (mark complete), verify both parties' dashboard shows READY_FOR_JOINT status.

## Implementation Notes

**Mutation changes (`convex/cases/create.ts`):** Add `isSolo: v.optional(v.boolean())` to the validator args. When `isSolo: true`, set both `initiatorUserId` and `inviteeUserId` to the calling user, create two `partyStates` rows (one with role `INITIATOR`, one with role `INVITEE`) in the same mutation, and skip invite token generation entirely. The case starts in `BOTH_PRIVATE_COACHING` status rather than `DRAFT_PRIVATE_COACHING`, since both parties are present immediately. Use `v.id("users")` for all user ID fields — never raw strings — so the generated `Id<"users">` type enforces correctness at compile time.

**`useSoloActingParty` hook (`src/hooks/useSoloActingParty.ts`):** This hook reads `?as=initiator` or `?as=invitee` from the URL via React Router's `useSearchParams()`. It maps the role string to the appropriate userId from the case document. The hook defaults to `'initiator'` when `?as=` is absent or carries an unrecognised value. In non-solo mode the hook is still called but returns the logged-in user's ID regardless of the param, keeping callers uniform. All phase views (PrivateCoachingView, ReadyForJointView, JointChatView) must call this hook and pass its results to every Convex query and action.

**Casing bridge — hook to query/action:** The hook returns `actingRole: "initiator" | "invitee"` (lowercase). T10b's `api.privateCoaching.messages` query and the `api.privateCoaching.generateAIResponse` action both use the uppercase literals `'INITIATOR' | 'INVITEE'` (per the established Convex validator pattern). Callers must map: `actingRole.toUpperCase() as 'INITIATOR' | 'INVITEE'`. Do this at the call site in each view — do not change the hook's return type or the Convex validator.

**`PrivateCoachingView` prop change:** T10b gave `PrivateCoachingView` a `soloRole?: 'INITIATOR' | 'INVITEE'` prop and had the parent (`CaseDetail`) derive and pass it. T12 removes this prop: `PrivateCoachingView` now calls `useSoloActingParty()` internally to get `actingRole`, then passes `as: actingRole.toUpperCase() as 'INITIATOR' | 'INVITEE'` to `useQuery(api.privateCoaching.messages, { caseId, as: ... })`. Update `CaseDetail.tsx` to stop passing `soloRole`. The query interface (`{ caseId, as? }`) is unchanged from T10b.

**`generateAIResponse` solo discriminator:** T10b's `generateAIResponse` action takes `{ caseId, userMessage }` and resolves the partyState by `userId + caseId`. In solo mode there are two partyStates for the same userId — the action cannot distinguish them without a discriminator. T12 adds `as: v.optional(v.union(v.literal('INITIATOR'), v.literal('INVITEE')))` to the action's args validator. When `as` is present, the action uses it to select the correct partyState (role-based lookup instead of userId-only lookup). In non-solo mode `as` is omitted and the existing userId-based lookup is the fallback. `PrivateCoachingView` passes `as: actingRole.toUpperCase()` to the `useAction` call.

**`PartyToggle` component (`src/components/PartyToggle.tsx`):** Use the shadcn/ui `RadioGroup` primitive (listed as a required primitive in the component-patterns design rule — never build a raw `role="radiogroup"` / `role="radio"` pair from scratch). `RadioGroup` already provides the correct ARIA semantics. The component reads the current value from `useSearchParams()` and writes back via `setSearchParams`. Render it only when `case.isSolo === true` — keep it out of the global nav for non-solo cases.

**Dashboard badge:** The SOLO badge is a small pill using `--accent-subtle` background and `--text-secondary` text — no custom color; do not introduce a new token for this. The `cases/list` query already returns `isSolo` from the schema, so no query changes are needed beyond reading the field.

**Audit log entries:** When `isSolo: true`, include `{ isSolo: true }` in the `metadata` field of any `auditLog` rows written during case creation. The `auditLog` schema uses `v.any()` for `metadata`, so this requires no schema change.

**Private data isolation in solo mode:** Even though both `partyStates` rows share the same `userId`, the `privateCoaching/messages` query scopes results via the `as?: 'INITIATOR' | 'INVITEE'` param (established in T10b) — not by raw `userId`. This discriminator is derived from `actingRole` in each phase view. A direct cross-role query (e.g., authenticated as INITIATOR but passing `as: 'INVITEE'`) must still return FORBIDDEN; that enforcement (from T10b) is unchanged.

**Queries used**

- `api.cases.create` — updated mutation; callers pass `{ ..., isSolo: true }`
- `api.cases.list` — existing query; reads `isSolo` from result to conditionally render SOLO badge
- `api.privateCoaching.messages` — existing query taking `{ caseId, as?: 'INITIATOR' | 'INVITEE' }` (T10b interface); views pass `as: actingRole.toUpperCase()` after `useSoloActingParty` resolves `actingRole`. **Not parameterised by `partyStateId`** — the `as` param is the role discriminator.
- `api.privateCoaching.generateAIResponse` — updated action; views pass `{ caseId, userMessage, as: actingRole.toUpperCase() }` in solo mode

**Invariants**

- A solo case always has exactly two `partyStates` rows; both `partyState.userId` fields equal the creating user's ID.
- A solo case never has an `inviteTokens` row.
- `useSoloActingParty` with `?as=invitee` must return `{ actingUserId: inviteeUserId, actingRole: 'invitee' }`; with `?as=initiator` (or absent) must return `{ actingUserId: initiatorUserId, actingRole: 'initiator' }`.
- The `?as=` param survives a full-page reload (URL-based, not memory-based).
- `api.privateCoaching.messages` with `as: 'INVITEE'` called by the INITIATOR identity must return FORBIDDEN (T10b enforcement, unchanged).
- `auditLog` rows written during solo case creation carry `metadata.isSolo === true`.

**Non-goals**

- Invite link generation for solo cases (explicitly skipped).
- Allowing a solo case to be converted to a two-party case post-creation.
- UI for the `?as=` toggle in non-solo case views.
- Changing the `api.privateCoaching.messages` query interface (T10b's `{ caseId, as? }` is the established contract).
