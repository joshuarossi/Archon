# CI pipeline + test infrastructure (GitHub Actions, Vitest, Playwright, CLAUDE\_MOCK stub, seed script)

This task wires together the complete quality-gate and development-environment bootstrap for Conflict Coach v1. With the schema (T1), auth (T2), and React shell (T3) in place, Phase 0 requires a CI pipeline, two test runners, a deterministic AI mock, and a repeatable seed script before any Phase 1 feature work begins. Without green CI and reliable local setup, every subsequent task will face unclear breakage signals and uncatchable regressions.

## Acceptance Criteria

- GitHub Actions CI workflow file exists at `.github/workflows/ci.yml` with four named jobs: lint, typecheck, unit, e2e.
- CI runs on every push to `main` and every pull request.
- Vitest is configured; `npm run test:unit` runs and exits 0 with at least one placeholder test.
- Playwright is configured; `npm run test:e2e` runs and exits 0 with at least one placeholder spec.
- When `CLAUDE_MOCK=true` env var is set, all Convex actions that call Anthropic use `aiMock.ts` instead; the mock streams back a deterministic string in 50ms-incremented chunks, then sets status `COMPLETE`.
- `convex/seed.ts` is callable with `npx convex run seed` and creates: one admin user (email configurable via env var), templates for 'workplace', 'family', 'personal' categories each with a minimal `globalGuidance` string and a published initial version.
- Running seed twice is idempotent (no duplicates).
- README documents the cold-start sequence: `git clone && npm install && cp .env.example .env.local && npx convex dev && npm run dev`.

## Implementation Notes

**GitHub Actions structure.** The four jobs are `lint`, `typecheck`, `unit`, and `e2e`. `lint` and `typecheck` run in parallel (no `needs:` dependency between them); `unit` may also run concurrently with both; `e2e` has `needs: [unit]` only. All jobs use `ubuntu-latest`, `node-version: '20'`, and `actions/cache` keyed on `package-lock.json`. The `lint` job runs ESLint with `--max-warnings 0` and then `npm run format:check` (Prettier). The `typecheck` job runs `npx tsc --noEmit`. All secrets (`ANTHROPIC_API_KEY`, `CONVEX_DEPLOY_KEY`) are injected as `${{ secrets.* }}` â€” never hardcoded. The e2e job sets `CLAUDE_MOCK: true` in its `env:` block and references a dedicated `CONVEX_DEPLOY_KEY` scoped to a separate CI Convex environment, not the developer's personal dev deployment.

**Vitest configuration.** Vitest uses `jsdom` as the test environment with `globals: true` and `src/test-setup.ts` importing `@testing-library/jest-dom`. Any test that imports from `convex/server` or `convex/values` must use `convex-test` (the in-memory Convex runtime) â€” do not mock Convex internals. For Convex actions that call Claude, mock `@anthropic-ai/sdk` at module level with `vi.mock()`. The placeholder test for this task need only be a trivial `it('placeholder', () => expect(true).toBe(true))` in a throwaway file; real feature tests are authored on their own tasks.

**Playwright configuration.** `playwright.config.ts` sets `baseURL: 'http://localhost:5173'`, `use: { headless: true }`, and a `globalSetup` script (`e2e/global-setup.ts`) that runs `npx convex run seed` against the CI Convex deployment before any spec files execute. Only the chromium project is needed for v1. The placeholder spec need only navigate to `baseURL` and assert a 200; real E2E specs are authored per-feature. Use accessible selectors (`getByRole`, `getByLabel`) everywhere â€” CSS selectors and `data-testid` are last resort. `CLAUDE_MOCK=true` is required for every CI run of Playwright; no real Anthropic API calls in tests.

**AI mock (`convex/lib/aiMock.ts`).** The module exports a named function `mockAIStream` as a drop-in stub for wherever the real `@anthropic-ai/sdk` streaming call is made. When `CLAUDE_MOCK=true`, every Convex action that calls Anthropic must branch to this stub. The stub streams back a deterministic string (e.g. `"[MOCK AI RESPONSE]"`) using `setInterval(..., 50)` for realistic chunk timing and terminates the stream after the full string is emitted, then sets the message row's `status` to `COMPLETE`. The implementation must be stateless and not depend on any schema specifics so that future AI action callers can import it without coupling.

**Seed script (`convex/seed.ts`).** This is a Convex `internalMutation` (not `mutation`) so that it cannot be invoked from client code â€” only from the CLI via `npx convex run seed` or from other internal Convex functions. It writes: one `users` row with `role: 'ADMIN'` (email read from env var `SEED_ADMIN_EMAIL`, defaulting to `admin@example.com`); and three `templates` rows â€” one each for `workplace`, `family`, and `personal` â€” each with a minimal `globalGuidance` string and a `templateVersions` row marked `isPublished: true`. Idempotency uses a **read-then-insert-or-skip** pattern (Convex has no native upsert): query `users` by email before inserting; query `templates` by category name before inserting; query `templateVersions` by `templateId` before inserting. Never call `db.insert()` if a matching row already exists. `convex-test` is the right way to unit-test seed; however, the primary acceptance check is running it twice against a real deployment and confirming no duplicate rows.

**TypeScript.** `tsconfig.json` must have `strict: true`, `noUncheckedIndexedAccess: true`, and `moduleResolution: "bundler"` (for the Vite/browser frontend). `convex/tsconfig.json` extends root and overrides `moduleResolution: "node16"` for Convex's CJS/ESM interop. `tsc --noEmit` in the typecheck job must pass without errors before any CI merge.

## Test-Gen Brief

### Contract

**Files**

- `.github/workflows/ci.yml` â€” CI workflow definition
- `vite.config.ts` â€” Vitest test block (`test.environment`, `test.globals`, `test.setupFiles`)
- `src/test-setup.ts` â€” Vitest global setup (imports `@testing-library/jest-dom`)
- `playwright.config.ts` â€” Playwright base config
- `e2e/global-setup.ts` â€” Playwright global setup script (runs seed before specs)
- `e2e/placeholder.spec.ts` â€” placeholder Playwright spec
- `src/__tests__/placeholder.test.ts` â€” placeholder Vitest spec
- `convex/lib/aiMock.ts` â€” AI mock stub module
- `convex/seed.ts` â€” Convex admin seed script

**Exports**

- `convex/lib/aiMock.ts`: named export `mockAIStream(ctx, caseId, messageRowId): Promise<void>` â€” value, async function
- `convex/seed.ts`: named export `seed` wrapped with `internalMutation()` â€” value, Convex internal mutation callable via `npx convex run seed`

**Signatures**

```ts
// convex/lib/aiMock.ts
export async function mockAIStream(
  ctx: ActionCtx,
  caseId: Id<"cases">,
  messageRowId: Id<"privateMessages"> | Id<"jointMessages">,
): Promise<void>;

// convex/seed.ts
export const seed = internalMutation({
  args: {},
  handler: async (ctx: MutationCtx): Promise<void> => { ... },
});
```

**Queries used**

- `npx convex run seed` â€” CLI invocation of the seed mutation; no client-side query calls in the seed itself
- `ctx.db.query("users").filter(...)` â€” inside seed, to check for existing admin user before inserting
- `ctx.db.query("templates").filter(...)` â€” inside seed, to check for existing templates before inserting
- `ctx.db.query("templateVersions").filter(...)` â€” inside seed, for idempotent version creation

**Invariants**

- Running `seed` zero, one, or any number of times produces exactly one admin user row, exactly three template rows (one per category), and exactly three `templateVersions` rows (one per template, `isPublished: true`).
- `aiMock.ts` never makes network calls; it is purely in-process.
- The `e2e` CI job never sets `ANTHROPIC_API_KEY` as a live value â€” `CLAUDE_MOCK=true` is the guard.
- `lint` and `typecheck` jobs do not share a `needs:` dependency â€” they are concurrent.
- ESLint runs with `--max-warnings 0`; a single warning fails the job.

**Non-goals**

- This ticket does not author any real feature tests (those ship with their feature tasks).
- This ticket does not configure staging or production deployment pipelines.
- This ticket does not implement the real Anthropic streaming logic (that is T10b, T14, T17b, T19).
- This ticket does not cover Dependabot, release automation, or branch protection rules.
- The seed script does not create case data or partyStates rows.
