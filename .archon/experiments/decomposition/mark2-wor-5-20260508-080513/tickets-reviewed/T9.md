# Shared ChatWindow component (ChatWindow, MessageBubble, MessageInput, StreamingIndicator)

The ChatWindow component is the reusable visual core of every AI conversation surface in Conflict Coach. It underpins Private Coaching (T10b), Joint Chat (T16), and Draft Coach (Phase 2), so the decisions made here propagate to every conversational view in the app. Phase 1 cannot close without this component because T10b explicitly depends on it for the PrivateCoachingView. The task is to build ChatWindow and its three sub-components â€” MessageBubble, MessageInput, and StreamingIndicator â€” as a self-contained, fully accessible unit that handles all three AI message lifecycle states (STREAMING, COMPLETE, ERROR) without polling or Convex awareness of its own.

## Acceptance Criteria

- ChatWindow accepts a `messages` array prop and renders each as a MessageBubble.
- User messages are visually distinguished from AI messages (different alignment or background).
- AI messages with `status: STREAMING` show content rendered so far plus a blinking cursor; copy button is hidden.
- AI messages with `status: COMPLETE` show full markdown-rendered content, a copy-to-clipboard button, and a relative timestamp.
- AI messages with `status: ERROR` show an error indicator and a 'Retry' button; clicking Retry calls an `onRetry(messageId)` callback.
- MessageInput submits on Enter key; Shift+Enter inserts a newline; Send button triggers the same action.
- ChatWindow auto-scrolls to the latest message unless the user has scrolled up (scroll position detection).
- Chat message container has `role='log'` and `aria-live='polite'` for screen reader accessibility.
- All interactive elements (Retry, Copy, Send) have visible labels or `aria-label`.
- Component is keyboard-navigable: Tab reaches Send button, Enter activates it.

## Implementation Notes

ChatWindow is a pure presentational component: it receives data via props and fires callbacks. It has no `useQuery` calls, no Convex imports, and no knowledge of which domain it belongs to. This is intentional â€” it keeps the component unit-testable with `@testing-library/react` + jsdom, without mocking Convex.

**Streaming pattern**: the reactive Convex query (owned by callers like PrivateCoachingView) pushes incremental `content` patches to a `privateMessages` row as the Claude stream progresses. ChatWindow renders whatever `content` it receives; it does not need to know it's streaming â€” that's communicated via `status`. The StreamingIndicator is shown only when `status === 'STREAMING'`; it is removed (not hidden) on `COMPLETE` per the motion design rule.

**Auto-scroll**: use a `ref` on the scroll container and a `useEffect` that watches the `messages` array length. On each new message, scroll to bottom only if `scrollTop + clientHeight >= scrollHeight - 50` (user is near the bottom). Track "user scrolled up" via an `onScroll` handler that sets a `userScrolled` boolean ref. Reset this flag whenever the user sends a message.

**Markdown rendering**: use `react-markdown` for AI message body content. Headings inside AI messages should render at `font-weight: 500` (not 700) per the typography rule â€” override via a `components` prop on `<ReactMarkdown>`.

**Message bubble visual rules** (from `component-patterns` design rule):
- User messages: right-aligned, `bg-[var(--bg-surface)]` background, no avatar.
- AI/Private Coach messages: left-aligned, `bg-[var(--accent-subtle)]` background, Sparkles icon.
- Coach messages in JointChatView (a future caller): left border `3px solid var(--coach-accent)`, `bg-[var(--coach-subtle)]`. ChatWindow should support a `variant` prop on MessageBubble for this distinction if callers need it; this ticket does not wire JointChatView â€” just ensure the prop exists.

**StreamingIndicator**: a thin vertical blinking cursor (1px Ă— 1em) via CSS `animation: blink 500ms step-end infinite`. Respect `prefers-reduced-motion: reduce` â€” in that environment, omit the animation entirely (show a static cursor or nothing).

**Accessibility**: the `role="log"` + `aria-live="polite"` pattern causes screen readers to announce new messages without interrupting ongoing reading. Do NOT set `aria-live="assertive"` here â€” it would interrupt the user on every token patch during streaming. Announce "Coach is replying" once when a STREAMING message first appears using a separate `aria-live` region that only updates on the STREAMINGâ†’content-starts transition, not on every patch.

**Copy button**: use the Clipboard API (`navigator.clipboard.writeText`). Show a checkmark icon for 2 seconds on success, then revert. The copy button is hidden (`display: none`, not `visibility: hidden`) during STREAMING â€” it must not be reachable via Tab while hidden.

**Retry button**: clicking Retry calls `onRetry(messageId)`. The ChatWindow does not know what re-scheduling a retry means â€” that's the caller's responsibility. `onRetry` is a required prop only when ERROR-state messages are possible; callers that never show errors can pass `() => {}`.

**Shadcn primitives**: wrap the message list in shadcn `ScrollArea` for consistent cross-browser scrollbar styling. The `MessageInput` textarea uses the shadcn `Textarea` primitive. The Send button uses the shadcn `Button` with `variant="primary"`.

**Focus ring**: all interactive elements must use the canonical Tailwind focus ring: `focus-visible:outline-2 focus-visible:outline-[color:var(--accent)] focus-visible:outline-offset-2`. Never remove focus rings; never use `:focus` without `:focus-visible`.

**Exhaustive status handling** (`assertNever`): `MessageBubble` renders different content depending on `message.status`. Use an exhaustive switch or if/else chain over `MessageStatus`. Add an `assertNever(status)` call in the default/else branch to produce a compile-time error if a new `MessageStatus` value is added without updating the component. Import `assertNever` from a shared utility (e.g., `src/lib/utils.ts`).

**`noUncheckedIndexedAccess` compliance**: because `tsconfig.json` sets `noUncheckedIndexedAccess: true`, avoid index-based access on the `messages` array (e.g., `messages[0]`). Use `.map()`, `.find()`, or `.at()` with an explicit undefined guard where needed. This is already natural for rendering (`.map()` over props), but be careful in scroll-position logic if you reference the last message.

**Signatures**

```typescript
// types.ts
export type MessageStatus = 'STREAMING' | 'COMPLETE' | 'ERROR';
export type MessageRole = 'user' | 'ai';

export interface ChatMessage {
  id: string;
  role: MessageRole;
  content: string;
  status: MessageStatus;
  createdAt: number; // Unix ms timestamp
}

// ChatWindow.tsx
export interface ChatWindowProps {
  messages: ChatMessage[];
  onSend: (text: string) => void;
  onRetry: (messageId: string) => void;
  isSubmitting?: boolean; // disables Send while AI is streaming
  className?: string;
}
export default function ChatWindow(props: ChatWindowProps): JSX.Element;

// MessageBubble.tsx
export interface MessageBubbleProps {
  message: ChatMessage;
  onRetry: (messageId: string) => void;
  variant?: 'private-coach' | 'joint-coach'; // default: 'private-coach'
}
export function MessageBubble(props: MessageBubbleProps): JSX.Element;

// MessageInput.tsx
export interface MessageInputProps {
  onSend: (text: string) => void;
  disabled?: boolean;
}
export function MessageInput(props: MessageInputProps): JSX.Element;

// StreamingIndicator.tsx
export function StreamingIndicator(): JSX.Element;
```

**Queries used**
- None. ChatWindow is fully presentational; it has no Convex imports.

**Invariants**
- The scroll container always has `role="log"` and `aria-live="polite"`.
- The copy button is never in the DOM (not just hidden) when `message.status === 'STREAMING'`.
- `onRetry` is called with the exact `message.id` string of the errored message â€” no mutation or side effect inside ChatWindow.
- StreamingIndicator never renders when `message.status !== 'STREAMING'`.
- Shift+Enter in `MessageInput` never triggers `onSend`; only unmodified Enter and the Send button trigger it.
- The Send button is never interactive when `disabled === true` (`opacity-50 cursor-not-allowed`, visually present).
- Auto-scroll does not fire when `userScrolled` is true and a new message arrives mid-scroll.
- `MessageBubble` has an `assertNever` branch that makes adding a new `MessageStatus` value a compile-time error.

**Non-goals**
- ChatWindow does not connect to Convex, call any mutation, or import from `convex/`.
- ChatWindow does not implement the Coach message left-border variant â€” it exposes the `variant` prop but callers (T16) are responsible for passing it.
- ChatWindow does not handle file attachments, images, or rich media.
- ChatWindow does not manage session state or message history persistence.
- This ticket does not wire ChatWindow into PrivateCoachingView (that is T10b's responsibility).
