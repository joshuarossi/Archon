The Draft Coach AI action is the backend intelligence behind the private drafting panel introduced in T18. It processes each message the user sends inside the Draft Coach panel, decides whether the user is ready to produce a final draft, streams the AI response back, and — on readiness detection — writes the polished draft to `draftSessions.finalDraft` so the UI can surface the `DraftReadyCard`. It is the fourth and final AI action in the system, and it operates under the same cost-tracking and privacy constraints as the three before it. Critically, its context window is scoped exclusively to the acting user's own data, making it the narrowest of all four roles.

## Acceptance Criteria

- Before calling Claude, calls `checkCostLimit(ctx, caseId)` from `convex/lib/costTracking.ts`; if `{ limited: true }`, throws `RATE_LIMITED` without invoking Claude.
- Draft Coach action context from `assemblePrompt(role=DRAFT_COACH)`: drafting user's joint-chat history + their own synthesisText. Does NOT include: other party's synthesisText, other party's privateMessages, other party's partyStates beyond what's in joint chat.
- Template `draftCoachInstructions` is injected into the system prompt if the case's pinned templateVersion has it; otherwise falls back to a default baseline Draft Coach instruction.
- Readiness detection regex covers: 'i'm ready', 'draft it', 'write the message', 'generate a draft', 'looks good, write it', or the canonical 'Generate a draft for me' string.
- When readiness is detected, the action produces a structured response containing `{ draft: '...' }` JSON; the parsed draft string is written to `draftSessions.finalDraft`.
- Non-readiness turns produce normal conversational responses.
- Streaming pattern is the same as private coaching: STREAMING row → token batches → COMPLETE.
- After streaming completes, calls `recordSpend(ctx, caseId, tokensUsed)` from `convex/lib/costTracking.ts`.
- Vitest unit test: readiness detection returns true for each of the 6 signal phrases and false for unrelated messages.
- Vitest unit test: context assembly for Draft Coach does not include other-party synthesisText.

## Implementation Notes

`generateResponse.ts` is a Convex **internalAction** — it calls the Anthropic API, so it cannot be a mutation, and it is never called directly from the client, so it must use `internalAction` (not `action`). It is registered at `internal.draftCoach.generateResponse`. The entry point is scheduled via `ctx.scheduler.runAfter(0, internal.draftCoach.generateResponse, ...)` from `draftCoach/sendMessage` (T18), passing `draftSessionId` and `caseId` as args.

**Cross-ticket arg name alignment (IMPORTANT).** T18's reviewed ticket currently calls the scheduler with `{ sessionId, caseId }` but T19's action expects `{ draftSessionId, caseId }`. These must match exactly — Convex validates args at the boundary and a mismatch causes a runtime validation failure. The canonical name is `draftSessionId` (matches the schema table `draftSessions`). Confirm T18's `sendMessage.ts` uses `{ draftSessionId, caseId }` before deploying.

**Context scoping** is the most important constraint. `assemblePrompt(role=DRAFT_COACH)` (implemented in T10a) already enforces the isolation: it fetches the acting party's own `synthesisText` and their own joint-chat history. `generateResponse.ts` simply calls `assemblePrompt` with the correct `actingUserId` and trusts T10a's boundary. It must NOT reach outside `assemblePrompt` to add additional context — adding context manually risks leaking the other party's data.

**Readiness detection** is a regex-based pre-pass run against the latest user message *before* calling Claude. Extract it into a named export (`detectReadiness`) so it can be tested in isolation without a Convex runtime. The 6 signal phrases are case-insensitive. The canonical phrase ("Generate a draft for me") is sent programmatically by the "Generate Draft" button in the panel (T18), so exact-match coverage is required.

**Structured output on readiness**: when `detectReadiness` returns true, add a system-prompt instruction forcing the response format to `{ "draft": "<message text>" }` JSON (use `response_format: { type: "json_object" }` or an equivalent critical instruction block). Parse the JSON server-side and write `draftSessions.finalDraft` via `ctx.runMutation(internal.draftCoach.writeFinalDraft, ...)`. If JSON parse fails on a readiness turn, retry the Claude call once before falling back to a generic "I wasn't able to produce a clean draft — please rephrase and try again" message inserted as a COMPLETE `draftMessages` row with no `finalDraft` write.

**Template instructions**: use `ctx.runQuery(internal.cases.getPinnedTemplateVersion, { caseId })` to look up the case's `templateVersionId`, then fetch the `templateVersions` row. If `draftCoachInstructions` is set, inject it as an additional system-prompt block. If the field is null/absent, inject a short default baseline: "Help the user compose a clear, non-inflammatory message to their counterpart. When the user is ready, produce only the message text as JSON."

**Streaming pattern**: identical to `generateAIResponse` in `privateCoaching/` — insert a `draftMessages` row with `status: "STREAMING"` via `ctx.runMutation(internal.draftCoach.insertStreamingRow, ...)`, accumulate token deltas, batch-patch content every ~50 ms via `ctx.runMutation(internal.draftCoach.patchDraftMessage, ...)`, set `status: "COMPLETE"` with `tokens` count in the final write via `ctx.runMutation(internal.draftCoach.completeDraftMessage, ...)`. `recordSpend` is called once after the final COMPLETE write, using the `usage` field from the end-of-stream event (never estimate during streaming).

**Cost limit check timing**: `checkCostLimit` is called once at the top of the handler, before any Claude invocation. If `{ softCapReached: true, limited: false }` is returned, Draft Coach does **not** apply soft-cap degradation (that path is Coach-specific per T17b). Draft Coach either proceeds normally (`limited: false`) or throws `RATE_LIMITED` (`limited: true`).

**Convex function-type reminder**: actions call `ctx.runMutation()` for DB writes — never write directly. Each `ctx.runMutation()` during streaming is a separate round-trip; this is expected and is not an atomicity problem.

**Model constants**: import `CLAUDE_SONNET` (e.g. `"claude-sonnet-4-5"`) from `convex/lib/models.ts` — do not inline the model string.

**Deployment sequencing note (from T18)**: T18's `sendMessage.ts` schedules `internal.draftCoach.generateResponse` by reference. If deploying T18 before T19 is merged, include a minimal stub `internalAction` at `convex/draftCoach/generateResponse.ts` that throws `new ConvexError("not yet implemented")` so Convex accepts the deployment. T19 replaces the stub. Do NOT leave the reference dangling.

**Queries used**
- `checkCostLimit(ctx, caseId)` from `convex/lib/costTracking.ts` — called before any Claude invocation
- `recordSpend(ctx, caseId, tokensUsed)` from `convex/lib/costTracking.ts` — called after streaming COMPLETE
- `assemblePrompt({ role: "DRAFT_COACH", caseId, actingUserId, recentHistory })` from `convex/lib/prompts.ts`
- `internal.cases.getPinnedTemplateVersion` — fetches the `templateVersions` row for `draftCoachInstructions`
- `internal.draftCoach.mutations.insertStreamingRow` — inserts `draftMessages` row with `status: "STREAMING"`
- `internal.draftCoach.mutations.patchDraftMessage` — batch-patches content during streaming
- `internal.draftCoach.mutations.completeDraftMessage` — sets `status: "COMPLETE"` + `tokens`
- `internal.draftCoach.mutations.writeFinalDraft` — writes `draftSessions.finalDraft` on readiness detection

**Invariants**
- `checkCostLimit` is always the first call in the handler; no Claude invocation precedes it
- Context passed to `assemblePrompt` contains only `actingUserId`'s data; the other party's `synthesisText` and `privateMessages` are never present
- `detectReadiness` is a pure function: same input always produces same output, no side effects
- `recordSpend` is called exactly once per action invocation that reaches COMPLETE
- `draftSessions.finalDraft` is set only when `detectReadiness` returns true and JSON parse succeeds
- The streaming row is always inserted before any Claude call, so the UI can show a STREAMING state immediately
- The action is an `internalAction` (never directly callable from the client); T18's `sendMessage` schedules it via `ctx.scheduler.runAfter(0, internal.draftCoach.generateResponse, { draftSessionId, caseId })`

**Non-goals**
- Does not implement the Draft Coach panel UI (`DraftCoachPanel.tsx`, `DraftReadyCard.tsx`) — that is T18
- Does not implement cost cap threshold logic — that is T17c; this task only calls the already-implemented `checkCostLimit`
- Does not apply the privacy filter (`privacyFilter.checkPrivacyLeak`) — context isolation via `assemblePrompt(role=DRAFT_COACH)` is the privacy mechanism for this role
- Does not handle the `draftCoach/sendFinalDraft` or `draftCoach/discardSession` mutations — those are T18
- Does not implement Coach facilitation or joint-chat routing — those are T17a/T17b
