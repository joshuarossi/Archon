T3 is the structural foundation for every Phase 1 UI task. Without a working React + Vite + Tailwind + shadcn/ui shell wired to Convex and React Router, none of the feature pages in P1 have a place to live. This task initialises the client project, installs the full UI stack, and creates stub route components for all 13 pages defined in tech spec §9.2 so feature work can begin on a predictable skeleton. It also configures TypeScript strict mode and confirms the dev server starts cleanly — acting as the "does the project even run?" gate for Phase 0.

## Acceptance Criteria

- All route shells from tech spec §9.2 exist as `.tsx` files: LandingPage, LoginPage, InviteAcceptPage, Dashboard, NewCasePage, CaseDetail, PrivateCoachingView, ReadyForJointView, JointChatView, ClosedCaseView, admin/TemplatesListPage, admin/TemplateEditPage, admin/AuditLogPage.
- React Router v6 routes are wired in `App.tsx`; navigating to each route renders the shell without a blank screen or error.
- Authenticated routes redirect to `/login` when there is no valid session.
- shadcn/ui Button, Card, Input, Select, and Textarea primitives are importable and render correctly.
- Tailwind CSS purge is configured; `npm run build` produces a bundle without Tailwind class bloat.
- `ConvexProvider` wraps the app; `useQuery` and `useMutation` hooks are importable from `convex/react`.
- `tsc --noEmit` passes with strict mode on.

## Implementation Notes

**Vite + TypeScript setup.** Use `@vitejs/plugin-react` for JSX. Configure `resolve.alias: { '@': path.resolve(__dirname, 'src') }` in `vite.config.ts` so imports use `@/` rather than deep relative paths. Mirror the alias with `"paths": { "@/*": ["./src/*"] }` in `tsconfig.json`. The Convex backend lives in `convex/` — never import from there in `src/`; it would bloat the browser bundle and fail at runtime.

The `vite.config.ts` must also include the Vitest test block (used by the CI `unit` job from T4):

```ts
// vite.config.ts
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';
import path from 'path';

export default defineConfig({
  plugins: [react()],
  resolve: { alias: { '@': path.resolve(__dirname, 'src') } },
  test: {
    environment: 'jsdom',
    globals: true,
    setupFiles: ['./src/test-setup.ts'],
  },
});
```

`src/test-setup.ts` imports `@testing-library/jest-dom` to activate the custom DOM matchers; it must exist even as a one-liner at this stage.

**TypeScript strict mode.** `tsconfig.json` must enable `"strict": true`, `"noUncheckedIndexedAccess": true`, and `"moduleResolution": "bundler"` (required for Vite's ESM + dynamic import model). `"jsx": "react-jsx"` and `"target": "ES2022"` complete the frontend config. The Convex codegen will write `convex/_generated/api.ts` and `convex/_generated/dataModel.ts`; rely on `Doc<"tableName">` and `Id<"tableName">` from those files rather than hand-crafting DB-row interfaces. `convex/tsconfig.json` typically overrides `moduleResolution` to `"node16"` for CJS/ESM interop; match that pattern.

```json
// tsconfig.json
{
  "compilerOptions": {
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "moduleResolution": "bundler",
    "jsx": "react-jsx",
    "target": "ES2022",
    "paths": { "@/*": ["./src/*"] }
  }
}
```

**Tailwind + shadcn/ui.** Configure `tailwind.config.ts` with `darkMode: ['class']` (required by shadcn) and `content: ['./index.html', './src/**/*.{ts,tsx}']`. Missing content globs is the single most common cause of "classes work in dev but vanish in production." `globals.css` must define the shadcn CSS custom properties (`--background`, `--foreground`, `--primary`, etc.) inside `@layer base`, plus all project-specific tokens from the design system (`--bg-canvas`, `--accent`, `--coach-accent`, etc.). Install shadcn primitives via CLI: `npx shadcn-ui@latest add button card input select textarea` — this correctly pulls Radix peer deps and writes component files under `src/components/ui/`.

**Convex + auth wiring.** `main.tsx` should wrap the app in `ConvexProvider` (from `convex/react`) using `VITE_CONVEX_URL` from the env. The Convex Auth provider (`ConvexAuthProvider` from `@convex-dev/auth/react`) should wrap the router — exact import path depends on the version pinned, but it must sit inside `ConvexProvider`. The `ConvexProvider` must be the outermost wrapper; everything else nests inside it.

**React Router v6 route tree.** Wire all 13 routes in `App.tsx` using `<Routes>` + `<Route>`. Protected routes (anything requiring authentication) should pass through a `ProtectedRoute` component that calls `useConvexAuth()` (or the equivalent Convex Auth hook), checks `isAuthenticated`, and renders `<Navigate to="/login" replace />` if false. Route shells are minimal: a heading with the page name is sufficient — they exist to confirm routing works and give feature tasks a file to expand.

**No data fetching in shells.** Route shells at this stage must not call `useQuery` or `useMutation`. That would create dependencies on the schema being deployed (T1) and auth being configured (T2), neither of which is a hard dependency for T3. Keep shells to pure rendering.

**Queries used**
- `useConvexAuth()` from `convex/react` — used by `ProtectedRoute` to check `isAuthenticated`; no DB query, just session state

**Invariants**
- `ConvexProvider` is the outermost wrapper in `main.tsx`; nothing above it can call Convex hooks
- `tailwind.config.ts` `content` array includes `'./src/**/*.{ts,tsx}'` and `'./index.html'`
- `tsconfig.json` has `"strict": true`, `"noUncheckedIndexedAccess": true`, and `"moduleResolution": "bundler"`
- All 13 route shells export a default function that returns a non-null JSX element
- No route shell file imports from `convex/` (except for hooks from `convex/react`)

**Non-goals**
- No real page content, data fetching, or business logic in any route shell
- No auth provider configuration (`convex/auth.ts`) — that is T2; this task only wires the client-side `ConvexAuthProvider` wrapper
- No Vitest or Playwright test files for page functionality — stubs only (`npm run test:unit` exits 0 with placeholder)
- No custom theme values beyond what shadcn CLI generates; token extensions (`--accent`, `--coach-accent`, etc.) are defined but colour values are the design-system defaults from the spec
