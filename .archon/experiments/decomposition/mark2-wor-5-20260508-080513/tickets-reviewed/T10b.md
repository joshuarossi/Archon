# Private coaching AI action — convex/privateCoaching/generateAIResponse.ts (US-06, US-07, FR-04)

Phase 1 establishes the core private coaching loop: a user sends a message, the AI coach responds in real time, and the response is visible only to that user. This task implements the Convex action that orchestrates the full AI call pipeline — cost check → compression → prompt assembly → streaming → spend recording — and wires the `PrivateCoachingView` React page to display the conversation using the shared `ChatWindow` component. It is the first live Claude integration in the codebase and sets the pattern every subsequent AI action will follow.

## Acceptance Criteria

- generateAIResponse action: before calling Claude, calls `checkCostLimit(ctx, caseId)` from `convex/lib/costTracking.ts`; if `{ limited: true }` is returned, throws `RATE_LIMITED` immediately without invoking Claude or inserting a new STREAMING row.
- Before constructing the message array, calls `compressIfNeeded(history, 60000)` from `convex/lib/compression.ts`; the returned (possibly compressed) array is what gets passed to `assemblePrompt` and onward to the SDK.
- Inserts a `privateMessages` row with `status: STREAMING`, calls `assemblePrompt` with role=PRIVATE_COACH on the (possibly compressed) history, calls Claude Sonnet (`claude-sonnet-4-5`) with streaming, batches token updates to the row every ~50ms, sets `status: COMPLETE` with `tokens` count on completion.
- After streaming completes successfully, calls `recordSpend(ctx, caseId, tokensUsed)` from `convex/lib/costTracking.ts` to accumulate token usage.
- PrivateCoachingView renders the party's private chat using the shared ChatWindow component (T9); only the calling user's private messages are fetched and rendered.
- On Claude API 429, the action retries once after 2 seconds; on second failure, sets message `status: ERROR`.
- On network timeout (>30s) or other non-retryable error, sets message `status: ERROR` with the applicable error code.
- A direct query for another user's private messages by any authenticated user returns FORBIDDEN (privacy enforcement at query layer).
- Vitest unit test: the Convex query `privateCoaching/messages` called by a user who is not the owner of the requested messages returns FORBIDDEN.
- Playwright (`privacy.spec.ts`, PRD §7.1 launch criterion #2, Test 1 — private message isolation): using two browser contexts, user B directly calls the `privateCoaching/messages` query for user A's caseId+userId; asserts the response is FORBIDDEN (hard access denial, not an empty array).
- Playwright: in solo mode, messages sent as initiator do not appear when toggled to invitee's private coaching view.

## Implementation Notes

`generateAIResponse` is a **Convex action** — the only function type that may call external APIs. It never writes to the DB directly; all DB mutations happen via `ctx.runMutation()`. Reads within the action (e.g., fetching the full message history for the compression step) must go through `ctx.runQuery()` — direct DB access is not available inside a Convex action. The action follows a strict ordered pipeline:

1. **Cost guard** — call `checkCostLimit(ctx, caseId)` from `convex/lib/costTracking.ts`. If `limited: true`, throw `createError('RATE_LIMITED', ...)` immediately. No STREAMING row should be inserted yet; failing early saves a wasted DB write.
2. **User message persistence** — insert the user's `privateMessages` row (`role: 'user'`, `status: 'COMPLETE'`) via `ctx.runMutation()`.
3. **History fetch** — retrieve existing `privateMessages` rows via `ctx.runQuery()` (not a direct DB read; actions require this indirection). Pass the full array to the next step.
4. **Compression** — call `compressIfNeeded(history, 60000)` from `convex/lib/compression.ts` on the full message history. Pass the returned array (possibly containing a leading summary message) to `assemblePrompt`. Do not pass the raw history.
5. **Prompt assembly** — call `assemblePrompt({ role: 'PRIVATE_COACH', caseId, actingUserId, recentHistory })` from `convex/lib/prompts.ts`. The returned `{ system, messages }` is the complete Claude payload.
6. **STREAMING row insertion** — insert a `privateMessages` row with `status: 'STREAMING'` before the `client.messages.stream()` call via `ctx.runMutation()`. This row's `_id` is the target for all subsequent patch mutations during streaming.
7. **Claude streaming** — use `client.messages.stream()` with model `SONNET` (from `convex/lib/models.ts`; do not inline the string). Accumulate token deltas; batch-write content to the STREAMING row every ~50ms via `ctx.runMutation()`. On stream end, set `status: 'COMPLETE'` and write the final `tokens` count from `usage.output_tokens ?? 0` (`noUncheckedIndexedAccess`: use nullish coalescing since the SDK type does not guarantee presence on every event).
8. **Spend recording** — call `recordSpend(ctx, caseId, tokensUsed)` from `convex/lib/costTracking.ts` after the Claude call completes successfully.
9. **Error handling** — on 429, wait 2 seconds (use `await new Promise(r => setTimeout(r, 2000))`) and retry once. On second 429, or on any 400/401/timeout, call `ctx.runMutation()` to set the STREAMING row's `status: 'ERROR'`. Never retry 401 (invalid key) or 400 (bad request).

**Privacy enforcement lives at the query layer.** The `privateCoaching/messages` query must resolve the caller's identity via `requireAuth(ctx)` and then compare the authenticated userId against the `partyStates` owner for the requested `caseId`. Any mismatch throws `createError('FORBIDDEN', ...)`. This must be enforced server-side even if the client never calls it wrong — it is the PRD §7.1 launch criterion #2 boundary.

**Solo mode isolation**: in solo mode, both `partyStates` rows belong to the same userId, but are distinguished by `role: 'INITIATOR'` vs `role: 'INVITEE'`. The `privateCoaching/messages` query must therefore accept an optional `as` argument (`'INITIATOR' | 'INVITEE'`) and use it as the partyStates discriminator when resolving which messages to return. Without this discriminator the query cannot distinguish the two parties when `userId` is identical. `PrivateCoachingView` derives `as` from the `?as=` URL search param and passes it to `useQuery(api.privateCoaching.messages, { caseId, as: soloRole })`. In non-solo mode the `as` field is omitted; the query then falls back to the unique `partyStates` row for the caller's `userId + caseId`.

**Model constants** must live in `convex/lib/models.ts` (create if not already present):
```typescript
export const SONNET = 'claude-sonnet-4-5';
export const HAIKU  = 'claude-haiku-4-5';   // reserved for Phase 2 joint-coaching nodes
```
Import from there — never inline model strings. `HAIKU` is declared now to establish the file and avoid a separate schema-touching PR when Phase 2 joint-coaching tasks (T17a/T17b/T17c) land.

**Client-side hook**: `generateAIResponse` is a Convex **action**, not a mutation. The PrivateCoachingView must call it via `useAction(api.privateCoaching.generateAIResponse)`, not `useMutation`. Using `useMutation` for an `action()`-wrapped function is a Convex type error.

**Cross-ticket symbols**:
- `assemblePrompt` imported from `convex/lib/prompts.ts` — owned by T10a; signature `assemblePrompt(opts: AssemblePromptOptions): { system: string; messages: ClaudeMessage[] }`.
- `compressIfNeeded` imported from `convex/lib/compression.ts` — owned by T10c; call site is `compressIfNeeded(history, 60000)`.
- `checkCostLimit` / `recordSpend` from `convex/lib/costTracking.ts` — owned by T10d.

**PrivateCoachingView** is a React route component that wraps the shared `ChatWindow` from T9. It subscribes to `useQuery(api.privateCoaching.messages, { caseId, as: soloRole })` and passes the result as the `messages` prop. The `onRetry` callback re-invokes `useAction(api.privateCoaching.generateAIResponse)` with the failed message's id. It does not manage any streaming state itself; the reactive query handles updates automatically as the STREAMING row is patched.


**Queries used**
- `api.privateCoaching.generateAIResponse` — called via `useAction` in PrivateCoachingView
- `api.privateCoaching.messages` — called via `useQuery` in PrivateCoachingView (with `{ caseId, as: soloRole }`)
- `checkCostLimit(ctx, caseId)` from `convex/lib/costTracking.ts` — called at action start
- `recordSpend(ctx, caseId, tokensUsed)` from `convex/lib/costTracking.ts` — called after streaming completes
- `compressIfNeeded(history, 60000)` from `convex/lib/compression.ts` — called before `assemblePrompt`
- `assemblePrompt({ role: 'PRIVATE_COACH', caseId, actingUserId, recentHistory })` from `convex/lib/prompts.ts` — called after compression
- `client.messages.stream(...)` from `@anthropic-ai/sdk` — only inside the action, model `SONNET`

**Invariants**
- `checkCostLimit` is always called before any Claude invocation; if `limited: true`, no STREAMING row is created and no Anthropic call is made.
- `compressIfNeeded` is always called before `assemblePrompt`; the compressed (or original) history is what reaches the SDK.
- The STREAMING row is inserted before `client.messages.stream()` starts; it is always resolved to `COMPLETE` or `ERROR` — never left in `STREAMING` state.
- `recordSpend` is called if and only if the Claude call completes without error.
- `messages` query returns only rows where the authenticated caller is the party owner of the requested `caseId`; any other caller receives `FORBIDDEN`, not an empty array.
- In solo mode, the `as` argument disambiguates INITIATOR vs INVITEE partyStates for the same userId; each party's messages are fetched and displayed independently.
- `client.messages.stream()` is called with the string value of `SONNET` constant — never an inlined model string.

**Non-goals**
- No joint messages (`jointMessages` table is untouched by this task).
- No synthesis triggering (that is T11's `markComplete` mutation).
- No soft-cap detection or boilerplate response (that is T17b/T17c in Phase 2; `checkCostLimit` in Phase 1 returns only `limited` boolean).
- No compression implementation (T10c owns `compressIfNeeded`).
- No prompt assembly changes (T10a owns `assemblePrompt`).
- No privacy filter (`privacyFilter.ts` is not called by private coaching — only by synthesis and Coach in Phase 2).
- No audit log writes for normal coaching (audit log is written on auth errors per tech spec §6.5, but not for normal 429 retries).
