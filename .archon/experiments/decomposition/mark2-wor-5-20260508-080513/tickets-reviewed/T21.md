This task implements the automated abandoned-case cleanup job for Conflict Coach, satisfying FR-11. As Phase 2 completes the joint-chat surface, cases that enter `JOINT_ACTIVE` status can stall indefinitely if both parties disengage — without a cleanup mechanism this leaves the database cluttered with zombie cases and produces incorrect dashboard counts. A daily Convex cron scans for joint sessions that have been inactive for more than 30 days and closes them gracefully through the established state machine, ensuring data integrity and consistent case lifecycle behavior across the application.

## Acceptance Criteria

- `convex/crons.ts` registers a daily cron that runs `checkAbandonedCases`.
- `checkAbandonedCases` action queries for all cases with `status: JOINT_ACTIVE` and `updatedAt < now - 30 days`.
- Each matching case is transitioned to `CLOSED_ABANDONED` via `stateMachine.transitionCase`; `closedAt` is set to the current timestamp.
- Cases in any other status (DRAFT_PRIVATE_COACHING, BOTH_PRIVATE_COACHING, READY_FOR_JOINT, any CLOSED status) are NOT touched by this cron.
- The mutation inside the cron is idempotent: running it twice on the same case does not error (already CLOSED_ABANDONED cases are skipped).
- Vitest unit test: given a set of cases with various `updatedAt` values, the abandonment logic correctly identifies cases older than 30 days and skips newer ones.

## Implementation Notes

**Convex function types.** `checkAbandonedCases` must be an `internalAction` — not a mutation — because it iterates a query result and fans out to multiple per-case writes. Convex mutations cannot loop over results and issue further writes without hitting transactional limits; an action using `ctx.runQuery()` + repeated `ctx.runMutation()` is the correct pattern. Each per-case close should be a separate `internalMutation` so that each status change is atomic and a failure on one case does not roll back progress on others.

**Scheduling.** Register the daily job in `convex/crons.ts` using `cronJobs()` from `convex/server`. A UTC off-peak hour (e.g., `{ hourUTC: 2, minuteUTC: 0 }`) is appropriate. Per the Convex scheduling rule: `ctx.scheduler.runAfter()` fires even if the calling context errors, so ensure `checkAbandonedCases` is fully idempotent.

**Index requirement.** The `cases` table must have an index on `status` (e.g., `by_status`) to avoid a full-table scan at query time. Convex requires explicit indexes for any non-`_id` filter predicate — a missing index causes a runtime error. Check the schema from T1; if `by_status` does not exist, add it. The `updatedAt` cutoff can be applied as a second filter on the indexed query or as a post-filter on the fetched rows.

**`listJointActiveBefore` query.** Add `listJointActiveBefore` as an `internalQuery` in `convex/cases.ts`. It accepts `{ cutoffMs: number }` and returns `Id<"cases">[]` — all case IDs where `status === "JOINT_ACTIVE"` and `updatedAt < cutoffMs`. This query uses the `by_status` index; the `updatedAt` filter may be applied either as an index range (if a compound `by_status_updated` index exists) or as a post-fetch filter. The query must not be callable externally — use `internalQuery`.

**`closedAt` write location.** Set `closedAt` inside the `internalMutation` that performs the `stateMachine.transitionCase` call — not in the action — so the timestamp is captured atomically alongside the status change.

**Idempotency.** The state machine from T1 is the authority on valid transitions. Review whether `transitionCase` is a no-op or throws when called on an already-`CLOSED_ABANDONED` case. If it throws `CONFLICT`, the per-case mutation must guard with a status check before calling `transitionCase`, or catch and discard the `CONFLICT` error specifically. This avoids a failed second run cascading into noise.

**Time control in tests.** Use `vi.useFakeTimers()` to set a deterministic `Date.now()` so the 30-day boundary is testable without real-time passage. Seed cases with explicit `_creationTime`/`updatedAt` values relative to the faked clock. Use `convex-test` for any test that imports from `convex/server` or `convex/values` — do not mock Convex internals.

- **Signatures**
  ```typescript
  // convex/crons.ts
  import { cronJobs } from "convex/server";
  import { internalAction, internalMutation } from "./_generated/server";
  import { v } from "convex/values";
  import { internal } from "./_generated/api";

  export const checkAbandonedCases = internalAction({
    args: {},
    handler: async (ctx): Promise<void> => {
      const cutoffMs = Date.now() - 30 * 24 * 60 * 60 * 1000;
      const staleCaseIds = await ctx.runQuery(
        internal.cases.listJointActiveBefore,
        { cutoffMs },
      );
      for (const caseId of staleCaseIds) {
        await ctx.runMutation(internal.crons.closeAbandonedCase, { caseId });
      }
    },
  });

  export const closeAbandonedCase = internalMutation({
    args: { caseId: v.id("cases") },
    handler: async (ctx, { caseId }): Promise<void> => {
      const c = await ctx.db.get(caseId);
      if (!c || c.status === "CLOSED_ABANDONED") return; // idempotent guard
      await transitionCase(ctx, caseId, "CLOSED_ABANDONED", {});
      await ctx.db.patch(caseId, { closedAt: Date.now() });
    },
  });

  const crons = cronJobs();
  crons.daily(
    "check abandoned cases",
    { hourUTC: 2, minuteUTC: 0 },
    internal.crons.checkAbandonedCases,
  );
  export default crons;

  // convex/cases.ts (addition)
  import { internalQuery } from "./_generated/server";
  import { v } from "convex/values";
  import { Id } from "./_generated/dataModel";

  export const listJointActiveBefore = internalQuery({
    args: { cutoffMs: v.number() },
    handler: async (ctx, { cutoffMs }): Promise<Id<"cases">[]> => {
      const rows = await ctx.db
        .query("cases")
        .withIndex("by_status", (q) => q.eq("status", "JOINT_ACTIVE"))
        .collect();
      return rows
        .filter((c) => c.updatedAt < cutoffMs)
        .map((c) => c._id);
    },
  });
  ```

- **Queries used**
  - `internal.cases.listJointActiveBefore({ cutoffMs: number }): Id<"cases">[]` — internal query returning IDs of all cases with `status === "JOINT_ACTIVE"` and `updatedAt < cutoffMs`; requires `by_status` index on `cases`
  - `internal.crons.closeAbandonedCase({ caseId: Id<"cases"> })` — per-case mutation called from `checkAbandonedCases`
  - `transitionCase(ctx, caseId, "CLOSED_ABANDONED", options)` — from `convex/stateMachine.ts` (T1)

- **Invariants**
  - Only `JOINT_ACTIVE` cases with `updatedAt < Date.now() - 30 days` are mutated
  - `closedAt` is always set to `Date.now()` at the time the mutation runs
  - Already-`CLOSED_ABANDONED` cases are skipped without throwing
  - No case in any other status is ever written to by this cron
  - Running the cron multiple times produces the same final state (idempotent)

- **Non-goals**
  - No email, push, or in-app notification to parties on auto-close
  - No admin UI or manual trigger endpoint for the cron
  - No handling of statuses other than `JOINT_ACTIVE`
  - No backfill of historically stale cases from before this cron was deployed
