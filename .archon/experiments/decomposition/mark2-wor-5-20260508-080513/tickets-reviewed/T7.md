# Invite acceptance flow — InviteAcceptPage + redeem mutation + invitee form (US-04)

Phase 1 of Conflict Coach requires a second party to join a case before joint work can begin. This task builds the complete invite-acceptance surface: the landing page a recipient sees when they open an invite URL, the atomic `invites/redeem` mutation that binds them to the case, and the follow-up form that captures their perspective before routing them into private coaching. It sits between T6 (which generates the token) and T10b (which delivers AI coaching), and directly enables the two-party flow the product is built around.

## Acceptance Criteria

- Navigating to `/invite/:token` when logged out shows the landing page with a 'Sign in to accept' button; after auth, the user is returned to `/invite/:token` and sees the Accept prompt.
- The landing page shows: what Conflict Coach is, the case category, the initiator's display name, and the initiator's stated main topic. It does NOT show the initiator's description or desired outcome.
- Clicking 'Accept' calls `invites/redeem(token)` which: validates the token is `ACTIVE` (throws `TOKEN_INVALID` if not), sets `cases.inviteeUserId`, creates a `partyStates` row for the invitee with role `INVITEE` and no form fields yet, marks the token `CONSUMED` (atomically in one Convex mutation), and transitions the case via `stateMachine.ts` to `BOTH_PRIVATE_COACHING`.
- After redeeming, the invitee is shown their own form (mainTopic, description, desiredOutcome fields) which submits via `cases/updateMyForm`; all three fields are required; after submission the invitee is redirected to PrivateCoachingView.
- `cases/updateMyForm` validates the caller is the invitee of this case and the case is in `DRAFT_PRIVATE_COACHING` or `BOTH_PRIVATE_COACHING` status; otherwise throws `FORBIDDEN` or `CONFLICT`.
- Reusing a consumed token shows an error message with login/dashboard options instead of an Accept button.
- A user who is already a party to the case cannot re-accept.
- Playwright: two browser contexts — user A creates case, copies invite link; user B opens link, registers, accepts, fills their form, and is redirected to PrivateCoachingView; both users see the case on their dashboards with correct status.

## Implementation Notes

**Auth redirect and token survival.** The `convex-auth` stack rule is explicit: OAuth redirects to Google and back, wiping `location.state` and any in-memory navigation state. The invite token must be stashed in `localStorage` before `signIn("google")` is called, and consumed in a `useEffect` on the return landing. Magic-link flow is simpler (same-origin redirect) but the same `localStorage` guard should apply for consistency.

**Atomic redeem mutation.** The `convex` stack rule forbids external I/O in mutations and requires `ctx.runMutation()` from actions for any non-transactional side effects. `invites/redeem` is a pure mutation — no Claude calls — so all four writes (update `cases.inviteeUserId`, insert `partyStates` row, mark `inviteTokens` row `CONSUMED`, call `transitionCase(ctx, caseId, "BOTH_PRIVATE_COACHING", ...)`) must occur in one mutation handler. Because Convex mutations are ACID, partial execution is not possible; either all four writes land or none do. `transitionCase` from `convex/stateMachine.ts` (T1) handles precondition validation and throws `CONFLICT` if the case is in an unexpected state. `invites/redeem` calls `requireAuth(ctx)` at the top of its handler; unauthenticated callers throw `UNAUTHENTICATED` before any reads.

**Landing page data.** `invites/getInviteInfo` is a reactive query that reads the `inviteTokens` row, then joins `cases` for category and `partyStates` (initiator) for display name and `mainTopic`. It must expose only these four fields — never `description` or `desiredOutcome` from the initiator's `partyStates`. Return a union type (`ACTIVE | CONSUMED | NOT_FOUND`) so the page can render the correct state without a separate error path.

**Invitee form.** After redeem succeeds, the component renders an inline form (not a new route) with `mainTopic`, `description`, and `desiredOutcome` fields. All three are required; inline validation errors below each field using `--danger` at 13px per the component-patterns rule. The `description` and `desiredOutcome` fields carry the "Private to you." label (12px, `--text-muted`, below the field label) per the tone-and-copy design rule — same treatment as the corresponding fields in `NewCasePage` (T6). On submit call `cases/updateMyForm`; on success call `useNavigate` to `/cases/:caseId`. The `cases/updateMyForm` mutation must verify the caller is the case invitee (not initiator) via `requireAuth(ctx)` and a `cases` row lookup — never trust client-passed role claims.

**"Already a party" guard.** Inside `invites/redeem`, after loading the `cases` row, check whether the authenticated user's ID already equals `cases.initiatorUserId` or `cases.inviteeUserId`. If so, throw `FORBIDDEN`. This covers both the case where the initiator accidentally opens their own link and where the invitee somehow re-triggers the flow.

**UI and design tokens.** The landing page is a centered card (max-width `400px`) with `--bg-surface` background, `--shadow-1` shadow, `--radius-lg` corners — matching the Login/invite card layout spec. The Accept button uses the `primary` variant. The invitee form follows the same layout rules as `NewCasePage` (T6): `max-w-[720px]`, `gap-4` between fields, error messages in `--danger` at 13px. Error states for consumed or not-found tokens follow the tone-and-copy rule: "This invitation link has already been used. If you have an account, sign in to access your dashboard. If you need a new link, ask [initiator name] to resend one." — never expose `TOKEN_INVALID` to the user.

**Routing.** `InviteAcceptPage` reads `token` from `useParams()`. With `noUncheckedIndexedAccess: true`, `useParams()` returns `string | undefined` — validate the token is defined at the route boundary before passing to Convex. `useNavigate()` handles all programmatic transitions; no `<a href>` for internal navigation per the react-router anti-pattern rule.

**Signatures**

```typescript

  args: { token: v.string() },
  handler: async (ctx, { token }: { token: string }): Promise<Id<"cases">> => { ... }

  // Throws TOKEN_INVALID if token is not ACTIVE or does not exist.
  // Throws FORBIDDEN if caller is already initiatorUserId or inviteeUserId.

  //             marks inviteTokens row CONSUMED, calls transitionCase(ctx, caseId, "BOTH_PRIVATE_COACHING").
});

// convex/cases/updateMyForm.ts
export const updateMyForm = mutation({
  args: {
    caseId: v.id("cases"),
    mainTopic: v.string(),
    description: v.string(),
    desiredOutcome: v.string(),
  },
  handler: async (
    ctx,
    args: { caseId: Id<"cases">; mainTopic: string; description: string; desiredOutcome: string }
  ): Promise<null> => { ... }
  // Throws UNAUTHENTICATED if not logged in.
  // Throws FORBIDDEN if caller is not the invitee of this case.
  // Throws CONFLICT if case status is not DRAFT_PRIVATE_COACHING or BOTH_PRIVATE_COACHING.
});
```

**Queries used**
- `api.invites.getInviteInfo` — loaded via `useQuery` on InviteAcceptPage; drives landing page content and consumed/not-found error states
- `api.invites.redeem` — called via `useMutation` when user clicks Accept
- `api.cases.updateMyForm` — called via `useMutation` on invitee form submit
- `internal.stateMachine.transitionCase` — called inside `redeem` to transition the case to `BOTH_PRIVATE_COACHING` (not called directly from the client)

**Invariants**
- All four writes in `redeem` (update cases row, insert partyStates, mark token CONSUMED, call transitionCase) are co-located in one Convex mutation — no partial commit is possible.
- `getInviteInfo` never exposes `initiatorPartyState.description` or `initiatorPartyState.desiredOutcome` in any response shape.
- A token with status `CONSUMED` or a token that does not exist must always produce `TOKEN_INVALID` from `redeem` — never silently succeed.
- `updateMyForm` must derive the caller's identity exclusively from `requireAuth(ctx)` — never trust a userId passed as a mutation argument.
- The invite token string in `localStorage` must be cleared immediately after being read and passed to the `/invite/:token` route, so it is not re-consumed on subsequent page loads.

**Non-goals**
- T7 does not implement private coaching AI responses (T10b).
- T7 does not build the Dashboard case list (T8).
- T7 does not implement solo mode or `PartyToggle` (T12).
- T7 does not send invite emails; the invite URL is shared manually by the initiator.
- T7 does not modify `inviteTokens` schema fields — those are defined in T1.
