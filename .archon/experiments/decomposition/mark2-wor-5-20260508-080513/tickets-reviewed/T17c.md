# Per-case AI cost tracking — real soft cap ($2) and hard cap ($10) enforcement (replaces T10d stub)

This task upgrades the placeholder cost-enforcement logic introduced in T10d into a fully operational cap system. In Phase 1, `checkCostLimit` always returned `{ limited: false }`; this task adds the real threshold checks so the platform can degrade gracefully at $2 and hard-stop at $10 without changing any caller signatures. It sits in Phase 2 because it depends on T17b having wired the soft-cap response path into the Coach action first — the enforcement function is only meaningful once a caller can act on the `softCapReached` flag.

## Acceptance Criteria

- The $2 soft cap threshold: when `cases.costUsd` crosses $2, `checkCostLimit` returns `{ softCapReached: true, limited: false }`; the threshold check is purely in `checkCostLimit` — reading and responding to this flag is the responsibility of each AI action caller (T17b for the Coach; T10b, T14, and T19 do not need to react to soft cap since degradation is Coach-specific).
- The $10 hard cap threshold: when `cases.costUsd` crosses $10, a mutation sets `cases.costLimited = true`; subsequent `checkCostLimit` calls return `{ limited: true }` causing all AI actions to throw `RATE_LIMITED` without calling Anthropic.
- The `costLimited` flag is persisted on the `cases` row; re-running any AI action after a service restart still reads the flag and fails immediately.
- `recordSpend` signature and behavior are unchanged from T10d (accumulates tokens and USD); this task adds the threshold checks inside `checkCostLimit`.
- Both functions remain callable from Convex actions via internal mutations.
- Vitest unit test: given a case with accumulated `costUsd` just above $2, `checkCostLimit` returns `{ softCapReached: true, limited: false }`.
- Vitest unit test: given a case with `costLimited: true`, `checkCostLimit` returns `{ limited: true }` without making any DB write.
- Vitest unit test: given a case flagged COST_LIMITED, invoking the synthesis action throws `RATE_LIMITED` without making any Anthropic API call.

## Implementation Notes

`convex/lib/costTracking.ts` was created by T10d with two exports: `checkCostLimit` and `recordSpend`. The Phase 1 stub reads `cases.costLimited` and returns `{ limited: true }` if set, `{ limited: false }` otherwise — it never sets `cases.costLimited` itself and never checks `cases.costUsd`. This task extends `checkCostLimit` to inspect `cases.costUsd` against the two thresholds.

**Convex function type constraints:** `checkCostLimit` is called from Convex actions, so it must perform its DB reads via `ctx.runQuery()` and its writes (the hard-cap flag mutation) via `ctx.runMutation()`. The mutation that sets `cases.costLimited = true` must be a separate registered Convex mutation in `convex/lib/costTracking.ts` (e.g., `internal.lib.costTracking.setCostLimited`); the action cannot write the DB directly. Similarly, the DB read for `costUsd` and `costLimited` must be performed via a registered internal query (e.g., `internal.lib.costTracking.getCostFields`) called with `ctx.runQuery()` — a bare exported JS function is not callable as a Convex function and will fail silently at runtime.

**Threshold constants:** Define `SOFT_CAP_USD = 2` and `HARD_CAP_USD = 10` as named constants at the top of `costTracking.ts`, not as magic numbers in the threshold checks. This keeps them trivially discoverable and easy to change.

**Evaluation order in `checkCostLimit`:** Check hard cap first — if `cases.costLimited` is already `true`, return `{ limited: true }` immediately without querying `costUsd`. Only if the hard-cap flag is absent should the function read `costUsd` and evaluate the soft-cap and hard-cap thresholds. This avoids an extra DB read for cases already hard-capped and keeps the function fast after the flag is set.

**Hard-cap threshold logic:** When `cases.costUsd >= HARD_CAP_USD` and `cases.costLimited` is not yet set, `checkCostLimit` must trigger a mutation to set `cases.costLimited = true`, then return `{ limited: true }`. This write is idempotent — the mutation should be a `patch` that only sets the flag if it isn't already true. Because `ctx.scheduler.runAfter` is not transactional, prefer calling the mutation directly via `ctx.runMutation()` inside the action rather than scheduling it.

**Return type extension:** The Phase 1 return type was `Promise<{ limited: boolean }>`. T10d's AC already specifies `Promise<{ softCapReached?: boolean, limited: boolean }>` as the intended signature — the `softCapReached` field is optional and absent in Phase 1. This task makes the field present (and `true`) when the $2 threshold is crossed but the $10 threshold is not. Callers that only destructure `limited` are unaffected.

**Testing approach:** Use `convex-test` for the Vitest tests so the Convex in-memory runtime handles the `cases` table reads and writes. For the synthesis action test (AC 8), mock `@anthropic-ai/sdk` via `vi.mock()` to confirm it is never called when `costLimited` is set. The existing Vitest tests in T10d (`recordSpend` accumulation, `checkCostLimit` with flag set) should still pass after this change; do not regress them.

## Test-Gen Brief

### Contract

**Files**
- Implementation: `convex/lib/costTracking.ts`
- Tests: `convex/lib/costTracking.test.ts`
- Synthesis action under test (for AC 8): `convex/synthesis/generate.ts`
- Synthesis action test: `convex/synthesis/generate.test.ts` (or co-located spec)

**Exports**
- `checkCostLimit` — named export, async function, callable from Convex actions.
- `recordSpend` — named export, async function (unchanged from T10d), callable from Convex actions.
- `setCostLimited` — named export, registered Convex `internalMutation`; called internally by `checkCostLimit` when the hard cap is crossed.
- `getCostFields` — named export, registered Convex `internalQuery`; reads `costUsd` and `costLimited` from the `cases` row; called internally by `checkCostLimit` via `ctx.runQuery()`. Must be wrapped with `internalQuery()` — a bare function is not callable as a Convex function.
- `SOFT_CAP_USD` — named constant export, `number`, value `2`.
- `HARD_CAP_USD` — named constant export, `number`, value `10`.

**Signatures**
```typescript
// convex/lib/costTracking.ts

export const SOFT_CAP_USD = 2;
export const HARD_CAP_USD = 10;

// Called from actions only
export async function checkCostLimit(
  ctx: ActionCtx,
  caseId: Id<"cases">
): Promise<{ softCapReached?: boolean; limited: boolean }>;

// Unchanged from T10d
export async function recordSpend(
  ctx: ActionCtx,
  caseId: Id<"cases">,
  tokensUsed: number
): Promise<void>;

// Internal mutation registered with Convex
export const setCostLimited = internalMutation({
  args: { caseId: v.id("cases") },
  handler: async (ctx, { caseId }) => { /* idempotent patch */ },
});

// Internal query registered with Convex — must use internalQuery(), not a bare function
export const getCostFields = internalQuery({
  args: { caseId: v.id("cases") },
  handler: async (ctx, { caseId }): Promise<{ costUsd: number; costLimited: boolean }> => {
    /* returns costUsd (default 0) and costLimited (default false) from the cases row */
  },
});
```

**Queries used**
- `ctx.runQuery(internal.lib.costTracking.getCostFields, { caseId })` — reads `costUsd` and `costLimited` from the `cases` row; defined as `getCostFields` (registered `internalQuery`) in the same file.
- `ctx.runMutation(internal.lib.costTracking.setCostLimited, { caseId })` — sets `cases.costLimited = true`.

**Invariants**
- If `cases.costLimited === true`, `checkCostLimit` always returns `{ limited: true }` and performs no DB write.
- If `cases.costUsd >= HARD_CAP_USD` and `costLimited` is not yet set, `checkCostLimit` writes the flag and returns `{ limited: true }` in the same call.
- If `cases.costUsd >= SOFT_CAP_USD` but `< HARD_CAP_USD`, `checkCostLimit` returns `{ softCapReached: true, limited: false }` and performs no DB write.
- If `cases.costUsd < SOFT_CAP_USD`, `checkCostLimit` returns `{ limited: false }` (no `softCapReached` field).
- `recordSpend` never modifies `cases.costLimited`; that field is owned exclusively by `setCostLimited`.
- Both threshold constants (`SOFT_CAP_USD`, `HARD_CAP_USD`) are referenced by name in the threshold checks — no magic numbers.
- `getCostFields` and `setCostLimited` are registered Convex functions (`internalQuery` and `internalMutation` respectively); neither is a bare exported function.

**Non-goals**
- This ticket does NOT change the `recordSpend` function behavior or signature.
- This ticket does NOT update any AI action caller (T10b, T14, T17b, T19); their signatures already handle the return type.
- This ticket does NOT implement the UI for `COST_LIMITED` cases (that is T26).
- This ticket does NOT reset or clear cost fields; there is no admin cost-reset feature in v1.
- This ticket does NOT change the Convex schema; the three cost fields (`costUsd`, `costTokens`, `costLimited`) were added in T1.
