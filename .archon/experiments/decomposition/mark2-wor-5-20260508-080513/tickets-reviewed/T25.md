This task closes out the NFR §5.4 accessibility requirement by auditing every view in the app and applying the fixes needed to reach WCAG 2.1 AA compliance. It exists in Phase 3 alongside the Playwright acceptance suite because accessibility failures caught here would also block the PRD §7.1 launch criteria: the axe-core CI check and the keyboard-only Playwright run are the automated gates that prove compliance. Without this pass, the app could ship with screen-reader-invisible chat regions, unlabeled icon buttons, or focus traps that leave keyboard users stranded at phase transitions.

## Acceptance Criteria

- Every route page has `<main>` landmark; navigation bar uses `<nav>`; Draft Coach panel uses `<aside>`.
- All chat message lists have `role='log'` and `aria-live='polite'`; new incoming messages are announced by screen readers without interrupting in-progress reading.
- Every `<button>` and interactive control has either visible text or a descriptive `aria-label`; no icon-only buttons without accessible name.
- All text-on-background combinations meet ≥ 4.5:1 contrast ratio (verified via axe-core in Playwright).
- When the user transitions from private coaching to the ReadyForJointView, focus moves to the ReadyForJointView's `<h1>` heading.
- When the user enters JointChatView from ReadyForJointView, focus moves to the message input.
- Playwright keyboard-only test: run the critical path (login → dashboard → new case → private coaching → mark complete) using only keyboard navigation (Tab, Enter, Space, Arrow keys); no step requires mouse.
- Playwright axe-core integration: zero critical or serious accessibility violations on Dashboard, PrivateCoachingView, JointChatView, and NewCasePage.

## Implementation Notes

**Landmark structure.** The `design_rules.accessibility` section states every page must have exactly one `<main>`. In practice this means auditing each route component (`Dashboard`, `NewCasePage`, `PrivateCoachingView`, `ReadyForJointView`, `JointChatView`) and ensuring they render `<main>` as the outermost semantic container. The shared `TopNav` component should already be `<nav>`, but verify it is not a bare `<div>`. The `DraftCoachPanel` (desktop right-side sheet, mobile full-screen bottom sheet) must be wrapped in `<aside>` — it is a complementary panel to the main chat, which is exactly the HTML semantics `<aside>` provides.

**Chat `aria-live` two-region pattern.** There are two distinct `aria-live` surfaces in `ChatWindow`:

1. `ChatWindow` itself renders `role="log" aria-live="polite"` on the message list container. Screen readers will announce new messages from this region passively.
2. A separate, visually-hidden `SrAnnouncer` component (rendered inside `ChatWindow`, `aria-live="polite" aria-atomic="true"`) fires discrete status messages at only two moments: when streaming begins ("Coach is replying") and when streaming ends (nothing announced). **Do not update this region on every token patch** — doing so causes over-announcement and interrupts ongoing screen-reader reading. The streaming cursor tick-patch pattern (DB row updated every ~50 ms) must never write to the `SrAnnouncer`.

The `SrAnnouncer` is visually hidden via Tailwind `sr-only` (clip, absolute position, 1×1px — never `display:none` or `visibility:hidden`, which silence `aria-live` in some browsers). This pattern is required by `design_rules.accessibility`: "announce once at `STREAMING → content starts`, and once at `COMPLETE`."

**Icon-only buttons.** The Send button, the Draft Coach trigger, any copy-to-clipboard or close buttons that render only a Lucide icon need an `aria-label` attribute. Audit using the axe-core `button-name` rule. Where a button already has visible text, no `aria-label` is needed. Where the label is in a `Tooltip`, the button still needs a direct `aria-label` because tooltips are not reliably exposed to all screen readers.

**Color contrast.** All token pairs defined in `design_rules.tokens` are stated to satisfy AA in both light and dark modes. If the axe-core run finds a violation, it will almost certainly be in custom compositions (e.g., muted timestamp text `--text-tertiary` on `--bg-surface-subtle`) rather than primary text. The fix is to adjust the specific component's text color to a higher-contrast token, not to change the token definition. Verify with axe-core's `color-contrast` rule in the Playwright check.

**Focus management on phase transitions — and all routes.** `design_rules.accessibility` specifies: "on navigation to a new route, move focus to the page's `<h1>`. Use a `useEffect` that calls `.focus()` on the heading ref after mount." This rule applies to **all** route components — Dashboard, NewCasePage, PrivateCoachingView, ReadyForJointView, and JointChatView — not only the two transitions explicitly gated in ACs 5 and 6. ACs 5 and 6 are the automated regression gates; the full rule is broader.

For the two specific transitions:
- **PrivateCoaching → ReadyForJointView** (AC5): add a `headingRef` on ReadyForJointView's `<h1>` with a mount-only `useEffect(() => { headingRef.current?.focus(); }, [])`.
- **ReadyForJointView → JointChatView** (AC6): focus goes to the message input (`<textarea>`) instead of a heading. Add an `inputRef` on `MessageInput`'s textarea and call `inputRef.current?.focus()` on JointChatView mount.

The heading and textarea must have `tabIndex={-1}` so they can receive programmatic focus without appearing twice in the tab order. Apply the canonical focus-ring class from `design_rules.accessibility` to these elements:
```
focus-visible:outline-2 focus-visible:outline-[color:var(--accent)] focus-visible:outline-offset-2
```

**`prefers-reduced-motion` for the streaming cursor.** `design_rules.motion` requires that the `StreamingIndicator` blink animation is disabled when `prefers-reduced-motion: reduce` is active. Apply Tailwind's `motion-safe:animate-[blink_500ms_step-end_infinite]` variant (or the equivalent CSS media query) so the cursor is rendered as a static bar (not blinking) under reduced-motion preferences. This ticket's axe-core run will flag a violation if the animation property itself causes issues; better to fix it preemptively.

**Keyboard-only Playwright test.** Use `page.keyboard.press('Tab')` to navigate and `page.keyboard.press('Enter')` or `Space` to activate. The test should never call `page.click()` or `page.fill()` via mouse coordinates — only `page.keyboard.type()` after Tab-navigating to an input and verifying focus with `page.evaluate(() => document.activeElement?.tagName)`. The shadcn/Radix components already handle keyboard interaction correctly; the main risks are custom icon buttons without keyboard support and modals that don't auto-focus their first child.

**axe-core integration.** Install `@axe-core/playwright` (`npm install --save-dev @axe-core/playwright`). In the Playwright spec, call `const results = await new AxeBuilder({ page }).analyze()` after navigating to each view. Assert `results.violations.filter(v => ['critical','serious'].includes(v.impact)).length` equals 0.

- `src/components/SrAnnouncer.tsx` — visually-hidden `aria-live="polite" aria-atomic="true"` element; updated only at stream start ("Coach is replying") and stream complete (cleared); never updated on token patches

- `e2e/accessibility.spec.ts` — new Playwright spec for keyboard-only test and axe-core checks

**Exports**
- No new public symbols are introduced. This ticket modifies JSX structure and adds ARIA attributes to existing exported components. `SrAnnouncer` is a new internal component used only by `ChatWindow` and is not exported from the package boundary.

**Queries used**
- None (this ticket makes no new Convex function calls)

**Invariants**
- Every route page has exactly one `<main>` element — never zero, never two.
- `ChatWindow` always renders with `role="log"` and `aria-live="polite"` on the message list container regardless of message count.
- `SrAnnouncer` text is updated at exactly two streaming lifecycle moments: stream start and stream complete. It is never updated on intermediate token patches.
- Icon-only buttons always carry `aria-label`; buttons with visible text do not need one.
- `tabIndex={-1}` on programmatically-focused headings/inputs ensures they appear in focus history on navigation but not twice in the tab order.
- `useEffect` focus calls run only on mount (`[]` dep array) — never on every render.
- The streaming cursor blink animation is wrapped in `motion-safe:` so it is suppressed under `prefers-reduced-motion: reduce`.

**Non-goals**
- This ticket does NOT implement dark mode (covered separately, if at all).
- This ticket does NOT rewrite the shared `ProtectedRoute` logic or routing structure.
- This ticket does NOT introduce new Convex functions or modify backend logic.
- This ticket does NOT add or modify design token values in `tailwind.config.ts` or `globals.css` — it only fixes component-level contrast issues by choosing higher-contrast tokens where violations occur.
- This ticket does NOT cover admin template views (out of the critical-path scope for the axe-core check).
