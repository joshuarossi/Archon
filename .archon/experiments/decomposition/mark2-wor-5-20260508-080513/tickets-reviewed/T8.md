# Dashboard + case detail routing (US-05, US-15)

Every party needs a reliable entry point into their cases and a routing hub that surfaces the correct phase-specific view without exposing data from other parties. T8 builds these two surfaces for Phase 1: `Dashboard.tsx`, which lists all the user's active and closed cases with a privacy-safe peek at the other party's phase-level status, and `CaseDetail.tsx`, which reads case status and delegates to `PrivateCoachingView`, `ReadyForJointView`, `JointChatView`, or `ClosedCaseView`. Both surfaces depend on T5 (auth) and T6 (case creation) being in place; downstream tasks (T10b, T11, T12, T15, T16) all rely on this routing layer being stable before they can wire their own views.

## Acceptance Criteria

- Dashboard lists all cases where the logged-in user is initiator or invitee, grouped into 'Active' and 'Closed' sections.
- Each case row displays: other party name (or 'Pending' if invitee hasn't joined), category, created date, current status label, and last activity date.
- The `cases/partyStates` query returns `{ self: PartyState, otherPhaseOnly: { hasCompletedPC: boolean } }` and nothing else about the other party.
- Clicking a case row navigates to `/cases/:caseId` which renders the correct subview: PrivateCoachingView for private coaching statuses, ReadyForJointView for READY_FOR_JOINT, JointChatView for JOINT_ACTIVE, ClosedCaseView for closed statuses.
- If a case's status requires the other party to have joined and they haven't, the case detail shows a waiting state with the invite link still accessible.
- Dashboard first meaningful paint occurs in under 1 second (measured via Playwright performance API in CI).
- Attempting to read another user's case detail via direct URL returns NOT_FOUND (or 404-equivalent redirect).

## Implementation Notes

**Convex queries.** `cases/list` is a reactive `query()` that runs two index scans ŌĆö one on `by_initiator` and one on `by_invitee` ŌĆö and merges the results. Both indexes must exist in `convex/schema.ts` (T1). Never full-table-scan. `cases/partyStates` is a second reactive query gated by case membership; it deliberately returns only `{ self: PartyState, otherPhaseOnly: { hasCompletedPC: boolean } }` to enforce privacy at the query layer, not the UI layer. `cases/get` is used by `CaseDetail` and must throw `NOT_FOUND` (via `errors.ts`) when the caller is neither initiator nor invitee. All three functions begin with `requireAuth(ctx)`.

**Reactive subscription pattern.** `Dashboard.tsx` calls `useQuery(api.cases.list)` ŌĆö no `useEffect` sync into `useState`. Convex pushes updates automatically; the component just re-renders. `CaseDetail` subscribes to `useQuery(api.cases.get, { caseId })` and uses the returned status to branch sub-view rendering. While the query is loading (`undefined` return), render the case detail skeleton.

**`useParams()` guard (TypeScript strictness).** Under `noUncheckedIndexedAccess` and `react-router`'s param types, `useParams<{ caseId: string }>().caseId` is `string | undefined`. Guard it before passing to `useQuery`:
```ts
const { caseId } = useParams<{ caseId: string }>();
if (!caseId) return null; // or redirect to /dashboard
const caseDoc = useQuery(api.cases.get, { caseId: caseId as Id<"cases"> });
```
Call `hooks.useCaseId()` if T3 provides a typed params hook; otherwise use the explicit guard above.

**Sub-view routing in CaseDetail.** The routing table is straightforward:
- `DRAFT_PRIVATE_COACHING` ŌåÆ `PrivateCoachingView` (invitee not yet joined ŌĆö show waiting banner with invite link)
- `BOTH_PRIVATE_COACHING` ŌåÆ `PrivateCoachingView`
- `READY_FOR_JOINT` ŌåÆ `ReadyForJointView`
- `JOINT_ACTIVE` ŌåÆ `JointChatView`
- `CLOSED_RESOLVED | CLOSED_UNRESOLVED | CLOSED_ABANDONED` ŌåÆ `ClosedCaseView`

Use a TypeScript `assertNever` helper to enforce compile-time exhaustiveness on the status switch:
```ts
function assertNever(x: never): never {
  throw new Error(`Unhandled case status: ${String(x)}`);
}
// In CaseDetail:
switch (caseDoc.status) {
  case "DRAFT_PRIVATE_COACHING":
  case "BOTH_PRIVATE_COACHING":
    return <PrivateCoachingView ... />;
  case "READY_FOR_JOINT":
    return <ReadyForJointView ... />;
  case "JOINT_ACTIVE":
    return <JointChatView ... />;
  case "CLOSED_RESOLVED":
  case "CLOSED_UNRESOLVED":
  case "CLOSED_ABANDONED":
    return <ClosedCaseView ... />;
  default:
    return assertNever(caseDoc.status);
}
```
In Phase 1 these sub-view components are stubs (T9, T10b, T11, T15, T16 fill them in); `CaseDetail` still must route correctly so downstream tasks can integrate cleanly.

**Status labels and glyphs.** The dashboard case row maps each status to one of the four status glyphs defined in the design system (states-and-feedback rule). Use:
- `ŌŚÅ` (green, `--accent`) ŌĆö "Your turn" (case is in a status where the viewer must act)
- `ŌŚŗ` (gray, `--text-tertiary`) ŌĆö "Waiting on the other party"
- `ŌŚÉ` (amber, `--warning`) ŌĆö "Ready for joint session" (`READY_FOR_JOINT`)
- `ŌŚ╝` (neutral, `--text-secondary`) ŌĆö Closed

Each glyph is wrapped in `<span aria-label="[status description]">` ŌĆö glyphs alone are inaccessible per the accessibility design rule.

**Glyph "Your turn" vs "Waiting" ŌĆö avoid per-row N+1 subscriptions.** Whether a viewer is on "Your turn" or "Waiting" depends on `partyStates.self` (e.g., has the viewer submitted their form?), not on `case.status` alone. Calling `api.cases.partyStates` for each dashboard row creates N reactive subscriptions and will fail the AC6 performance gate. Resolution: extend `cases/list` to include the viewer's own `PartyState` inline in each returned row (a projection, not a join ŌĆö the query already has the viewer's userId and can fetch their `partyStates` doc in the same request). The returned item type becomes `Doc<"cases"> & { selfPartyState: PartyState }`. The `otherPhaseOnly` field is NOT included in the list response ŌĆö only `self` ŌĆö to preserve the privacy boundary.

**Dashboard layout.** `max-w-[720px] mx-auto`, `px-4 md:px-8`. Two sections: "Active" and "Closed", each with a section heading. Case rows render as shadcn `Card` components with `--shadow-1` at rest. Load state: three skeleton rows using `animate-pulse` on `bg-[color:var(--bg-surface-subtle)]` blocks that match the real row shape. Empty state copy (design system tone-and-copy rule): `"No cases yet. When you're ready to work through something, start a new case."`

**Performance gate (AC6).** Playwright measures `performance.getEntriesByName('first-contentful-paint')` and asserts `< 1000ms`. The query must resolve fast enough to clear this bar ŌĆö ensure the Convex index scans are tight (no full-table merges, no N+1 lookups per row).

**Auth guard.** `CaseDetail` and `Dashboard` are inside the authenticated route tree established in T3. The Convex `cases/get` server-side NOT_FOUND guard is the hard enforcement; the client-side redirect is a UX shortcut only.

- `playwright/dashboard.spec.ts` ŌĆö Playwright specs for AC2, AC5, AC6, AC7
- `playwright/case-routing.spec.ts` ŌĆö Playwright spec for AC4 (or inline in `solo-full-flow.spec.ts`)

**Exports**
- `api.cases.list` ŌĆö named Convex query export (via `convex/_generated/api`)
- `api.cases.get` ŌĆö named Convex query export
- `api.cases.partyStates` ŌĆö named Convex query export
- `Dashboard` ŌĆö default React component export from `src/routes/Dashboard.tsx`
- `CaseDetail` ŌĆö default React component export from `src/routes/CaseDetail.tsx`

**Signatures**
```ts
// convex/cases/list.ts
// Returns each case with the caller's own PartyState embedded (avoids N+1 per-row partyStates calls).
// Throws UNAUTHENTICATED if not logged in.
export const list = query({
  args: {},
  handler: async (ctx): Promise<(Doc<"cases"> & { selfPartyState: PartyState })[]>
});

// convex/cases/get.ts
// Throws UNAUTHENTICATED if not logged in.
// Throws NOT_FOUND if caller is neither initiatorUserId nor inviteeUserId
// (callers must not be able to distinguish whether a case exists from whether they can access it).
export const get = query({
  args: { caseId: v.id("cases") },
  handler: async (ctx, { caseId }): Promise<Doc<"cases">>
});

// convex/cases/partyStates.ts
// Throws UNAUTHENTICATED if not logged in.
// Throws FORBIDDEN if caller is not a party; NOT_FOUND if case not found.
export const partyStates = query({
  args: { caseId: v.id("cases") },
  handler: async (
    ctx,
    { caseId }
  ): Promise<{
    self: PartyState;
    otherPhaseOnly: { hasCompletedPC: boolean };
  }>
});

// src/routes/CaseDetail.tsx
// Props: none (reads caseId from useParams(); guards undefined before passing to useQuery)
// Renders sub-view based on case.status with assertNever exhaustiveness check
export default function CaseDetail(): JSX.Element;
```

**Cross-ticket note.** `PartyState` in the signatures above refers to `Doc<"partyStates">` from T1's schema. If T1 exports a named `PartyState` type alias over `Doc<"partyStates">`, use that alias throughout for consistency with the plan. If T1 does not export a named alias, use `Doc<"partyStates">` directly and update the AC3 acceptance criterion text accordingly in your implementation comments.

**Queries used**
- `api.cases.list` ŌĆö called in `Dashboard.tsx` via `useQuery`
- `api.cases.get` ŌĆö called in `CaseDetail.tsx` via `useQuery`
- `api.cases.partyStates` ŌĆö called wherever per-party phase status is needed (NOT called per dashboard row ŌĆö see glyph N+1 note above)

**Invariants**
- `cases/list` never returns cases where the calling user is neither `initiatorUserId` nor `inviteeUserId`.
- `cases/get` throws `NOT_FOUND` (not `FORBIDDEN`) when the caller has no membership ŌĆö callers must not be able to distinguish whether a case exists from whether they can access it.
- `cases/partyStates.otherPhaseOnly` never contains raw message content, form field values, or synthesisText.
- `CaseDetail` renders the correct sub-view for every possible `case.status` value ŌĆö an unhandled status value is a compile-time exhaustiveness error via `assertNever`, not a runtime blank.
- `cases/list` includes `selfPartyState` inline so Dashboard never issues per-row reactive subscriptions to `partyStates`.

**Non-goals**
- Does not implement `PrivateCoachingView`, `ReadyForJointView`, `JointChatView`, or `ClosedCaseView` beyond stub placeholders (those are T9, T10b, T11, T15, T16).
- Does not implement the solo mode `PartyToggle` (T12).
- Does not build the case creation form (T6).
- Does not paginate the dashboard case list (v1 scope: all cases rendered reactively).
