# ReadyForJointView — synthesis display and 'Enter Joint Chat' CTA

`ReadyForJointView` is the bridge between private coaching and the joint session. Once both parties have completed their private coaching and the synthesis action (T14) has written personalized guidance to each party's `partyStates.synthesisText`, this view presents that guidance and offers the 'Enter Joint Chat' CTA. The view belongs to Phase 2 (P2): it is the first thing a user sees after the synthesis generation completes and serves as the gateway into the live, real-time joint session (T16). It enforces the privacy boundary between synthesis texts — each party sees only their own — and gates the `JOINT_ACTIVE` transition behind an explicit user action.

## Acceptance Criteria

- ReadyForJointView is rendered by CaseDetail when `case.status` is `READY_FOR_JOINT`.
- Shows the calling party's `synthesisText` (rendered as markdown) once `synthesisGeneratedAt` is set.
- While synthesis is not yet generated, shows a 'Generating your guidance…' loading state (spinner or skeleton).
- Shows only the calling party's synthesis; does NOT render the other party's synthesis text.
- 'Enter Joint Chat' button is shown below synthesis.
- Clicking 'Enter Joint Chat' calls a `jointChat/enterSession` mutation that transitions `case.status` from `READY_FOR_JOINT` to `JOINT_ACTIVE` (only on first entry; subsequent calls are idempotent).
- After entering, both parties see the JointChatView; the Coach posts an opening message grounded in the case's main topic.
- Playwright: in solo mode, after both parties complete private coaching, toggle to each party and verify their respective synthesis text appears (not each other's).

## Implementation Notes

`CaseDetail.tsx` already routes sub-views by `case.status` (established in T8). Add a `READY_FOR_JOINT` branch that renders `<ReadyForJointView caseId={caseId} />`. The status check is purely a React-side routing decision — no new mutation is needed for the CaseDetail branch itself.

`ReadyForJointView` subscribes to a query (`api.jointChat.mySynthesis`) that returns `{ synthesisText: string | null, synthesisGeneratedAt: number | null }` for the calling party's `partyStates` row. Because `synthesisGeneratedAt` is written at the end of the synthesis action (T14), the query value transitions from `null` to a real string without a page refresh — Convex's reactive push handles this automatically. Do not poll or use `setInterval`.

The loading state while synthesis is pending should use Tailwind's `animate-pulse` skeleton (per the `states-and-feedback` design rule) rather than a spinner. Skeleton blocks should mirror the card layout of the final synthesis display: a heading-height block, then 3–4 paragraph-height blocks.

Synthesis text from Claude is markdown. Use `react-markdown` to render it. Per the typography design rule, `##` headings inside the synthesis card should be rendered at `font-weight: 500` (not `700`) — apply a custom `components` override to `ReactMarkdown` if needed. The synthesis card itself should use `--bg-surface`, `--border-default`, `--radius-lg`, `--shadow-1`, and `p-6` (reading-heavy card padding), consistent with the `component-patterns` design rule.

`jointChat/enterSession` is a Convex **mutation** (not an action), because it performs only a DB write: it calls `stateMachine.transitionCase(ctx, caseId, 'JOINT_ACTIVE')`. Guard with `requireAuth(ctx)` and verify the caller is a party to this case. The idempotency contract: if `case.status` is already `JOINT_ACTIVE`, return without error (do not re-call `transitionCase`). After transitioning, the Coach opening message is posted — this should be scheduled via `ctx.scheduler.runAfter(0, internal.jointChat.generateCoachResponse, { caseId })` rather than executed inline, keeping the mutation fast and consistent with the Convex convention of handing heavy work to actions. `generateCoachResponse` is implemented in T17a; `enterSession` is forward-coupled to it (see Non-goals).

Per the accessibility design rule, when `ReadyForJointView` mounts (route transition), focus should move to the view's `<h1>` heading. Use a `useEffect` with a ref on the heading to call `.focus()` after mount.

For the Playwright solo-mode test, `useSoloActingParty()` (T12) must resolve the correct party from the `?as=` query param before calling `api.jointChat.mySynthesis`. Each party's synthesis is stored on their own `partyStates` row, so toggling the `?as=` param changes which row is queried. `mySynthesis` accepts an optional `actingUserId` arg to support this pattern.

## Test-Gen Brief

### Contract

**Files**
- `src/routes/ReadyForJointView.tsx` — component implementation and export
- `src/routes/CaseDetail.tsx` — routes to `ReadyForJointView` on `READY_FOR_JOINT` status
- `convex/jointChat/enterSession.ts` — mutation that transitions to `JOINT_ACTIVE`
- `convex/jointChat/mySynthesis.ts` — query returning calling party's synthesisText + synthesisGeneratedAt
- `playwright/solo-full-flow.spec.ts` — E2E test covering AC7 and AC8

**Exports**
- `ReadyForJointView` — default export from `src/routes/ReadyForJointView.tsx`; React component
- `enterSession` — named Convex mutation export from `convex/jointChat/enterSession.ts`; registered with `mutation()`
- `mySynthesis` — named Convex query export from `convex/jointChat/mySynthesis.ts`; registered with `query()`

**Signatures**
```typescript
// src/routes/ReadyForJointView.tsx
export default function ReadyForJointView({ caseId }: { caseId: Id<"cases"> }): JSX.Element;

// convex/jointChat/enterSession.ts
export const enterSession = mutation({
  args: { caseId: v.id("cases") },
  handler: async (ctx, { caseId }): Promise<void>,
});

// convex/jointChat/mySynthesis.ts
export const mySynthesis = query({
  args: {
    caseId: v.id("cases"),
    actingUserId: v.optional(v.id("users")),
  },
  handler: async (ctx, args): Promise<{
    synthesisText: string | null;
    synthesisGeneratedAt: number | null;
  }>,
});
```

**APIs used**
- `api.jointChat.mySynthesis` — query; called by `ReadyForJointView` to get calling party's synthesis
- `api.jointChat.enterSession` — mutation; called on 'Enter Joint Chat' click
- `api.cases.get` (or `api.cases.partyStates`) — query; called by `CaseDetail` for status-based routing
- `ctx.scheduler.runAfter(0, internal.jointChat.generateCoachResponse, { caseId })` — scheduled inside `enterSession` to post Coach opening message (implemented in T17a)

**Invariants**
- `mySynthesis` returns only the calling party's row; a query for the other party's synthesis must return `FORBIDDEN`
- `enterSession` is idempotent: if `case.status` is already `JOINT_ACTIVE`, returns without error and does not call `transitionCase` again
- `ReadyForJointView` is never rendered when `case.status !== 'READY_FOR_JOINT'`; `CaseDetail` owns this gate
- Loading skeleton is shown while `synthesisGeneratedAt` is null; synthesis card is shown when it is set
- 'Enter Joint Chat' button is present in the rendered output whenever `synthesisText` is non-null (no separate gate)

**Non-goals**
- Does not implement synthesis generation logic (T14 owns that)
- Does not implement `JointChatView` content or the Coach opening-message action (T16/T17a–T17b); `enterSession` schedules `internal.jointChat.generateCoachResponse` as a forward reference to T17a — T15 must compile and run without T17a being complete, so the scheduler call should be conditional or the internal function registered as a stub in T17a's landing
- Does not handle any case status other than `READY_FOR_JOINT` (CaseDetail routes those to other views)
- Does not add a unit test for `enterSession` beyond idempotency (the stateMachine transition is tested in T1's Vitest spec)
