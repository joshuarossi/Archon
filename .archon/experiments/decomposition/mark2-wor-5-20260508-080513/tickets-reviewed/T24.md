# Audit log UI — AuditLogPage

The admin audit log page gives administrators a reverse-chronological view of all privileged actions taken within Conflict Coach — template creation, publishing, archiving, and any other actions written to the `auditLog` table. This view is the transparency and accountability surface for platform operators, and it exists in Phase 3 as a companion to the template management UI built in T23. Non-admin users must never be able to reach it.

## Acceptance Criteria

- AuditLogPage is accessible only to ADMIN users; non-admin navigating to `/admin/audit` is redirected.
- Page lists audit log entries from the `auditLog` table in reverse-chronological order.
- Each row shows: formatted timestamp, admin's display name (looked up from `users`), action string, target type, target ID.
- Page handles empty audit log state with an appropriate empty state message.
- Page loads and renders within 1 second for up to 200 audit log entries.
- Playwright `admin-templates.spec.ts`: after admin creates and publishes a template, audit log page shows the corresponding entries.

## Implementation Notes

**Route and role gate.** `AuditLogPage` lives at `/admin/audit`. The admin role guard follows the same pattern used in T23's `TemplatesListPage` and `TemplateEditPage`: `requireAuth(ctx)` → check `user.role === 'ADMIN'` → throw `FORBIDDEN` (server side) or redirect to `/dashboard` (client side). Never rely on a client-side role check alone — the Convex `auditLog/list` query must verify the caller's role before returning data.

**Convex query design.** The `auditLog` table was defined in T1 with all required fields. The `list` query should return entries ordered by `_creationTime` descending. Because Convex requires an explicit index for any non-`_id` sort, the schema must include an index on `_creationTime` (or the equivalent timestamp field) for the `auditLog` table. If the schema from T1 does not already include this index, add it now — do not perform a full-table scan.

**Display name resolution.** Each `auditLog` row contains an `adminUserId` (or equivalent FK). The query must resolve the admin's `displayName` from the `users` table within the same query handler (a single `ctx.db.get()` per row is acceptable for ≤200 rows; no N+1 concern at this scale). Return the resolved name as part of the result shape — do not leave user ID look-up to the client.

**`AuditLogEntry` type ownership.** Define `AuditLogEntry` in `convex/auditLog/types.ts` and import it into both `convex/auditLog/list.ts` (for the handler return annotation) and `src/components/AuditLogTable.tsx` (for the props type). Convex server files must not import from React component files — the dependency must flow the other way.

**Canonical action strings.** The action strings written by T23 are `'TEMPLATE_CREATED'` (on template create), `'TEMPLATE_PUBLISHED'` (on template publish), and `'TEMPLATE_ARCHIVED'` (on template archive). The `list` query returns these verbatim in the `action` field. Do not normalise or transform them client-side; render the raw string in the Action column. The Playwright assertions for AC6 use `/create/i` and `/publish/i` regexes, which match these canonical values.

**Performance budget.** The AC requires render within 1 second for up to 200 entries. No virtualization library is required at this scale; a plain mapped list in a `ScrollArea` (shadcn) is sufficient. If the Convex reactive query returns quickly, the DOM render time is negligible. The bottleneck is the query round-trip; ensure the index is in place. The performance test seeds 200 entries via `npx convex run seed:auditLog` — implement the seed script at `convex/seed/auditLog.ts` (see Contract.Files).

**Empty state.** The `states-and-feedback` design rule mandates text-only empty states with non-cutesy copy. Use: `"No audit log entries yet. Admin actions will appear here."` Match the admin templates empty state pattern from T23.

**Table layout.** Use a semantic `<table>` or a `<dl>` list — not a grid of `<div>`s — so screen readers can navigate rows with standard table commands. Columns: Timestamp | Admin | Action | Target Type | Target ID. Timestamp should be formatted as a human-readable absolute date + time (e.g., `May 8, 2026 at 3:42 PM`) — not a relative `"5 minutes ago"` string, since audit logs are archival.

**Typography and tokens.** Table header text: `label / meta` scale (14px, weight 500). Row text: body scale (15px, weight 400). Timestamps in `--text-tertiary`. Target IDs are technical identifiers — render in `var(--font-mono)` at 13px. Use `--bg-canvas` as the page background, `--bg-surface` for the table card, `--border-default` for row dividers.

**Playwright integration.** The AC explicitly targets `admin-templates.spec.ts`. Extend the existing spec (built for T23) to add a `test` block that, after the create-and-publish template flow, navigates to `/admin/audit` and asserts that at least two audit entries appear — one for the create action and one for the publish action. This avoids a separate spec file and leverages already-seeded state.

### Contract

**Files**
- `convex/auditLog/types.ts` — shared `AuditLogEntry` type; named export `AuditLogEntry`
- `convex/auditLog/list.ts` — Convex query; exports `list`; imports `AuditLogEntry` from `./types`
- `convex/seed/auditLog.ts` — Convex run script for performance seeding; callable via `npx convex run seed:auditLog`
- `convex/schema.ts` — `auditLog` table definition (pre-existing from T1); must include index on `_creationTime`
- `src/routes/admin/AuditLogPage.tsx` — React component; default export; imports `AuditLogEntry` from `convex/auditLog/types`
- `src/components/AuditLogTable.tsx` — presentational table component; named export `AuditLogTable`; imports `AuditLogEntry` from `convex/auditLog/types`
- `playwright/admin-templates.spec.ts` — E2E spec; extended with audit log assertions

**Exports**
- `AuditLogEntry` — named export from `convex/auditLog/types.ts`; TypeScript interface
- `list` — named export from `convex/auditLog/list.ts`; Convex `query` callable as `api.auditLog.list`
- `AuditLogPage` — default export from `src/routes/admin/AuditLogPage.tsx`; React component
- `AuditLogTable` — named export from `src/components/AuditLogTable.tsx`; React component

```typescript
  entries: AuditLogEntry[];

```

**Queries used**
- `api.auditLog.list` — called via `useQuery(api.auditLog.list)` in `AuditLogPage`

**Invariants**
- `api.auditLog.list` throws `FORBIDDEN` if the caller's `user.role` is not `'ADMIN'`; client never receives data for non-admin callers
- Entries are always returned ordered by `_creationTime` descending (newest first)
- `adminDisplayName` is always a resolved string; never a raw `Id<"users">` passed to the client
- The `/admin/audit` route redirects non-admin authenticated users to `/dashboard`; unauthenticated users to `/login`
- Canonical action strings written to the `action` field: `'TEMPLATE_CREATED'`, `'TEMPLATE_PUBLISHED'`, `'TEMPLATE_ARCHIVED'`

**Non-goals**
- Search, filter, or date-range selection of audit log entries
- Pagination with cursor / infinite scroll (plain list for ≤200 entries is sufficient)
- Writing to the `auditLog` table (that is the responsibility of T23 and other mutation tasks)
- Exporting audit log entries to CSV or PDF (P2 deferred)
- Displaying fields beyond timestamp, admin name, action, target type, and target ID
