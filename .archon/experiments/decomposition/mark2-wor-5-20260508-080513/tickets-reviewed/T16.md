# Joint Chat UI + Real-Time Messages (US-09, FR-07)

With synthesis delivered and both parties holding their personal guidance, the next milestone in Phase 2 is opening the shared conversational channel where resolution can actually happen. This task builds `JointChatView.tsx` ‚ÄĒ the first screen where both parties see each other's messages ‚ÄĒ and wires the two Convex functions that drive it: the `jointChat/messages` reactive query and the `jointChat/sendUserMessage` mutation. It is the foundational joint-session surface that Draft Coach (T18), Coach facilitation (T17a/b), and case closure (T20) all attach to.

## Acceptance Criteria

- JointChatView subscribes to `jointChat/messages` for the case; messages update reactively on both clients without a page refresh.
- User messages appear immediately (optimistic UI) and are confirmed once the mutation completes.
- Coach messages appear with a distinct visual treatment (e.g., 'Coach' label, different color).
- Calling `sendUserMessage` when `case.status != JOINT_ACTIVE` throws `CONFLICT`.
- Calling `sendUserMessage` when the caller is not a party to the case throws `FORBIDDEN`.
- Joint chat first meaningful paint ‚Č§ 500ms from navigation (Playwright performance measurement).
- Message list is accessible: `role='log'`, `aria-live='polite'`, new messages announced to screen readers.
- Playwright (`privacy.spec.ts`, PRD ¬ß7.1 launch criterion #2, Test 4 ‚ÄĒ joint chat positive control): user A sends a message in joint chat; user B's joint chat view receives the message reactively without a page refresh (confirms joint visibility is intentional and working).
- Playwright (`privacy.spec.ts`, PRD ¬ß7.1 launch criterion #2, Test 5 ‚ÄĒ cross-case isolation): user B calls `cases/get` for a case they are not a party to (owned by user A and a third user); asserts the response is NOT_FOUND.
- Playwright: in solo mode, send a message as initiator, toggle to invitee, verify the message appears.

## Implementation Notes

`JointChatView` is rendered by `CaseDetail` when `case.status === 'JOINT_ACTIVE'` (wired in T8). It uses the shared `ChatWindow` component (T9) for the message list and input ‚ÄĒ do not rebuild these primitives here.

**Reactive data pattern:** Use `useQuery(api.jointChat.messages, { caseId })` directly ‚ÄĒ no `useState` copy. Convex pushes updates on every `jointMessages` table change, so streaming Coach tokens appear automatically as T17b patches the row. Do not poll.

**Optimistic UI:** Call `useMutation(api.jointChat.sendUserMessage)` with an optimistic updater that immediately appends the user's message to the local list before the round-trip. The `useMutation` hook in `convex/react` supports `optimisticUpdate` for this. If the mutation fails (e.g., `CONFLICT`), Convex rolls back the optimistic state automatically.

**Coach message styling:** Per `component-patterns`, Coach messages use `left border: 3px solid var(--coach-accent)`, background `var(--coach-subtle)`, and a `‚ü°` glyph prefix with a 'Coach' label. Private Coach messages in T10b use `--accent-subtle` ‚ÄĒ this joint-chat Coach style (`--coach-accent` / `--coach-subtle`) applies exclusively here. Map `authorType: 'COACH'` on each `jointMessages` row to trigger this variant in `MessageBubble`.

**sendUserMessage mutation:** The mutation lives in `convex/jointChat/sendUserMessage.ts`. It must: (1) call `requireAuth(ctx)`, (2) verify the caller is `cases.initiatorUserId` or `cases.inviteeUserId`, (3) verify `cases.status === 'JOINT_ACTIVE'`, (4) insert the `jointMessages` row with `status: 'COMPLETE'` and `authorType: 'USER'`, and (5) schedule the Coach evaluation action via `ctx.scheduler.runAfter(0, internal.jointChat.generateCoachResponse, { caseId, messageId })`. The Coach action itself is implemented in T17a/b ‚ÄĒ this task only schedules the hook.

**Accessibility:** The chat container must have `role="log"` and `aria-live="polite"`. Per the `accessibility` design rule, announce "Coach is replying" once at streaming start (STREAMING state) and once more at COMPLETE ‚ÄĒ never re-announce on every token batch. The message input (`MessageInput` from T9) handles Enter-to-send and Shift-Enter-for-newline.

**Performance:** Convex reactive queries are low-latency by design. The ‚Č§ 500ms first meaningful paint constraint is primarily a Playwright measurement gate, not a rendering budget requiring heroic optimization. Ensure no blocking data fetch blocks the initial render (use `useQuery` returning `undefined` while loading; show skeleton from `states-and-feedback` design rule).

**Privacy tests (Tests 4 and 5):** Test 4 confirms joint visibility is intentional ‚ÄĒ it's a positive-control assertion, not an isolation check. Test 5 uses `cases/get` (the same query used by the dashboard) against an unrelated case to enforce `NOT_FOUND`. Both tests belong in `playwright/privacy.spec.ts` as part of PRD ¬ß7.1 launch criterion #2.

## Test-Gen Brief

### Contract

**Files**
- `src/routes/JointChatView.tsx` ‚ÄĒ main view component (imports ChatWindow from T9)
- `convex/jointChat/messages.ts` ‚ÄĒ Convex query; returns `Doc<"jointMessages">[]` for the caller's case
- `convex/jointChat/sendUserMessage.ts` ‚ÄĒ Convex mutation; inserts user message and schedules Coach eval
- `playwright/privacy.spec.ts` ‚ÄĒ Test 4 (joint chat positive control) and Test 5 (cross-case isolation)
- `playwright/solo-full-flow.spec.ts` ‚ÄĒ solo mode message visibility assertion

**Exports**
- `JointChatView` ‚ÄĒ default export, React functional component (route-level)
- `api.jointChat.messages` ‚ÄĒ Convex query (named export via generated `api`)
- `api.jointChat.sendUserMessage` ‚ÄĒ Convex mutation (named export via generated `api`)

**Signatures**
```typescript
// convex/jointChat/messages.ts
export const messages = query({
  args: { caseId: v.id("cases") },
  handler: async (ctx, { caseId }): Promise<Doc<"jointMessages">[]> => { ... }
});

// convex/jointChat/sendUserMessage.ts
export const sendUserMessage = mutation({
  args: { caseId: v.id("cases"), content: v.string() },
  handler: async (ctx, { caseId, content }): Promise<Id<"jointMessages">> => { ... }
});

// src/routes/JointChatView.tsx
export default function JointChatView({ caseId }: { caseId: Id<"cases"> }): JSX.Element
```

**Queries used**
- `api.jointChat.messages` ‚ÄĒ called by `useQuery` in `JointChatView`
- `api.cases.get` ‚ÄĒ called by Test 5 to assert NOT_FOUND on cross-case access

**Invariants**
- `messages` query must verify the caller is a party to the case; returns an empty array (not FORBIDDEN) when the caller is a party and no messages exist yet.
- `sendUserMessage` never inserts a message for a non-`JOINT_ACTIVE` case; throws `CONFLICT` if status has changed by the time the mutation runs.
- Optimistic UI must roll back silently when the mutation returns an error ‚ÄĒ no stale message remains in the list.
- Coach messages (`authorType: 'COACH'`) receive `--coach-accent` / `--coach-subtle` styling; all other messages receive the standard `MessageBubble` treatment.
- `sendUserMessage` schedules `internal.jointChat.generateCoachResponse` (T17b) via `ctx.scheduler.runAfter(0, ...)` ‚ÄĒ the Coach action is fire-and-forget from the mutation's perspective. (`generateCoachResponse` is an `internalAction` and must be referenced via `internal.*`, not `api.*`.)

**Non-goals**
- This task does NOT implement the Coach AI action itself (T17a/T17b).
- This task does NOT implement Draft Coach panel (T18) or case closure (T20).
- This task does NOT implement cost-cap UI banners (T26).
- This task does NOT add the `jointChat/enterSession` mutation ‚ÄĒ that belongs to T15.
- Access-control for `jointChat/messages` is enforced via party membership, not FORBIDDEN; draft messages (from T18's `draftMessages` table) are never returned by this query.
