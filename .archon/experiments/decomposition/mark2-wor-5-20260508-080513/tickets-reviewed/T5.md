# Authentication UI — login, session persistence, logout

Phase 1 of Conflict Coach requires users to sign in before accessing any case data. This ticket builds `LoginPage.tsx` — the single entry point for both magic-link email and Google OAuth authentication — and wires the session lifecycle: persistence across browser reloads, invite-token survival through OAuth redirects, and clean logout with protected-route enforcement. Without this gate, no Phase 1 feature work can be tested end-to-end by a real user.

## Acceptance Criteria

- LoginPage renders a magic-link email input and a 'Continue with Google' button.
- Submitting the magic-link form sends a sign-in email; clicking the link in the email authenticates the user and redirects to `/dashboard`.
- Google OAuth button initiates the OAuth flow and redirects to `/dashboard` on success.
- If the user navigated to `/invite/:token` before logging in, the token is preserved in `localStorage` and the user is redirected back to `/invite/:token` after authentication.
- Refreshing the browser while authenticated keeps the session active (no re-login required).
- Clicking 'Log out' calls `signOut()`, clears the Convex session, and redirects to `/login`.
- After logout, navigating to `/dashboard` redirects back to `/login`.
- Auth UI passes Playwright keyboard-only navigation (tab to each control, Enter to submit).
- No password input field exists anywhere in the UI.

## Implementation Notes

`LoginPage.tsx` uses `useAuthActions()` from `@convex-dev/auth/react` to call `signIn("resend", { email })` for magic link and `signIn("google")` for OAuth. The `useConvexAuth()` hook from `convex/react` provides `isAuthenticated` and `isLoading` — use these to drive the post-auth redirect logic, not local state.

> **Import path note.** `useAuthActions()` comes from `@convex-dev/auth/react` (the Convex Auth companion package), NOT from `convex/react`. The `convex/react` package provides `useConvexAuth()`. The project's `convex-auth` stack rule contains a minor inaccuracy on this point — do not "fix" the import to `convex/react` or `useAuthActions` will be undefined.

**Invite token survival through OAuth redirect.** Google OAuth sends the browser to an external URL and back. Any navigation state (`location.state`, in-memory variables) is lost. Define a named constant for the localStorage key:

```typescript
const PENDING_INVITE_TOKEN_KEY = 'pendingInviteToken';
```

Use this constant everywhere the token is read or written — never use the bare string. The pattern is: immediately before calling `signIn("google")`, write the current invite token to `localStorage` under `PENDING_INVITE_TOKEN_KEY`. On the return landing (and on every login completion), check `localStorage` for this key; if present, consume it, clear it, and call `useNavigate()` to send the user to `/invite/:token`. This must happen in a `useEffect` that fires after `isAuthenticated` becomes true.

**Protected routes.** Wrap all authenticated routes in a `ProtectedRoute` component that reads `isAuthenticated` from `useConvexAuth()`. When `isLoading` is true, render a skeleton or null — do not redirect and do not render children. When `isAuthenticated` is false and loading is done, render `<Navigate to="/login" replace />`.

**Convex Auth is in beta** — pin the `@convex-dev/auth` version in `package.json` and document the swap plan to Clerk in comments if blocking bugs surface.

**Keyboard UX.** The form must be standard HTML: `<form>` with `onSubmit`, `<input type="email">` for the magic-link field, and a `<button type="submit">`. This gives Enter-to-submit for free. The Google button is a separate `<button type="button">`. Tab order naturally reaches both controls and the submit button.

**Styling.** Use the login card max-width of `400px` centered per the layout rules. The card uses `--bg-surface`, `--border-default`, `--radius-lg`, `--shadow-1`. Button labels follow the copy conventions: "Send magic link" (primary variant) and "Continue with Google" (secondary variant). No exclamation marks. No password input — this is a hard requirement from the passwordless auth model.

## Test-Gen Brief

### Contract

**Files**
- Implementation: `src/routes/LoginPage.tsx`
- Protected route guard: `src/components/ProtectedRoute.tsx`
- Auth hook (invite token logic): inline `useEffect` in `LoginPage.tsx` or extracted to `src/hooks/usePostAuthRedirect.ts`
- Playwright spec: `e2e/auth.spec.ts`
- Component unit test: `src/routes/LoginPage.test.tsx`

**Exports**
- `LoginPage` — default export, React functional component, no props
- `ProtectedRoute` — named export, React functional component, props: `{ children: React.ReactNode }`

**Signatures**
```typescript
// src/routes/LoginPage.tsx
export default function LoginPage(): JSX.Element;

// src/components/ProtectedRoute.tsx
export function ProtectedRoute({ children }: { children: React.ReactNode }): JSX.Element;

// src/hooks/usePostAuthRedirect.ts (if extracted)
export function usePostAuthRedirect(): void;
// Side effect: on isAuthenticated transition to true, reads localStorage key
// PENDING_INVITE_TOKEN_KEY ('pendingInviteToken'); if present, navigates to
// /invite/:token and clears the key; otherwise navigates to /dashboard if
// current path is /login.
```

**Queries used**
- `useConvexAuth()` from `convex/react` — reads `{ isAuthenticated, isLoading }`
- `useAuthActions()` from `@convex-dev/auth/react` — provides `signIn(provider, args?)` and `signOut()`
- No Convex DB queries on this page (user row is created server-side on first login in T2)

**Invariants**
- `signIn("google")` must only be called after writing `PENDING_INVITE_TOKEN_KEY` to `localStorage` when the current path matches `/invite/:token`.
- `ProtectedRoute` must never render children when `isAuthenticated` is false and `isLoading` is false.
- `ProtectedRoute` must not redirect while `isLoading` is true (prevents flash) — render skeleton/null instead.
- No `<input type="password">` element exists anywhere in the component tree rendered by `LoginPage`.
- `localStorage.removeItem(PENDING_INVITE_TOKEN_KEY)` is called exactly once, immediately after the invite redirect navigation, to prevent stale token reuse.

**Non-goals**
- This ticket does not implement the invite acceptance page (`InviteAcceptPage`) — that is T7.
- This ticket does not implement the `users` table upsert on first login — that is wired in T2 (`convex/auth.ts`).
- This ticket does not validate that the magic-link email was actually delivered; the Playwright test for that path uses Convex Auth's test helper or mocked email flow.
- Admin role gating and admin route protection are not in scope here.
