# Private coaching backend (send message, AI response, mark complete, crisis detection, token tracking)

This task implements the server-side foundation for the private coaching phase — the first AI-interactive leg of the mediation process. Each party has a fully isolated chat channel with a Private Coach AI; nothing in this module is visible to the other party. The backend must handle the full message lifecycle from user submission through Claude streaming, accumulate token usage toward the per-case cost cap established in planning assumptions, flag crisis-keyword content so the client can surface a CrisisCard, and provide an idempotent mark-complete mutation that triggers synthesis scheduling when both parties are done.

## Acceptance Criteria

- `privateCoaching/myMessages`: returns only caller's messages
- `generateAIResponse`: streams into privateMessages row
- `generateAIResponse`: increments cases.tokenCount
- `privateCoaching/markComplete`: idempotent

## Implementation Notes

**Module location and shape.** The query and mutations live in `convex/privateCoaching.ts`; the action is declared as an `internalAction` in the same file (or a companion `convex/internal/privateCoaching.ts`). Mark it `internalAction` — it must never be callable directly from the client.

**Message flow.** `sendUserMessage` (mutation) inserts the user's message row with `status: "COMPLETE"` and immediately inserts a second `privateMessages` row with `role: "AI"`, `content: ""`, `status: "STREAMING"`. It then calls `ctx.scheduler.runAfter(0, internal.privateCoaching.generateAIResponse, { caseId, userId })`. This two-step insert ensures the reactive client sees the STREAMING placeholder before the action has a chance to write.

**Streaming flush pattern.** Inside `generateAIResponse`, the action opens a `stream: true` Claude call. It accumulates incoming delta tokens in a local string buffer and calls an internal `_flushStreamingChunk` mutation approximately every 50ms (using `Date.now()` deltas, not `setInterval` — Convex actions do not have a persistent event loop). On completion, a final `_finalizeMessage` mutation sets `status: "COMPLETE"` and records the total `tokens` field on the message row. The 50ms cadence is a best-effort heuristic; a single flush per response is acceptable for the initial implementation.

**Private Coach system prompt.** No template is applied. The hardcoded system prompt from tech spec §6.3.1 is injected verbatim:
> "You are a calm, curious, non-judgmental listener helping a person articulate their perspective in an interpersonal conflict. Ask clarifying questions. Reflect what they say. Help them identify what they actually want, what they're feeling, and what the other person might be thinking. Do not take sides. Do not tell them they're right or wrong. Your only goal is to help them prepare to communicate with the other party clearly and calmly."
This comes from `convex/lib/prompts.ts` (`assemblePrompt({ role: "PRIVATE_COACH" })`), which is T6.

**Context assembly.** Before calling Claude, the action reads: (1) the caller's `partyStates` row for `mainTopic`, `description`, `desiredOutcome`; (2) all prior `privateMessages` for `{ caseId, userId }` ordered by `createdAt`. It does NOT read any other party's data. Pass these to `compressIfNeeded` from `convex/lib/transcriptCompression.ts` (T7) before building the final messages array.

**Token counting and cost accumulation.** After the Claude call completes, the action calls an internal mutation that patches `cases.tokenCount` with `(existingCount ?? 0) + responseTokens`. The `usage` object from the Claude streaming response provides `input_tokens` + `output_tokens`; sum both. The cap enforcement logic (soft/hard) is out of scope for this ticket and is handled in T17 — this ticket only accumulates.

**Crisis keyword detection.** The task title includes "crisis detection." The backend's role is lightweight: after inserting the user's message, check `content` against a small list of crisis keywords (configurable constant in the module, e.g. `["suicide", "self-harm", "hurt myself", "kill myself", "abuse", "emergency"]`). If matched, add a `isCrisis: true` field to the returned message row or a separate metadata flag on the `privateMessages` row. The client (T25) reads this to conditionally render the `CrisisCard`. Because `privateMessages` schema does not include `isCrisis` in T2, the preferred approach for v1 is to detect on the client by inspecting the raw message content — the backend does not need a schema change. Document this decision in a code comment.

**`markComplete` idempotency.** The mutation sets `partyStates.privateCoachingCompletedAt = Date.now()` only if not already set. It then checks whether the other party's `partyStates.privateCoachingCompletedAt` is also set. If both are now complete, it calls `ctx.scheduler.runAfter(0, internal.synthesis.generate, { caseId })`. Running `markComplete` a second time must be a no-op: no duplicate scheduler call, no error.

**CLAUDE_MOCK guard.** At the top of `generateAIResponse`, check `process.env.CLAUDE_MOCK === "true"`. If true, skip the Claude API call, insert a canned AI message ("This is a mock coach response for testing."), wait 300ms to simulate latency, then set `status: "COMPLETE"`. This guard is critical for CI and E2E tests.

**Privacy invariant.** `myMessages` must query with the index `by_case_and_user` using `{ caseId, userId: user._id }`. It must never fall through to a query that returns messages by `caseId` alone. The `by_case` index on `privateMessages` is server-side only (context assembly); it is never exposed via any query reachable from the client.

### Queries used
- `ctx.db.query("privateMessages").withIndex("by_case_and_user", q => q.eq("caseId", caseId).eq("userId", userId))` — in `myMessages`
- `ctx.db.query("partyStates").withIndex("by_case", q => q.eq("caseId", caseId))` — in `markComplete` (reads both party rows)
- `ctx.db.patch(messageId, { content, status })` — in `_flushStreamingChunk`, `_finalizeMessage`
- `ctx.db.patch(caseId, { tokenCount: newCount })` — in `_finalizeMessage`
- `ctx.scheduler.runAfter(0, internal.synthesis.generate, { caseId })` — in `markComplete`
- `assemblePrompt({ role: "PRIVATE_COACH", ... })` from `convex/lib/prompts.ts`
- `compressIfNeeded(messages, anthropicClient, budgetTokens)` from `convex/lib/transcriptCompression.ts`
- `requireAuth(ctx)` from `convex/lib/auth.ts`
- `assertPartyToCase(ctx, user, caseId)` from `convex/lib/auth.ts`

### Invariants
- `myMessages` MUST filter by both `caseId` AND `userId` — never returns another user's messages.
- `sendUserMessage` inserts the STREAMING AI placeholder in the same mutation as the user message — it is never possible for the user message to exist without an accompanying AI placeholder.
- `markComplete` calling `ctx.scheduler.runAfter` for synthesis is idempotent: the guard `if (!myState.privateCoachingCompletedAt)` prevents double-scheduling.
- `generateAIResponse` MUST be `internalAction` — no client callable path should exist.
- Token accumulation uses `(existing ?? 0) + delta`; never overwrites; never sets to zero.

### Non-goals
- Cap enforcement (soft/hard budget limits) — handled in T17.
- The other party's message visibility — enforced by index choice, no separate filter needed.
- Crisis keyword DB schema changes — client-side detection only in v1.
- Synthesis generation logic — only the scheduler call (`ctx.scheduler.runAfter`) is in scope; T13 owns the action body.
