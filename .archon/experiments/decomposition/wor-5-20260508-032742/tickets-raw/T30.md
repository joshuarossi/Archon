# Solo mode UI (PartyToggle, useSoloActingParty hook, all views integrated)

Solo mode lets a single developer or demo user play both parties in one browser session, running the entire mediation flow end-to-end without a second account. This ticket adds the `PartyToggle` segmented control (visible only when `case.isSolo === true`), the `useSoloActingParty` hook that reads the active role from the URL search param, and wires the toggle into every data query across private coaching, synthesis, joint chat, dashboard, and the PhaseHeader so the entire UI consistently reflects which party perspective is active.

## Acceptance Criteria

- PartyToggle: segmented control; only rendered when case.isSolo=true
- Toggle state in ?as=initiator|invitee URL param; preserved on refresh
- useSoloActingParty(): returns role for selected party
- All queries respect toggle for solo mode
- Both synthesisTexts differ in solo mode
- PartyToggle absent when case.isSolo=false
- Solo cases show 'Solo test' badge on dashboard and PhaseHeader

## Implementation Notes

Solo mode is a view-level feature only ŌĆö the DB schema stores `isSolo: v.boolean()` on the `cases` table and pre-binds the same `userId` to both `INITIATOR` and `INVITEE` slots in `partyStates`. No backend changes are needed in this ticket; T9 already handles the data model. The UI layer needs to intercept which `partyStates` row is "me" depending on the toggle.

The toggle state lives in the URL query param `?as=initiator|invitee` (React Router v6 `useSearchParams`). This means the page refreshes correctly, the browser back button reverses the toggle, and the URL is shareable for debugging ŌĆö no `localStorage` required. Default when the param is absent is `initiator`.

`useSoloActingParty()` reads the URL param and returns a typed `'INITIATOR' | 'INVITEE'` role. The hook should be a thin wrapper ŌĆö no side effects, no writes. Downstream queries that accept a `role` or `userId` parameter derive the correct acting user from this hook. For non-solo cases the hook is irrelevant; callers are expected to guard on `case.isSolo` before invoking toggle logic.

The `PartyToggle` component is rendered in the `PhaseHeader` right slot (see design doc ┬¦3.2 and tech spec ┬¦9.3). It is `null` when `case.isSolo === false` ŌĆö absent entirely, no empty div. The segmented control displays the initiator's and invitee's display names, derived from the `partyStates` rows already loaded by `CaseDetail` via `useOutletContext`.

The 'Solo test' badge renders in two places: the dashboard case row (adjacent to the case title) and the `PhaseHeader` center section. It uses `coach-accent` at 20% opacity as background ŌĆö the same lavender used for the toggle's active state ŌĆö making solo context visually unmistakable (design doc ┬¦6.10: "Tester should never confuse solo mode with a real conversation.").

Because private coaching messages, synthesis texts, and joint chat auth all rely on the caller's `userId`, solo mode's queries need to forward the correct acting `userId` (initiator's vs. invitee's `partyState` row) rather than the logged-in user's raw identity. Both `userId` values are the same account but the `role` column on `partyStates` differentiates them. Verify that `privateCoaching/myMessages`, `jointChat/mySynthesis`, and `cases/partyStates` all accept an explicit `role` override when `isSolo` is true.

## Stack Practices

- URL search param: useSearchParams from React Router v6. actingAs defaults to 'initiator'.
- useSoloActingParty: reads URL param, returns 'INITIATOR' | 'INVITEE'.
- PartyToggle: two buttons as segmented control, coach-accent for active state.
- Anti-pattern: do NOT store solo party selection in localStorage ŌĆö use URL param.

## Design Practices

- PartyToggle (design doc ┬¦3.2): segmented control in PhaseHeader, right-aligned. Two segments: 'Viewing as: [Initiator Name]' and '[Invitee Name]'. Active segment: bg-coach-accent (#8B7AB5) at 100% opacity, white text, border-radius 6px. Inactive: --bg-surface-subtle, --text-secondary. Container: bg-surface, border --border-default, border-radius 8px, 4px padding. Only rendered when case.isSolo === true ŌĆö absent otherwise (design doc ┬¦9.3).
- PartyToggle transition: switching parties = 150ms bg crossfade ease-out. The toggle is 'prominent' (design doc ┬¦3.2) to make solo context unmistakable.
- 'Solo test' badge (design doc): text-xs Inter, bg-coach-accent at 20% opacity, text-coach-accent, px-1.5 py-0.5 rounded-full. Appears in dashboard case rows and PhaseHeader when case.isSolo. Short text: 'Solo'.
- Visual differentiation: in solo mode, the PhaseHeader should visually signal the active party ŌĆö consider a subtle left border or background tint in the party's avatar color to reinforce which perspective is active.
- Accessibility: PartyToggle buttons use role='radio' / aria-checked pattern, or role='tab', aria-selected. Group has aria-label='Viewing as party'. Toggle state changes announce via aria-live='polite' region: 'Now viewing as Initiator.' URL param update does not itself announce ŌĆö use the live region.
