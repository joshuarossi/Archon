# Authentication ā€” Convex Auth with magic link and Google OAuth

This task wires up the authentication layer that gates every subsequent feature in Conflict Coach. Because the entire privacy and authorization model rests on knowing who the caller is ā€” `ctx.auth.getUserIdentity()` is called at the top of every Convex function ā€” authentication must be solid and correct before any other backend work can proceed. The goal is a working Convex Auth configuration with two providers (email magic link via Resend, Google OAuth) and an automatic `users` table upsert on first login.

## Acceptance Criteria

- User registers and logs in via magic link
- User logs in via Google OAuth
- First login creates a users row with role='USER'
- Subsequent logins upsert the users row
- Session persists across browser reload
- Session expires after 30 days of inactivity
- ctx.auth.getUserIdentity() returns the identity in any Convex function after login

## Implementation Notes

Convex Auth is the chosen provider (PRD Q1 resolved in its favor). The Clerk swap path is documented in README but not implemented. Auth configuration lives in `convex/auth.config.ts`; magic link uses the Resend provider. The Convex client in `main.tsx` must be wrapped in `<ConvexAuthProvider>` around (or wrapping) `<ConvexProvider>` ā€” the exact nesting order is prescribed by `@convex-dev/auth`.

The `users` table (defined in T2) has fields: `email`, `displayName` (optional), `role` (union `"USER" | "ADMIN"`), and `createdAt`. The upsert on first login must key by `email` obtained from `ctx.auth.getUserIdentity()`. Promotion to `ADMIN` is a manual Convex Dashboard action in v1; no self-serve admin claim UI is built.

The tech spec (Ā§4.2) specifies that `ctx.auth.getUserIdentity()` returns the authenticated identity or `null` ā€” and that every query, mutation, and action begins by checking for this identity and then looking up or upserting the `users` row. The authorization helper module for that pattern is built in T4 (which depends on T3), so this task's responsibility is: make login work and ensure a `users` row exists after login.

Environment variables required (also in `.env.example` from T1): `GOOGLE_OAUTH_CLIENT_ID`, `GOOGLE_OAUTH_CLIENT_SECRET`. These are stored as Convex environment variables, not in `.env` directly. The Resend email provider also needs `AUTH_RESEND_KEY` (or the equivalent Convex Auth env var name for Resend) configured in the Convex dashboard.

Session expiry at 30 days of inactivity should be configured in the Convex Auth settings ā€” check `@convex-dev/auth` docs for the `sessionDurationMs` or equivalent config option.

**Signatures**

**Queries used**
- `ctx.db.query("users").withIndex("by_email", q => q.eq("email", identity.email)).unique()` ā€” check for existing user
- `ctx.db.insert("users", { email, role: "USER", createdAt: Date.now() })` ā€” create on first login
- `ctx.db.patch(existingUser._id, {})` ā€” upsert path (no-op patch or update displayName if provided)

**Invariants**
- `role` defaults to `"USER"` on insert; never set to `"ADMIN"` programmatically during login
- `email` is the stable key ā€” if a user logs in via Google and then magic link with the same email, they get the same `users` row
- A `users` row must exist before any subsequent function calls (T4's `requireAuth` will throw `USER_NOT_FOUND` if not)
- `ctx.auth.getUserIdentity()` must return non-null inside any Convex function called by a logged-in user after this task is complete

**Non-goals**
- Admin role assignment UI (manual Convex Dashboard action in v1)
- Clerk integration (swap path documented in README but not implemented)
- Email notification on login (deferred to v1.1)
- Password-based registration (not in v1)
- The authorization helpers (`requireAuth`, `assertPartyToCase`, `assertAdmin`) ā€” those are T4

## Stack Practices

- Install: `npm install @convex-dev/auth @auth/core`.
- Auth config file is `convex/auth.config.ts`.
- Magic link provider: use `Resend` email provider.
- Google OAuth: requires `GOOGLE_OAUTH_CLIENT_ID` and `GOOGLE_OAUTH_CLIENT_SECRET` in Convex environment variables.
- Client side: import `useAuthActions` from `@convex-dev/auth/react`.
- Anti-pattern: do NOT store auth tokens in localStorage manually.
- ConvexAuthProvider must wrap ConvexProvider in main.tsx.
