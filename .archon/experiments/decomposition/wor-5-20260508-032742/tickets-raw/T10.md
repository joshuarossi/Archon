# Invite Flow Backend (redeem mutation, preview query, token validation)

The invite flow is the gate through which a second party enters a Conflict Coach case. Without it, every case is one-sided. This ticket implements the two Convex functions that power the `/invite/:token` route: `invites/preview` (a public, unauthenticated query that shows safe non-private case metadata) and `invites/redeem` (the authenticated mutation that atomically binds the invitee to the case and marks the token consumed). Token lifecycle correctness — specifically the guarantee that a single token can only ever be redeemed once — is a hard safety requirement, not a nice-to-have.

## Acceptance Criteria

- `invites/redeem`: validates token, atomically binds invitee, marks token CONSUMED
- `invites/redeem` on CONSUMED token: throws TOKEN_INVALID
- `invites/preview`: returns non-private fields only

## Implementation Notes

The `inviteTokens` table (defined in T2) stores each token as a 32-character URL-safe string with status `ACTIVE | CONSUMED | REVOKED`, indexed by `by_token`. The redemption mutation must use a single transaction to:

1. Look up the token via `ctx.db.query("inviteTokens").withIndex("by_token", q => q.eq("token", token)).unique()`.
2. If not found or `status !== "ACTIVE"`, throw `ConvexError({ code: "TOKEN_INVALID", ... })`.
3. Prevent the caller from redeeming their own invite (i.e., if the caller is the `cases.initiatorUserId`), throw `FORBIDDEN`.
4. Patch `cases.inviteeUserId` to the caller's user id and `cases.status` from `DRAFT_PRIVATE_COACHING` to `BOTH_PRIVATE_COACHING` (only if it was still `DRAFT_PRIVATE_COACHING`; if the invitee slot is already bound, throw `TOKEN_INVALID`).
5. Insert a `partyStates` row for the invitee (role `INVITEE`) with form fields left empty.
6. Patch the `inviteTokens` row: `status: "CONSUMED"`, `consumedAt: Date.now()`, `consumedByUserId`.

All six steps happen in one Convex mutation transaction — if any step throws, none of the writes persist.

`invites/preview` is a Convex **query** (not a mutation) with **no auth requirement**. It exists solely so the `/invite/:token` page can render case metadata before the user logs in. It must return only non-private fields: `initiatorDisplayName` (from `users`), `mainTopic` (from the initiator's `partyStates` row — this is the shared field written by the initiator in the case creation form), and `category` (from `cases`). It must **never** return: description, desiredOutcome, private coaching messages, synthesis, or any invitee data. If the token is not found, return `null`. If the token is `CONSUMED`, return `{ status: "CONSUMED" }` so the client can show the right error state.

Authorization for `invites/redeem` uses `requireAuth(ctx)` from `convex/lib/auth.ts` (T4). The token itself is the credential — no case-level party check is needed (the caller is not yet a party).

Token generation (in `cases/create`, T9) uses `crypto.randomUUID().replace(/-/g, '').slice(0, 32)`. Tokens are stored in plaintext in v1 (acceptable because consumption is instant and the token has no post-use value — noted in tech spec §11 checklist).

### Queries used
- `ctx.db.query("inviteTokens").withIndex("by_token", q => q.eq("token", token)).unique()`
- `ctx.db.get(tokenRow.caseId)` — to fetch the case
- `ctx.db.query("partyStates").withIndex("by_case_and_user", q => q.eq("caseId", caseId).eq("userId", initiatorUserId)).unique()` — to fetch initiator's `mainTopic` for preview
- `ctx.db.get(caseDoc.initiatorUserId)` — to fetch `users` row for display name
- `ctx.db.patch(caseId, { inviteeUserId, status: "BOTH_PRIVATE_COACHING", updatedAt })` — in redeem
- `ctx.db.patch(tokenId, { status: "CONSUMED", consumedAt, consumedByUserId })` — in redeem
- `ctx.db.insert("partyStates", { caseId, userId, role: "INVITEE", ... })` — in redeem

### Invariants
- A token in status `CONSUMED` or `REVOKED` can never be redeemed; `redeem` throws `TOKEN_INVALID`.
- `redeem` is idempotent only in the sense that a second call will throw `TOKEN_INVALID` — it does NOT silently succeed.
- `preview` never exposes the initiator's `description`, `desiredOutcome`, or any private coaching content.
- `preview` never exposes any invitee data.
- The initiator cannot redeem their own invite (`cases.initiatorUserId === caller.userId` → `FORBIDDEN`).
- All six writes in `redeem` succeed atomically or none are committed.

### Non-goals
- Token revocation UI (admin-only, not built in v1).
- Email delivery of invite links (that's T9's responsibility — this ticket only validates and redeems).
- Auth or session creation — `redeem` assumes the caller is already authenticated.
- Updating `cases.status` for any transition other than `DRAFT_PRIVATE_COACHING → BOTH_PRIVATE_COACHING`.

## Stack Practices

- Atomicity: Convex mutations are fully transactional.
- invites/preview is a Convex query — no auth required for preview.
- Anti-pattern: do NOT check token validity in invites/preview.
