# Joint chat entry mutation and Coach facilitation backend

T14 implements the backend machinery that transitions a case into live joint-session mode and keeps it productively facilitated by an AI Coach. It covers the `jointChat/enter` mutation (which advances the case from `READY_FOR_JOINT` to `JOINT_ACTIVE`), the reactive queries that serve joint messages and each party's private synthesis text, the `jointChat/sendUserMessage` mutation that persists user turns and schedules Coach evaluation, and the `jointChat/generateCoachResponse` internalAction that runs the two-stage Haiku-gate â†’ Sonnet facilitation pipeline. This task sits at the heart of the product: without it, neither party can exchange messages in the joint session or receive Coach guidance.

## Acceptance Criteria

- `jointChat/enter`: READY_FOR_JOINTâ†’JOINT_ACTIVE, idempotent
- `generateCoachResponse`: Haiku classifies; Sonnet only for non-NORMAL_EXCHANGE
- T5 filter on Coach Sonnet response; fallback on failure
- Increments tokenCount; enforces HARD_CAP

## Implementation Notes

**Domain file:** All functions for this ticket live in `convex/jointChat.ts`. Follow the one-file-per-domain convention established in T9 and T12.

**`jointChat/enter` mutation:** Reads the current `cases` row and asserts the caller is a party (via `assertPartyToCase` from `convex/lib/auth.ts`). If `case.status === 'JOINT_ACTIVE'` the mutation returns without writing â€” idempotency is required because both parties call this independently and the second caller must not throw. If `case.status !== 'READY_FOR_JOINT'` throw `ConvexError(CONFLICT)` from the codes defined in T4. On the happy path, call the `transitionToJointActive` pure function from `convex/lib/stateMachine.ts` (T8) to get the new status, then write it back to `cases`.

**Reactive queries:**
- `jointChat/messages({ caseId })`: Index on `jointMessages.by_caseId` (defined in T2 schema). Assert caller is party + case is not in a private-only state (i.e., status must not be `DRAFT_PRIVATE_COACHING`). Return the full `JointMessage[]` array.
- `jointChat/mySynthesis({ caseId })`: Read the caller's `partyStates` row and return `{ text: synthesisText }` if populated, or `null` if synthesis has not yet been written. Assert party membership.

**`jointChat/sendUserMessage` mutation:** Inserts a `jointMessages` row with `authorUserId`, `caseId`, `content`, `status: 'SENT'`, `role: 'USER'`. Then calls `ctx.scheduler.runAfter(0, internal.jointChat.generateCoachResponse, { caseId, triggerType: 'USER_MESSAGE' })`. Assert case is `JOINT_ACTIVE`; throw `CONFLICT` otherwise. Detect `@coach` mention (or similar) inside the mutation body â€” NOT on the client â€” per the anti-pattern note in stack practices.

**`generateCoachResponse` internalAction â€” two-stage pipeline:**

*Stage 1 â€” Haiku gate (classification):*
Check `CLAUDE_MOCK` guard first (return a canned `NORMAL_EXCHANGE` result in mock mode). Fetch the most recent user joint message. Call `claude-haiku-4-5-20251001` with a concise single-turn classification prompt requesting one of: `INFLAMMATORY`, `PROGRESS`, `QUESTION_TO_COACH`, `NORMAL_EXCHANGE`. Use `max_tokens: 50` (non-streaming). If the result is `NORMAL_EXCHANGE` and `triggerType` is not `TIMER` and not `AT_MENTION`, return early â€” no Sonnet call. This gate is the primary cost-control mechanism.

*Stage 2 â€” Sonnet facilitation:*
Assemble the prompt via `assemblePrompt({ role: 'COACH', ... })` from `convex/lib/prompts.ts` (T6). Build the context window from joint chat history + BOTH parties' `synthesisText` values (never raw private messages). Run the assembled context through `compressIfNeeded` from `convex/lib/transcriptCompression.ts` (T7) to stay within the 60k-token budget. Call `claude-sonnet-4-5` with streaming. Stream tokens into a `jointMessages` row (insert a `STREAMING` row before the loop; update content on each chunk; flip to `COMPLETE` on finish).

*Privacy filter:* After generation completes, call `filterResponse(coachOutput, [...initiatorPrivateMessages, ...inviteePrivateMessages])` from `convex/lib/privacyFilter.ts` (T5). If the filter rejects, retry up to 2 times. On final failure, write a generic fallback message (`"I need to take a moment â€” please continue your conversation."`) and log a flag for admin review via the `auditLog` table.

*Token accounting:* After each Sonnet call, compute approximate token usage (input tokens + output tokens, approximated via word-count if exact counts are unavailable). Call an `internalMutation` that increments `cases.tokenCount` by the usage delta. Before making any Sonnet call, check `cases.tokenCount` against the `HARD_CAP` constant from `convex/lib/pricing.ts` (T4). If `tokenCount >= HARD_CAP`, set `case.status = 'COST_LIMITED'` and return without calling the API â€” token enforcement in T17 will add the full degradation behavior, but the hard gate must be present here so no call is ever made past the cap.

**`CLAUDE_MOCK` guard:** Mirror the pattern from T12. At the top of `generateCoachResponse`, if `process.env.CLAUDE_MOCK === 'true'`, skip all API calls. For the Haiku gate return `QUESTION_TO_COACH` (so Sonnet stage runs in tests). For Sonnet, write a canned `COMPLETE` coach message to `jointMessages`.

**@-mention detection:** Inside `jointChat/sendUserMessage`, check `content.includes('@coach')` (case-insensitive). If true, pass `triggerType: 'AT_MENTION'` to the scheduled action so the Haiku gate is bypassed.

- `sendUserMessage` must throw `CONFLICT` if `case.status !== 'JOINT_ACTIVE'`
- @-mention detection happens inside `sendUserMessage`, never on the client

## Stack Practices

- Haiku classification: single non-streaming call, max_tokens 50.
- Anti-pattern: do NOT hardcode @-mention as a client-side feature â€” detection happens inside the mutation body.
- CLAUDE_MOCK guard: same process.env.CLAUDE_MOCK check as T12.
