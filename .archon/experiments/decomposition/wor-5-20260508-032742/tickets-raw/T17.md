# AI cost tracking hardening: budget enforcement, soft-cap degradation, Anthropic error handling

T12–T15 each accumulate token counts into `cases.tokenCount` as they run. T17 is a cross-cutting hardening pass that reads those counts and acts on them: enforcing the $2 soft cap (Coach response degrades to boilerplate), the $10 hard cap (case marked `COST_LIMITED`, AI fully disabled), and handling upstream Anthropic failures (429 rate limits and network timeouts) uniformly across all four AI actions — `generateAIResponse` (private coaching), `synthesis/generate`, `generateCoachResponse` (joint chat), and `generateResponse` (Draft Coach). This task exists because AI cost control and error resilience are cross-cutting concerns that cannot be fully owned by any single feature task; adding them as a final hardening pass ensures consistent behavior across the whole AI surface.

## Acceptance Criteria

- Soft cap: generateCoachResponse posts boilerplate at $2 threshold
- Hard cap: all four AI actions return early at $10; set COST_LIMITED
- 429: retry once after 2000ms; second 429 → RATE_LIMITED
- Network timeout >30s: message status=ERROR; throw AI_ERROR

## Implementation Notes

`convex/lib/pricing.ts` was introduced in T4 and already exports the soft cap, hard cap, and per-token pricing constants. T17 adds a shared helper — `checkCostCap(ctx, caseId)` — that reads `cases.tokenCount` and returns one of `'ok' | 'soft_cap' | 'hard_cap'`. This helper is called at the top of each of the four internal AI actions before any Anthropic API call is made.

**Soft-cap behaviour (≥$2, <$10):** Only `generateCoachResponse` degrades — it posts a templated boilerplate message to `jointMessages` ("I'm limited right now — consider summarizing where you are and whether you've reached agreement.") and returns without calling Anthropic. The other three AI actions (`generateAIResponse`, `synthesis/generate`, `generateResponse`) return early without modifying message state. The `cases.status` is NOT changed at the soft cap — the case remains active.

**Hard-cap behaviour (≥$10):** All four AI actions set `cases.status` to `'COST_LIMITED'` via an internal mutation and return immediately. No Anthropic call is made. The UI (T27) renders a banner and hides Draft Coach; parties can still exchange plain messages via `jointChat/sendUserMessage`.

**429 handling:** Wrap the Anthropic streaming call in a try/catch. On a 429 response, wait exactly 2000ms (`await new Promise(r => setTimeout(r, 2000))`), then retry once. If the second attempt also throws a 429, catch it and throw `new ConvexError({ code: 'RATE_LIMITED', message: 'Rate limit hit after retry', httpStatus: 429 })`. Do NOT use exponential backoff or recursion.

**Timeout handling (>30s):** Use `Promise.race([anthropicCall, timeout(30_000)])`. If the timeout fires first, set the in-flight message row to `status: 'ERROR'` via internal mutation and throw `new ConvexError({ code: 'AI_ERROR', message: 'Response timeout', httpStatus: 503 })`. The UI message bubble for `status: 'ERROR'` renders a Retry button (built in T19a).

**Token accumulation** was specified in T12–T15 acceptance criteria. T17 does not change how tokens are counted — it only adds enforcement on top of the existing `cases.tokenCount` value.

**CLAUDE_MOCK guard:** When `process.env.CLAUDE_MOCK === 'true'`, skip all cap and retry logic — return immediately with canned responses as defined in T12. The cap check must be gated behind `if (!process.env.CLAUDE_MOCK)`.

The `COST_LIMITED` status transition belongs to the state machine (`convex/lib/stateMachine.ts`, introduced in T8). If `COST_LIMITED` is not already a valid transition target in `stateMachine.ts`, add it; the transition is `JOINT_ACTIVE → COST_LIMITED` triggered by the hard cap.

Error codes `RATE_LIMITED` and `AI_ERROR` are defined in T4 (`convex/lib/auth.ts` error constants). Import them from there.

## Test-Gen Brief

**Files**

- Implementation helpers: `convex/lib/pricing.ts` (extend with `checkCostCap`), modifications to `convex/privateCoaching.ts`, `convex/synthesis.ts`, `convex/jointChat.ts`, `convex/draftCoach.ts`
- New test file: `convex/lib/__tests__/costEnforcement.test.ts`
- Existing test files for AI actions (if present): add cases to each

**Exports**

- `convex/lib/pricing.ts`:
  - `SOFT_CAP_USD` — named value export, number, `2`
  - `HARD_CAP_USD` — named value export, number, `10`
  - `COST_PER_TOKEN` — named value export, number (Sonnet input/output blended rate)
  - `checkCostCap(tokenCount: number): 'ok' | 'soft_cap' | 'hard_cap'` — named function export, pure (no ctx), synchronous
- No new public mutations or queries are introduced by T17; all changes are inside internalActions already exported by T12–T15

**Signatures**

```typescript
// convex/lib/pricing.ts
export const SOFT_CAP_USD: number;
export const HARD_CAP_USD: number;
export const COST_PER_TOKEN: number; // blended rate; adjust when Anthropic changes pricing
export function checkCostCap(tokenCount: number): 'ok' | 'soft_cap' | 'hard_cap';

// Usage inside each AI internalAction (not exported):
const capStatus = checkCostCap(caseDoc.tokenCount ?? 0);
if (capStatus === 'hard_cap') { /* set COST_LIMITED, return */ }
if (capStatus === 'soft_cap') { /* degrade or early-return */ }
```

**Queries used**

- `ctx.db.get(caseId)` — reads `cases.tokenCount` and `cases.status`
- `ctx.runMutation(internal.cases.setCostLimited, { caseId })` — internal mutation that sets `status: 'COST_LIMITED'` (add to `convex/cases.ts` if not present)
- `ctx.runMutation(internal.jointChat.appendMessage, { ... status: 'ERROR' })` — existing internal mutation from T14
- `new ConvexError({ code: ERROR_CODES.RATE_LIMITED, ... })` — from T4 constants
- `new ConvexError({ code: ERROR_CODES.AI_ERROR, ... })` — from T4 constants

**Invariants**

- `checkCostCap` is pure: given the same `tokenCount`, always returns the same result with no side effects
- The 429 retry fires exactly once — never a third attempt
- The 30s timeout resolves via `Promise.race`; no unbounded wait
- When `CLAUDE_MOCK=true`, cap checks, retries, and timeout logic are entirely bypassed
- Hard cap always transitions case to `COST_LIMITED`; soft cap never changes `cases.status`
- The soft-cap boilerplate message is only posted by `generateCoachResponse`, not by other AI actions

**Non-goals**

- Does NOT implement any frontend UI for COST_LIMITED (that is T27)
- Does NOT change token counting logic (owned by T12–T15)
- Does NOT add exponential backoff or circuit-breaker patterns
- Does NOT notify admins at the soft cap (noted in spec as future work; no infrastructure for it in v1)
- Does NOT add new public Convex queries or mutations beyond `internal.cases.setCostLimited`

## Stack Practices

- Cap check at the top of each AI action using values from convex/lib/pricing.ts.
- Retry: fixed 2s delay, not exponential.
- Anti-pattern: do NOT let AI errors bubble up as unhandled Convex action failures.
