# Case Closure Backend

Once both parties are in the joint session, they need a structured way to formally close the case — whether by mutual agreement, unilateral walkaway, or system-detected abandonment. This task implements all closure mutations plus the daily Convex cron that auto-closes stale `JOINT_ACTIVE` cases after 30 days of inactivity. It completes the server-side case lifecycle, enabling the closure UI (T29) to wire directly against these endpoints.

## Acceptance Criteria

- `jointChat/proposeClosure`: sets closureProposed; writes to auditLog
- `jointChat/confirmClosure`: atomically closes case as CLOSED_RESOLVED
- Cron: daily scan for JOINT_ACTIVE cases with 30d inactivity

## Implementation Notes

**State machine integration.** All status transitions in this task go through `convex/lib/stateMachine.ts` (T8). The closure path ultimately drives `JOINT_ACTIVE → CLOSED_RESOLVED`, `JOINT_ACTIVE → CLOSED_UNRESOLVED`, or `JOINT_ACTIVE → CLOSED_ABANDONED`. Each of these is a transition exported from the state machine and called inside a Convex mutation — the mutation reads current case status, calls the transition function, and if it returns a new status, writes it back atomically.

**Propose / confirm flow.** Two separate mutations implement the two-step closure:
1. `jointChat/proposeClosure` — sets `partyStates.closureProposed = true` for the calling party, writes the proposed `closureSummary` to the `cases` row (not confirmed yet), and appends an `auditLog` entry with `action: "CLOSURE_PROPOSED"`, `targetType: "case"`, and `targetId: caseId`.
2. `jointChat/confirmClosure` — reads both parties' `partyStates` rows; throws `CONFLICT` if the other party has not proposed; sets `partyStates.closureConfirmed = true` for the confirming party, then sets `cases.status = CLOSED_RESOLVED`, `cases.closedAt = Date.now()`, and appends `auditLog` entry `"CLOSURE_CONFIRMED"`.

**Reject closure.** `jointChat/rejectClosure` clears `partyStates.closureProposed` for whichever party had set it; this removes the proposal banner for both parties. The case remains `JOINT_ACTIVE`.

**Unilateral close.** `jointChat/unilateralClose` takes `{ caseId, reason }` and immediately transitions the case to `CLOSED_UNRESOLVED`. It records `cases.closedAt`, sets `closureSummary` to the `reason` string (for audit readability), and writes `auditLog` entry `"CASE_CLOSED_UNRESOLVED"`. Authorization: any party to a `JOINT_ACTIVE` case can call this.

**Cron for abandoned cases.** `convex/crons.ts` registers a daily job via `cronJobs()` from `convex/server`. The job body is an internal mutation that:
1. Queries all cases where `status == "JOINT_ACTIVE"`.
2. Filters to those where `updatedAt < Date.now() - 30 * 24 * 60 * 60 * 1000`.
3. For each match, sets `status = "CLOSED_ABANDONED"` and `closedAt = Date.now()`.
4. Writes an `auditLog` entry for each closure: `action: "CASE_AUTO_ABANDONED"`, `actorUserId` is a system sentinel (requires a system user ID or a special constant — use the cases' `initiatorUserId` as actor for v1 since there is no system user row).

Do not use `setInterval`, external cron services, or `setTimeout`. The Convex `cronJobs()` scheduler is the only correct mechanism (tech spec §5.3).

**Authorization.** All mutations call `requireAuth(ctx)` and `assertPartyToCase(ctx, caseId, user._id)` from `convex/lib/auth.ts` (T4). The cron internal mutation bypasses user auth (it is an `internalMutation`).

**Audit log shape.** All audit writes use the `auditLog` table defined in T2: `{ actorUserId, action, targetType: "case", targetId: caseId.toString(), metadata: { summary?, reason? }, createdAt: Date.now() }`.

**`partyStates.closureProposalSummary`.** The planning assumptions added `closureProposalSummary: v.optional(v.string())` to `partyStates`. Use this field on the proposing party's row to hold the per-party record of what they proposed. The canonical `closureSummary` on the `cases` row is the shared source of truth.

**Idempotency guards.** `proposeClosure` should be idempotent if called twice by the same party (re-writing the same proposal is allowed; no error). `confirmClosure` throws `CONFLICT` if the case is already `CLOSED_*`.

### Queries used
- `ctx.db.get(caseId)` — load case
- `ctx.db.query("partyStates").withIndex("by_case", q => q.eq("caseId", caseId)).collect()` — both party states
- `ctx.db.query("cases").withIndex("by_initiator", ...)` / full table scan with filter — cron uses `ctx.db.query("cases").filter(q => q.eq(q.field("status"), "JOINT_ACTIVE"))` with updatedAt check
- `ctx.db.patch(caseId, { status, closedAt, closureSummary })` — update case
- `ctx.db.patch(partyStateId, { closureProposed, closureConfirmed, closureProposalSummary })` — update party state
- `ctx.db.insert("auditLog", { ... })` — audit entry
- `requireAuth(ctx)` from `convex/lib/auth.ts` (T4)
- `assertPartyToCase(ctx, caseId, user._id)` from `convex/lib/auth.ts` (T4)
- State machine transition functions from `convex/lib/stateMachine.ts` (T8):
  - `transitionToClosedResolved(currentStatus)` — called in confirmClosure
  - `transitionToClosedUnresolved(currentStatus)` — called in unilateralClose
  - `transitionToClosedAbandoned(currentStatus)` — called in checkAbandonedCases

### Invariants
- `cases.status` only changes via stateMachine transition functions; never via raw `ctx.db.patch` without calling the transition first.
- A `CLOSED_*` case can never be re-opened by any of these mutations; all throw `CONFLICT` if `case.status` is already a closed state.
- `auditLog` entry is written in the same mutation transaction as the state change — they are atomic.
- `confirmClosure` requires exactly one party to have `closureProposed = true` before it can proceed.
- `crons.ts` uses `internalMutation` — it is not callable from the client.
