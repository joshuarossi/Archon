# Case CRUD Backend (create, get, list, partyStates, updateMyForm)

This ticket implements the foundational Convex backend for case management: creating a case with its initiator context and invite token, listing and fetching cases with party-scoped authorization, reading combined party state (self in full, other party at phase-level only), and updating a party's form fields. These functions are the bedrock on which every phase-specific backend and every frontend view depend. Without them, there is no case to coach, invite, or close.

## Acceptance Criteria

- `cases/create`: creates rows, returns caseId + invite URL
- `cases/list`: returns only caller's cases
- `cases/get`: returns NOT_FOUND if caller not a party
- `cases/partyStates`: returns self (full) and other (phase-only)

## Implementation Notes

`convex/cases.ts` exports all five public Convex functions in this ticket. They call `requireAuth` and `assertPartyToCase` from `convex/lib/auth.ts` (T4) at the top of every function; there is no inline authorization logic.

**`cases/create`** resolves the active template version for the given `templateId` (or the category default from T11), then writes three rows in one mutation transaction:
1. A `cases` row with `status: "DRAFT_PRIVATE_COACHING"`, `schemaVersion: 1`, and the `templateVersionId` pinned at creation time (invariant 1 from §3.2 — it never changes). Include the planning-assumption additions: `inviteeName` (initiator's answer to "What's their name?") and `tokenCount: 0`.
2. A `partyStates` row for the initiator (`role: "INITIATOR"`).
3. An `inviteTokens` row with a 32-char crypto-random token (`crypto.randomUUID().replace(/-/g,'') + crypto.randomUUID().replace(/-/g,'')` sliced to 32 chars), status `ACTIVE`.

For **solo mode** (`isSolo: true`), skip the invite token entirely and instead immediately write a second `partyStates` row for the same userId with `role: "INVITEE"`, and set `inviteeUserId` on the case to the same userId. Advance the status to `BOTH_PRIVATE_COACHING`. The party toggle (`?as=initiator|invitee`) in the UI filters by role, not userId, so two rows for the same user is the correct data shape per the planning assumptions.

Return `{ caseId, inviteUrl }` where `inviteUrl = "${SITE_URL}/invite/${token}"` — read `SITE_URL` from `process.env.SITE_URL` (available in Convex actions/mutations via environment variables).

**`cases/list`** queries both `by_initiator` and `by_invitee` indexes for the caller's userId, merges the results, and returns the combined array sorted by `updatedAt` descending. Never use `.collect()` without an index.

**`cases/get`** looks up the case by `caseId` and returns `null` (caller receives `NOT_FOUND`) if the caller is neither `initiatorUserId` nor `inviteeUserId`. No sensitive partyState data is returned from this function — it returns the `cases` row fields only.

**`cases/partyStates`** returns `{ self: PartyState, otherPhaseOnly: { status, hasCompletedPC } }`. The `self` object includes all `partyStates` fields including `synthesisText`. The `otherPhaseOnly` object must **never** include `synthesisText`, `description`, `desiredOutcome`, or any other private coaching content from the other party — only `{ status, hasCompletedPC: !!otherPartyState.privateCoachingCompletedAt }`. This privacy invariant is load-bearing.

**`cases/updateMyForm`** finds the caller's `partyStates` row by the `by_case_and_user` index and patches `mainTopic`, `description`, and `desiredOutcome`. It throws `CONFLICT` if the case has advanced past `BOTH_PRIVATE_COACHING` (form editing is only valid during private coaching phases).

**Queries used**
- `ctx.db.query("cases").withIndex("by_initiator", q => q.eq("initiatorUserId", userId)).collect()`
- `ctx.db.query("cases").withIndex("by_invitee", q => q.eq("inviteeUserId", userId)).collect()`
- `ctx.db.get(caseId)` — for cases/get and partyStates
- `ctx.db.query("partyStates").withIndex("by_case", q => q.eq("caseId", caseId)).collect()` — in partyStates
- `ctx.db.query("partyStates").withIndex("by_case_and_user", ...)` — in updateMyForm
- `ctx.db.query("templates").withIndex("by_category", ...)` — in cases/create for default template lookup
- `requireAuth(ctx)` from `convex/lib/auth.ts`
- `assertPartyToCase(ctx, caseId, userId)` from `convex/lib/auth.ts`

**Invariants**
- `cases.templateVersionId` is set at creation and never mutated by any function in this file.
- `cases/partyStates` never returns `synthesisText`, `description`, or `desiredOutcome` for the other party.
- `cases/list` never returns cases the caller is not party to.
- `cases/create` with `isSolo: true` must produce two `partyStates` rows (same userId, different roles) and set `inviteeUserId = initiatorUserId` on the case.
- `inviteTokens` row is only created for non-solo cases.
- `cases/updateMyForm` throws `CONFLICT` if `case.status` is not `DRAFT_PRIVATE_COACHING` or `BOTH_PRIVATE_COACHING`.
- Token must be exactly 32 chars and contain only URL-safe characters (hex from UUID after stripping dashes).

**Non-goals**
- This ticket does NOT implement invite redemption (`invites/redeem` — T10).
- This ticket does NOT implement the synthesis trigger or state-machine advancement beyond initial creation (those are T8, T13).
- This ticket does NOT implement case status transitions other than the initial `DRAFT_PRIVATE_COACHING` (or `BOTH_PRIVATE_COACHING` for solo).
- This ticket does NOT implement `jointChat/enter` or any joint-chat functions (T14).
- Admin template management is T11 — this ticket only reads the current template version.
