# Draft Coach backend (session lifecycle, AI conversation, readiness detection, send gate)

The Draft Coach is a private, per-user AI assistant embedded inside the joint chat phase. Its job is to help one party craft a message before it ever enters the shared conversation — giving users a low-stakes space to work out tone, intent, and phrasing. This task implements the full server-side lifecycle: creating a draft session, streaming AI replies, detecting when the user is ready for a polished draft, and handing that draft off into the joint chat through the existing `jointChat/sendUserMessage` mutation. Because Draft Coach conversations are private to the drafter, nothing in this module may ever expose the other party's synthesis text or private coaching content.

## Acceptance Criteria

- `draftCoach/startSession`: creates draftSessions row; throws CONFLICT if case not JOINT_ACTIVE
- Readiness detection: produces { 'draft': '...' } JSON; writes to draftSessions.finalDraft
- `draftCoach/sendFinalDraft`: reads finalDraft; calls jointChat/sendUserMessage internally
- Draft Coach conversation invisible to other party

## Implementation Notes

The `draftSessions` table (defined in T2) tracks per-session state: `status` (`ACTIVE` | `SENT` | `DISCARDED`), `finalDraft`, and the `userId` + `caseId` foreign keys. Each user may have at most one active draft session per case at a time — `startSession` should enforce this by checking for an existing `ACTIVE` session and either returning it or throwing `CONFLICT`.

The `draftCoach/session` query (referenced in `api.draftCoach.session`) returns `{ session, messages } | null` for the calling party. It must only return rows owned by `ctx.auth.getUserIdentity()` — never rows belonging to the other party.

The `generateResponse` internalAction follows the same streaming pattern established in T12 (private coaching): insert a `STREAMING` `draftMessages` row, stream Sonnet tokens in, flush via an internal mutation every ~50 ms, then set `status=COMPLETE`. The prompt is assembled via `assemblePrompt({ role: 'DRAFT_COACH', ... })` from T6 using the category-specific template if available.

**Readiness detection** occurs inside `generateResponse`. After building the AI reply, the action checks whether the returned text parses as `{ "draft": "..." }` JSON (or can be extracted as such). If yes, it writes the cleaned draft string to `draftSessions.finalDraft` via an internal mutation — this is the signal that triggers the `DraftReadyCard` in the UI (T28).

**Privacy boundary** (load-bearing): `generateResponse` builds Draft Coach context from: (a) the drafting user's own `jointMessages`, (b) their own `synthesisText` from `partyStates`. It must NOT query the other party's `synthesisText` or `privateMessages`. Use an indexed query filtering by `userId` for `partyStates`; never `.collect()` all party states for a case and pick.

`draftCoach/sendFinalDraft` is a mutation that reads `draftSessions.finalDraft`, calls `ctx.runMutation(internal.jointChat.sendUserMessage, { caseId, content: finalDraft })`, then sets `draftSessions.status = 'SENT'`. This is the only path through which a Draft Coach conversation enters the shared space.

`draftCoach/discardSession` simply sets `status = 'DISCARDED'`.

`CLAUDE_MOCK=true` applies the same guard as T12: `if (process.env.CLAUDE_MOCK === 'true') { /* return canned stub */ }`.

**Queries used**
- `ctx.db.get(sessionId)` — load draft session
- `ctx.db.query("draftMessages").withIndex("by_session", q => q.eq("draftSessionId", sessionId))` — load session messages
- `ctx.db.query("partyStates").withIndex("by_case_user", q => q.eq("caseId", ...).eq("userId", ...)).first()` — caller's own party state only
- `ctx.db.query("jointMessages").withIndex("by_case", q => q.eq("caseId", ...)).order("asc").take(N)` — joint history for context
- `ctx.runMutation(internal.jointChat.sendUserMessage, ...)` — from `sendFinalDraft`
- `assemblePrompt({ role: "DRAFT_COACH", ... })` from `convex/lib/prompts.ts` (T6)
- `compressIfNeeded(...)` from `convex/lib/transcriptCompression.ts` (T7)

**Invariants**
- `session` query never returns another user's session or messages
- `generateResponse` never reads the other party's `synthesisText` or `privateMessages`
- `sendFinalDraft` throws if `finalDraft` is null (no draft ready yet)
- `startSession` throws `CONFLICT` if case status is not `JOINT_ACTIVE`
- `draftSessions.status` transitions are one-way: `ACTIVE → SENT` or `ACTIVE → DISCARDED`; no re-activation
- `sendMessage` throws if session is not `ACTIVE`

**Non-goals**
- UI components (DraftCoachPanel, DraftReadyCard) — those are T28
- Cost cap enforcement beyond token accumulation — that is T17
- Any access to the other party's private coaching messages or synthesis text

## Stack Practices

- `draftCoach/sendFinalDraft` calling `jointChat/sendUserMessage` via `ctx.runMutation`.
- Privacy enforcement: action must NOT read synthesisText for the other party.
- Anti-pattern: do NOT use `ctx.db.query().collect()` to build context — use index with limit.
