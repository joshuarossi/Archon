# Privacy Response Filter Module

The privacy filter is the second enforcement layer in Conflict Coach's defense-in-depth privacy model (PRD §6.3). It acts as a server-side gate that runs before any AI-generated output is persisted or streamed to a user: if the response contains a substring of ≥ 8 consecutive whitespace-tokenized words drawn from the other party's private messages, the response is rejected for regeneration. Both the Synthesis action (T13) and the Joint Chat Coach action (T14) depend on this module, so it must be complete and well-tested before either AI feature is built.

## Acceptance Criteria

- `filterResponse(response, otherPartyMessages): { passed, matchedSubstring? }`
- passed=false on 8+ consecutive token window match
- Unit tests covering boundary cases and performance

## Implementation Notes

The filter solves a narrow, deterministic problem: "does this string contain a run of ≥ 8 consecutive tokens that also appears as a run in any of the other party's private messages?" Token boundary is whitespace only — splitting on `/\s+/` and filtering empty strings, both inputs lowercased before comparison. This is intentionally simpler than embedding-similarity (tech spec TQ2 locked to token-substring for v1).

**Algorithm:** For each source message in `otherPartyMessages`, tokenize it and slide a window of size 8 across all token positions, joining each window into a single string (space-separated), inserting them into a `Set<string>`. Then slide a window of size 8 across the candidate `response` tokens; if any window string is found in the set, return `{ passed: false, matchedSubstring: windowString }`. Build the set once across all source messages before scanning the response so the hot path is O(response tokens).

**Why 8 tokens?** Short phrases (greetings, common sentences) produce false positives at lower thresholds. 8 consecutive whitespace-token words is long enough to be distinctive while remaining cheap to compute.

**Usage pattern:** Called from `synthesis/generate` (T13) and `generateCoachResponse` (T14) inside Convex internalActions, which are plain async Node.js functions — there is no Convex runtime dependency to avoid; the constraint is simply that the module does not import anything from `convex/server` or `convex/_generated`, so it remains testable in a plain Vitest environment.

**Performance note:** With realistic transcript sizes (hundreds of messages, each tens of words) the set can reach tens of thousands of entries. Pre-computing it in a `Set<string>` keeps lookup O(1). If otherPartyMessages is empty, return `{ passed: true }` immediately without building the set.

## Test-Gen Brief

**Files**
- Implementation: `convex/lib/privacyFilter.ts`
- Tests: `convex/lib/__tests__/privacyFilter.test.ts`

**Exports**
- `filterResponse` — named export, value (function)

**Signatures**
```typescript
export type FilterResult = { passed: true } | { passed: false; matchedSubstring: string };

export function filterResponse(
  response: string,
  otherPartyMessages: string[],
): FilterResult;
```

**Queries used**
- None. Pure in-memory computation; no external calls.

**Invariants**
1. If `otherPartyMessages` is empty, always returns `{ passed: true }`.
2. If `response` has fewer than 8 whitespace tokens, always returns `{ passed: true }` (no window can form).
3. A token run of exactly 7 consecutive matching tokens returns `{ passed: true }`.
4. A token run of exactly 8 consecutive matching tokens returns `{ passed: false }`.
5. A token run of more than 8 consecutive matching tokens returns `{ passed: false }` (any sub-window of 8 will match).
6. Comparison is case-insensitive: upper/lower variation does not defeat detection.
7. Extra whitespace (tabs, multiple spaces) does not produce empty tokens in the window.
8. The function is synchronous — it never returns a Promise.
9. `matchedSubstring` in a failing result is the exact 8-token window string that triggered the match (lowercased, space-joined).

**Non-goals**
- Does not detect paraphrasing, synonyms, or semantic similarity (embedding-based detection is deferred to v1.1 per tech spec TQ2).
- Does not modify the response or suggest alternatives.
- Does not import or use anything from `convex/server`, `convex/values`, or `convex/_generated/`.
- Does not handle crisis detection (that is handled in the prompt/UI layer, T12/T25).

## Stack Practices

- Pure TypeScript module — no Convex runtime imports.
- Tokenize by whitespace: `text.toLowerCase().split(/\s+/).filter(Boolean)`.
- Pre-compute N-gram windows into a Set<string> for O(1) lookup.
- Function must be synchronous.
