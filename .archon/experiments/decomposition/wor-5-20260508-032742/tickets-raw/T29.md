# Closure Flow UI and Closed Case View

The case closure interaction is the final act of mediation — it must give both parties a clear, dignified way to name how the process ended. This ticket implements the shadcn/ui Dialog-based closure modal (three distinct resolution paths: Resolved, Not Resolved, and Take a Break), the per-party rejection flow that clears the proposal banner, and the `/cases/:id/closed` read-only archive view with tabbed access to the joint chat transcript, the caller's private coaching history, and their personalized guidance synthesis. Both parties navigate reactively to the closed view the moment the case status transitions, requiring no refresh.

## Acceptance Criteria

- Closure modal: three options with correct mutations
- Resolved: summary textarea required (min 10 chars); calls proposeClosure
- Not resolved: warning styling; calls unilateralClose
- Take a break: closes modal, no state change
- Rejection: rejectClosure; clears banner
- Confirmation: confirmClosure; both parties route to /closed (reactive)
- /cases/:id/closed: outcome label, closure summary, read-only tabs
- Input and Draft Coach absent in closed view

## Implementation Notes

The closure modal is triggered from within `JointChatView.tsx` (T27), which already renders the closure proposal banner. The modal itself is a `shadcn/ui Dialog` — it does not need to be a separate route; it should be co-located in `src/components/closure/ClosureModal.tsx` and imported into `JointChatView`. Three distinct option blocks are styled as interactive cards; each maps to a distinct Convex mutation from T16.

The **Resolved** path calls `jointChat/proposeClosure` — this sets `closureProposed` on the case and writes to the audit log. Both parties then see the closure proposal banner (built in T27). The second party (or the same party in solo mode) calls `jointChat/confirmClosure` to atomically close as `CLOSED_RESOLVED`. The **Not Resolved** path calls `jointChat/unilateralClose` which closes the case immediately as `CLOSED_UNRESOLVED` for both parties. **Take a Break** simply closes the Dialog with no mutation — it is an escape hatch.

Reactive routing to `/cases/:id/closed` happens inside `JointChatView.tsx` by watching `caseData.case.status` in a `useEffect`. When status enters any `CLOSED_*` value, `navigate('../closed', { replace: true })` fires. This keeps the routing reactive without polling.

The `/cases/:id/closed` page fetches case data via `useOutletContext` (established in T24's `CaseDetail.tsx` layout route). The three tabs — Joint Chat, My Private Coaching, My Guidance — each render their respective read-only data. The `ChatWindow` in all tabs should have `aria-live` removed (no new messages expected) and no `MessageInput`. The `DraftCoachPanel` must not mount in the closed view.

For the `closureSummary` textarea validation in the Resolved path, use controlled state and derive `isValid = closureSummary.trim().length >= 10` inline — do not reach for a form library. The Convex mutation `proposeClosure` should receive the summary as a parameter.

The `partyStates.closureProposalSummary` field (added to the schema in T2) is what backs the closure summary display on the `/closed` page. The `jointChat/mySynthesis` query provides the synthesis tab content — already wired in T26.

Both CLOSED_RESOLVED and CLOSED_UNRESOLVED outcomes must display correctly. CLOSED_ABANDONED (invitee declined invite, set in T8/T10) produces a view with an "Abandoned" outcome badge but no closure summary section.

## Stack Practices

- Closure modal: shadcn/ui Dialog with custom content and three option buttons.
- Tab component: shadcn/ui Tabs, TabsList, TabsTrigger, TabsContent. Default tab: 'Joint Chat'.
- Reactive route update to /closed: watch caseData.case.status in JointChatView useEffect.
- Summary textarea validation: isValid = closureSummary.trim().length >= 10.
- Anti-pattern: do NOT render the other party's private coaching tab in closed view.

## Design Practices

- Closure modal (design doc §4.11): shadcn/ui Dialog, shadow-3 elevation. Title: 'How would you like to close this case?' Header in 500 weight --text-primary. Three option blocks styled as cards with border --border-default, 16px padding. Hover: bg-surface-subtle, 150ms ease-out.
- 'Resolved' option: --accent accent-left-border (4px) on selected card. Textarea 'Briefly describe what you agreed to.' (5 rows, border --border-strong). Character count: right-aligned 12px --text-tertiary. 'Propose Resolution' = primary --accent, disabled until textarea.length >= 10.
- 'Not resolved' option: warning treatment — border-left 4px --danger, card bg subtle danger tint on select. 'Close without resolution' button = shadcn/ui Button variant='destructive' styled with --danger. 'This closes the case immediately for both of you.' in --text-secondary.
- 'Take a break' option: ghost/neutral card. 'Close' button = secondary outlined --text-secondary. No state change.
- Closed case view (design doc §4.12): bg-canvas page. Header: case name, category pill, closure date in --text-secondary, outcome badge (Resolved = --accent, Not Resolved = --danger, Abandoned = --text-tertiary). Outcome badge: 12px text, bg-surface-subtle, rounded-full, px-2 py-1.
- Closure summary (when Resolved): bg-accent-subtle (#DCE7E0), 16px padding, border-radius 8px, ShieldCheck icon, --text-primary 15px prose. Prominent placement below header.
- Closed case tabs: shadcn/ui Tabs. TabsList with three triggers: 'Joint Chat' | 'My Private Coaching' | 'My Guidance'. Active trigger: --accent border-bottom. ChatWindow in readOnly mode (no MessageInput, no Draft Coach button).
- 'This case is closed' banner: bg-surface-subtle, border-bottom --border-default, 12px --text-secondary, centered. Full-width between PhaseHeader and tab content. NO input, NO Draft Coach.
- Accessibility: closure Dialog focus-trapped, Esc closes (shadcn/ui handles). Tabs use correct ARIA tab pattern (role='tablist', role='tab', role='tabpanel') via shadcn/ui. Closed state read-only ChatWindow: message list still has role='log' but aria-live removed (no new messages expected).
