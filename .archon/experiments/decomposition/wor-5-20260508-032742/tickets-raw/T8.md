# Case State Machine Module

The case state machine is the authoritative gatekeeper for every lifecycle transition in Conflict Coach. Every mutation that changes `cases.status` must call one of the named transition functions defined here — no mutation is permitted to set status directly. This module is a pure, side-effect-free TypeScript module: it receives the current status value, validates the transition, and returns the new status (or throws a typed error if the transition is illegal). Centralizing this logic in one file makes it trivially testable and prevents state-skipping bugs from surfacing in individual mutations. T8 also explicitly covers the invitee-decline path (`DRAFT_PRIVATE_COACHING → CLOSED_ABANDONED`), which was added as a planning assumption and must be verified by name in the test suite.

## Acceptance Criteria

- Each transition is pure: takes state, returns new status or throws CONFLICT
- DRAFT_PRIVATE_COACHING→CLOSED_ABANDONED: invitee decline, explicitly tested
- All 9 valid transitions and 8+ illegal transitions tested

## Implementation Notes

The state diagram from the tech spec (§5.1) defines the following valid states:

```
DRAFT_PRIVATE_COACHING → BOTH_PRIVATE_COACHING   (invitee accepts + joins)
DRAFT_PRIVATE_COACHING → CLOSED_ABANDONED         (invitee declines — planning assumption)
BOTH_PRIVATE_COACHING  → READY_FOR_JOINT          (both mark PC complete + synthesis done)
READY_FOR_JOINT        → JOINT_ACTIVE             (first party enters joint chat)
JOINT_ACTIVE           → CLOSED_RESOLVED          (mutual closure confirmed)
JOINT_ACTIVE           → CLOSED_UNRESOLVED        (unilateral close)
JOINT_ACTIVE           → CLOSED_ABANDONED         (cron: 30-day inactivity)
JOINT_ACTIVE           → COST_LIMITED             (hard budget cap hit)
COST_LIMITED           → CLOSED_RESOLVED          (parties can still close manually)
```

That is exactly 9 valid transitions. The `COST_LIMITED` status was added in T2 (`planning_assumptions[2]`) and must appear in both the schema and this module. `CLOSED_RESOLVED`, `CLOSED_UNRESOLVED`, and `CLOSED_ABANDONED` are terminal: no further transition is valid from them.

Each exported function follows the same contract: it accepts the current `CaseStatus` string, asserts it is the expected source state, and returns the target state. If the source state is wrong it throws a `ConvexError` with `code: "CONFLICT"`. Because these are pure functions, the caller (the Convex mutation) is responsible for the actual `ctx.db.patch()` call with the returned value.

The module has **zero** imports from `convex/values`, `convex/server`, or the Anthropic SDK — it is a plain TypeScript module that Vitest can import directly without any Convex runtime. The `ConvexError`-compatible shape is achieved by throwing a plain `Error` subclass or a simple `{ code, message }` object that callers match on; whichever approach is used must be consistent with how `T4`'s `ConvexError` codes are defined so mutations can re-throw correctly. If T4 is complete, import the error factory from `convex/lib/errors.ts`; if not, inline a minimal throw with `code: "CONFLICT"` that will be wired up when T4 lands.

The test file (`convex/lib/__tests__/stateMachine.test.ts`) should be written as a standard Vitest spec. Each valid transition gets its own `it()` block; illegal transitions are grouped. The decline-invite path (`inviteeDeclines`) must have its own named `it('inviteeDeclines: DRAFT_PRIVATE_COACHING → CLOSED_ABANDONED')` block to satisfy the AC's "explicitly tested" requirement.

### Signatures

```typescript
// Type alias matching the union defined in convex/schema.ts (T2)
type CaseStatus =
  | "DRAFT_PRIVATE_COACHING"
  | "BOTH_PRIVATE_COACHING"
  | "READY_FOR_JOINT"
  | "JOINT_ACTIVE"
  | "CLOSED_RESOLVED"
  | "CLOSED_UNRESOLVED"
  | "CLOSED_ABANDONED"
  | "COST_LIMITED";

// Each function: assert source state; return target state; throw on mismatch
function inviteeJoins(status: CaseStatus): CaseStatus;
function inviteeDeclines(status: CaseStatus): CaseStatus;
function bothCompletePrivateCoaching(status: CaseStatus): CaseStatus;
function enterJoint(status: CaseStatus): CaseStatus;
function confirmClosure(status: CaseStatus): CaseStatus;
function unilateralClose(status: CaseStatus): CaseStatus;
function abandonByInactivity(status: CaseStatus): CaseStatus;
function hitBudgetCap(status: CaseStatus): CaseStatus;
function closeCostLimited(status: CaseStatus): CaseStatus;
```

### Queries Used

None — this is a pure computation module with no external calls.

### Invariants

- Every function returns a value only when the input is the **exact** expected source state; any other input throws.
- Terminal states (`CLOSED_RESOLVED`, `CLOSED_UNRESOLVED`, `CLOSED_ABANDONED`) are never valid inputs to any transition function.
- `BOTH_PRIVATE_COACHING` and `DRAFT_PRIVATE_COACHING` cannot transition directly to `JOINT_ACTIVE`, `CLOSED_RESOLVED`, or `CLOSED_UNRESOLVED`.
- `COST_LIMITED` can only move to `CLOSED_RESOLVED` (via `closeCostLimited`); it cannot move to `CLOSED_UNRESOLVED` or `CLOSED_ABANDONED` via a state-machine function (those closures still apply at the cron / unilateral layer outside this module — TBD if callers need additional functions, but the plan specifies 9 valid transitions).
- The thrown error must carry `code: "CONFLICT"` to match the error code table in T4.

### Non-goals

- This module does NOT read from or write to the Convex database (`ctx.db`).
- It does NOT validate business rules beyond status correctness (e.g., it does not check whether both `partyStates` rows have `status === "PRIVATE_COACHING_COMPLETE"` before allowing `bothCompletePrivateCoaching` — that pre-check belongs in the calling mutation).
- It does NOT handle `partyStates` status transitions (per-party phase state is separate from case status).
- No COST_LIMITED → CLOSED_UNRESOLVED or COST_LIMITED → CLOSED_ABANDONED transition functions are defined here; those paths are handled by `unilateralClose` and `abandonByInactivity` being called by mutations/crons that separately guard on the status.

## Stack Practices

- Pure TypeScript — zero Convex runtime or SDK imports.
- Export one named function per transition.
- Anti-pattern: do NOT read from ctx.db inside state machine functions.
