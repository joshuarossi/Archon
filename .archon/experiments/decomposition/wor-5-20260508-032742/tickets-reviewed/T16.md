# Case Closure Backend

Once both parties are in the joint session, they need a structured way to formally close the case — whether by mutual agreement, unilateral walkaway, or system-detected abandonment. This task implements all closure mutations plus the daily Convex cron that auto-closes stale `JOINT_ACTIVE` cases after 30 days of inactivity. It completes the server-side case lifecycle, enabling the closure UI (T29) to wire directly against these endpoints.

## Acceptance Criteria

- `jointChat/proposeClosure`: sets closureProposed; writes to auditLog
- `jointChat/confirmClosure`: atomically closes case as CLOSED_RESOLVED
- Cron: daily scan for JOINT_ACTIVE cases with 30d inactivity

## Implementation Notes

**State machine integration.** All status transitions in this task go through `convex/lib/stateMachine.ts` (T8). Import the module as a namespace to avoid a naming collision with the local `confirmClosure` mutation export:

```typescript
import * as stateMachine from '../lib/stateMachine';
```

Then call:
- `stateMachine.confirmClosure(caseDoc.status)` — inside the `confirmClosure` mutation
- `stateMachine.unilateralClose(caseDoc.status)` — inside the `unilateralClose` mutation
- `stateMachine.abandonByInactivity(caseDoc.status)` — inside `checkAbandonedCases` for `JOINT_ACTIVE` cases

T8's functions throw plain objects `{ code: "CONFLICT", message: string }` (not `ConvexError`). Catch them and re-throw using the `ConvexError` factory from `convex/lib/errors.ts` (T4):

```typescript
try {
  const newStatus = stateMachine.confirmClosure(caseDoc.status);
} catch (e) {
  throw makeConvexError((e as { code: string; message: string }).code, (e as { message: string }).message);
}
```

**Propose / confirm flow.** Two separate mutations implement the two-step closure:
1. `jointChat/proposeClosure` — sets `partyStates.closureProposed = true` for the calling party, writes `summary` to the calling party's `partyStates.closureProposalSummary` (NOT to `cases.closureSummary` — that field is only written on confirm), and appends an `auditLog` entry with `action: "CLOSURE_PROPOSED"`, `targetType: "case"`, and `targetId: caseId.toString()`.
2. `jointChat/confirmClosure` — reads both parties' `partyStates` rows; throws `CONFLICT` if the opposing party (not the caller) has not set `closureProposed = true`; reads the proposing party's `closureProposalSummary` and writes it to `cases.closureSummary`; calls `stateMachine.confirmClosure(caseDoc.status)` to get the new status; atomically patches `cases` with `{ status: "CLOSED_RESOLVED", closedAt: Date.now(), closureSummary }`; sets `partyStates.closureConfirmed = true` for the confirming party; appends `auditLog` entry `"CLOSURE_CONFIRMED"`.

**Why `partyStates.closureProposalSummary` and not `cases.closureSummary` on propose?** `cases.closureSummary` is the canonical shared summary, written only once (at confirm), to avoid overwriting it if the proposal is rejected or re-proposed with a different text.

**Reject closure.** `jointChat/rejectClosure` reads both partyStates rows, clears `closureProposed = false` and `closureProposalSummary` for any party that had `closureProposed = true` (there may be at most one), and appends an `auditLog` entry with `action: "CLOSURE_REJECTED"`. The case remains `JOINT_ACTIVE`.

**Unilateral close.** `jointChat/unilateralClose` takes `{ caseId, reason }` and immediately calls `stateMachine.unilateralClose(caseDoc.status)` to get `CLOSED_UNRESOLVED`. It patches `cases` with `{ status: "CLOSED_UNRESOLVED", closedAt: Date.now(), closureSummary: reason }` and writes `auditLog` entry `"CASE_CLOSED_UNRESOLVED"` with `metadata: { reason }`. Authorization: any party to a `JOINT_ACTIVE` case can call this.

**Cron for abandoned cases.** `convex/crons.ts` registers a daily job via `cronJobs()` from `convex/server`. The job body is an internal mutation that:
1. Queries all cases where `status === "JOINT_ACTIVE"` using a full table scan with filter.
2. Also queries all cases where `status === "COST_LIMITED"` (T8 note: COST_LIMITED→CLOSED_ABANDONED is a caller-side direct patch, not a state-machine function; patch these rows directly without calling `stateMachine.abandonByInactivity`).
3. Filters both result sets to those where `updatedAt < Date.now() - 30 * 24 * 60 * 60 * 1000`.
4. For `JOINT_ACTIVE` matches: calls `stateMachine.abandonByInactivity(status)` to validate the transition, then patches `{ status: "CLOSED_ABANDONED", closedAt: Date.now() }`.
5. For `COST_LIMITED` matches: patches `{ status: "CLOSED_ABANDONED", closedAt: Date.now() }` directly (no state machine call — this is the documented bypass per T8 Implementation Notes).
6. Writes an `auditLog` entry for each closure: `action: "CASE_AUTO_ABANDONED"`, `actorUserId: caseDoc.initiatorUserId` (system sentinel for v1 — no system user row exists).

Do not use `setInterval`, external cron services, or `setTimeout`. The Convex `cronJobs()` scheduler is the only correct mechanism (tech spec §5.3).

**Authorization.** All mutations call `requireAuth(ctx)` and `assertPartyToCase(ctx, caseId, user._id)` from `convex/lib/auth.ts` (T4). The cron internal mutation bypasses user auth (it is an `internalMutation`).

**Audit log shape.** All audit writes use the `auditLog` table defined in T2: `{ actorUserId, action, targetType: "case", targetId: caseId.toString(), metadata: { summary?, reason? }, createdAt: Date.now() }`.

**`partyStates.closureProposalSummary`.** The planning assumptions added `closureProposalSummary: v.optional(v.string())` to `partyStates`. Use this field on the proposing party's row to hold the per-party record of what they proposed. The canonical `closureSummary` on the `cases` row is the shared source of truth, written only by `confirmClosure`.

**Idempotency guards.** `proposeClosure` should be idempotent if called twice by the same party (re-writing the same proposal is allowed; no error). `confirmClosure` throws `CONFLICT` if the case is already `CLOSED_*`.

### Queries used
- `ctx.db.get(caseId)` — load case
- `ctx.db.query("partyStates").withIndex("by_case", q => q.eq("caseId", caseId)).collect()` — both party states for confirmClosure, rejectClosure, proposeClosure
- `ctx.db.query("cases").filter(q => q.eq(q.field("status"), "JOINT_ACTIVE")).collect()` — cron: all active cases (then filter by updatedAt in JS)
- `ctx.db.query("cases").filter(q => q.eq(q.field("status"), "COST_LIMITED")).collect()` — cron: all cost-limited cases (direct patch, no state machine)
- `ctx.db.patch(caseId, { status, closedAt, closureSummary })` — update case
- `ctx.db.patch(partyStateId, { closureProposed, closureConfirmed, closureProposalSummary })` — update party state
- `ctx.db.insert("auditLog", { ... })` — audit entry
- `requireAuth(ctx)` from `convex/lib/auth.ts` (T4)
- `assertPartyToCase(ctx, caseId, user._id)` from `convex/lib/auth.ts` (T4)
- State machine functions from `convex/lib/stateMachine.ts` (T8) — imported as `stateMachine.*`:
  - `stateMachine.confirmClosure(currentStatus)` — called in `confirmClosure` mutation
  - `stateMachine.unilateralClose(currentStatus)` — called in `unilateralClose` mutation
  - `stateMachine.abandonByInactivity(currentStatus)` — called in `checkAbandonedCases` for JOINT_ACTIVE cases only
- T8 error re-throw: catch `{ code: "CONFLICT" }` plain objects → re-throw as `ConvexError` via T4's factory (`makeConvexError` from `convex/lib/errors.ts`)

### Invariants
- `cases.status` only changes via `stateMachine.*` transition functions (for JOINT_ACTIVE→* paths); COST_LIMITED→CLOSED_ABANDONED is the one documented bypass where the cron patches directly.
- A `CLOSED_*` case can never be re-opened by any of these mutations; all throw `CONFLICT` if `case.status` is already a closed state.
- `auditLog` entry is written in the same mutation transaction as the state change — they are atomic.
- `confirmClosure` requires the opposing party (not the caller) to have `closureProposed = true` before it can proceed.
- `cases.closureSummary` is written only by `confirmClosure` (copying from the proposing party's `partyStates.closureProposalSummary`). `proposeClosure` writes only to `partyStates.closureProposalSummary`.
- `crons.ts` uses `internalMutation` — `checkAbandonedCases` is not callable from the client.
- `rejectClosure` always writes an `auditLog` entry with `action: "CLOSURE_REJECTED"`, even if no proposal was active (it is a no-op but still audited).

## Stack Practices

- Cron setup: `convex/crons.ts` using `cronJobs()` from `convex/server`. Export `default crons` — do NOT export `checkAbandonedCases` as the default.
- Anti-pattern: do NOT use `setInterval` or external cron services.
- **Import T8's stateMachine as a namespace** (`import * as stateMachine from '../lib/stateMachine'`) to avoid name collision between the local `confirmClosure` mutation export and the T8 pure function of the same name. Call `stateMachine.confirmClosure(caseDoc.status)`, `stateMachine.unilateralClose(...)`, `stateMachine.abandonByInactivity(...)`.
- **Re-throw T8 errors as ConvexError.** T8 functions throw plain `{ code: "CONFLICT", message: string }` objects (not Error subclasses, not ConvexError). In every mutation that calls a stateMachine function, wrap the call in try/catch and re-throw using the `makeConvexError` factory from `convex/lib/errors.ts` (T4).
- **These are regular Convex `mutation` handlers — `ctx.db` is allowed directly.** Unlike the `internalAction` pattern in T14/T15, closure mutations are ordinary mutations and do NOT need `ctx.runQuery`/`ctx.runMutation` wrappers for database access.
- **COST_LIMITED bypass in cron.** Per T8's documented bypass, `COST_LIMITED → CLOSED_ABANDONED` has no state-machine function. The cron patches these rows directly; add a comment referencing the T8 Non-goals section so future readers understand this is intentional.

## Design Practices

- **Two-step propose/confirm vs one-step unilateral close.** Mutual closure requires explicit agreement from both parties (propose + confirm) to prevent surprise terminations. Unilateral close is deliberately one-step and available to any party — it is the escape hatch when one party wants to leave regardless of the other's position. The asymmetry matches the product intent: negotiated closure preserves the "CLOSED_RESOLVED" status and shared summary; unilateral closure produces "CLOSED_UNRESOLVED" and signals the breakdown.
- **`cases.closureSummary` written only on confirm.** Writing the canonical summary only at confirm-time (copied from the proposing party's `partyStates.closureProposalSummary`) means rejected proposals leave no trace on the case record. If `proposeClosure` wrote to `cases.closureSummary` directly, a rejected proposal would require a compensating write to clear it; the per-party `closureProposalSummary` field is the staging area that avoids this.
- **`initiatorUserId` as cron audit sentinel.** The cron has no authenticated user, but `auditLog.actorUserId` is `v.id("users")` (non-optional). In v1, using the case's `initiatorUserId` as a sentinel is the lowest-friction option that keeps the audit log schema intact. A dedicated system user row would be cleaner but requires a migration and is deferred; the sentinel is documented in a comment so the upgrade path is clear.
