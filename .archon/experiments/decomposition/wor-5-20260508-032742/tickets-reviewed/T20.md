# Landing Page and Login/Register Page

T20 implements the two entry-point views for Conflict Coach: the public landing page at `/` and the authentication page at `/login`. Together these are the first surfaces new users encounter, so they must clearly communicate the product's calm, trustworthy purpose and provide a minimal-friction path into the app. The landing page converts skeptical visitors by explaining the process in plain language; the login page completes authentication via magic link or Google OAuth with zero password friction, wiring directly into the Convex Auth providers configured in T3.

## Acceptance Criteria

- `/` displays tagline, 3-step explainer, privacy section, footer with links
- `/` has 'Start a case' CTA routing to /login or /dashboard
- `/login`: centered ~400px card; email + magic link button; Google OAuth button
- Submitting email triggers magic link; form replaced by confirmation state
- Error state inline (not a toast)
- Fine print links to /terms and /privacy
- After auth: redirect to /dashboard

## Implementation Notes

The landing page (§4.1 of the tech spec) is intentionally sparse: a single-sentence tagline, a three-step explainer (Private Coaching → Shared Conversation → Resolution), a privacy trust section, and a footer. Per design doc §1.1 ("Calm over Clever"), there are no testimonials, no pricing, and no aggressive repeated CTAs. The sole primary action is "Start a case" which routes logged-out users to `/login` and logged-in users directly to `/dashboard`. This routing decision can be made with `useConvexAuth().isAuthenticated` from `@convex-dev/auth/react`.

While `isLoading` from `useConvexAuth` is true, the landing page CTA should render in its default state without routing (avoid pre-render flash). On the login page, do not call `navigate('/dashboard')` until both `isLoading === false` and `isAuthenticated === true`.

The login page (§4.2) is a single centered card with two auth paths:
1. **Magic link**: `const { signIn } = useAuthActions(); await signIn('resend', { email })` — disable the "Send magic link" button while the call is in-flight (use local `isPending` state), then on success replace the form with a confirmation message.
2. **Google OAuth**: `await signIn('google')` — triggers redirect flow.

Both calls are already configured in T3 (`convex/auth.config.ts`). After successful auth, Convex Auth handles the session; the page should watch `isAuthenticated` and — only when `isLoading` is also false — `navigate('/dashboard', { replace: true })`.

Error states must render inline below the triggering input (not as a toast), since design doc §6.2 reserves toasts for system-level events. Use a try/catch around `signIn()` and store the error in local state. Render with `role="alert"` so screen readers announce it.

The `/terms` and `/privacy` routes referenced in the footer are static placeholder routes that simply render the page title; they do not require a separate task.

Route file locations follow the existing structure: `src/routes/LandingPage.tsx` and `src/routes/LoginPage.tsx`. The T19a/T19b shared components (Button via shadcn/ui, Lucide icons) are available since T20 depends on T19a and T19b.

### Queries used

- `useAuthActions` from `@convex-dev/auth/react` — `{ signIn }` for both magic link and Google OAuth

### Invariants
- The "Start a case" CTA must never navigate a logged-in user to `/login`; it must go directly to `/dashboard`.
- While `isLoading` is `true`, the landing CTA must not perform any navigation (no premature redirect before auth state is known).
- Magic link submission must call `signIn('resend', { email })` — never a POST to a custom endpoint.
- Google OAuth must call `signIn('google')` — never a custom redirect.
- Error state must not use a toast; it must render inline with `role="alert"`.
- After auth completes (`isAuthenticated === true` AND `isLoading === false`), navigate to `/dashboard` with `replace: true` so the back-button doesn't loop back to `/login`.
- While a magic link request is in-flight (`isPending === true`), the "Send magic link" button must be disabled.
- The magic-link-sent confirmation state must replace the form entirely (not appear alongside it).
- No password input field exists anywhere in `LoginPage`.
- Both pages must render a single `<main>` landmark element wrapping their primary content.
- The `<h1>` on both pages must carry `tabIndex={-1}` so React Router's focus management can announce route changes to screen readers (per T33 pattern).

### Non-goals
- `/terms` and `/privacy` static pages are not part of this ticket.
- No animated hero graphics or illustrations.
- No post-login onboarding flow — straight to `/dashboard`.
- No "Didn't get it? Try Google instead." fallback email-delay handling (that is a UX enhancement, not a v1 requirement here, though noted in the spec's error table at §7.x).

## Stack Practices

- Magic link submit: `const { signIn } = useAuthActions(); await signIn('resend', { email })`.
- Google OAuth: `await signIn('google')`.
- In-flight guard: use local `isPending` boolean; set `true` before `signIn` call, `false` in finally block; bind to button's `disabled` prop.
- Navigation guard: `if (!isLoading && isAuthenticated) navigate('/dashboard', { replace: true })` — never navigate when `isLoading` is true.
- Error display: catch errors in try/catch; render inline with `role="alert"`.
- Anti-pattern: do NOT use `<form action=...>` — handle via onSubmit + e.preventDefault().

## Design Practices

- Landing page tone (design doc §1.1 Calm over Clever): one-sentence tagline ('A calm place to work through a difficult conversation.'), short subhead. No testimonials, no aggressive repeated CTAs, no gamification. Colors --bg-canvas background, --text-primary body. Primary CTA 'Start a case' uses --accent (#6B8E7F) filled button — the sole primary action on the page (design doc §1.4).
- Landing 3-step explainer: minimal iconographic layout using Lucide icons (MessageCircle, Users, ShieldCheck at 1.5px stroke). Max-width 720px centered. Section padding 48px vertical. No decorative illustrations — presence over polish (design doc §1.5).
- Landing footer: --text-tertiary, 12px, border-top --border-default. Links to /terms and /privacy. Per design doc §4.1, no pricing section.
- Login page (design doc §4.2): single centered card ~400px wide, bg-surface, shadow-2 ('0 4px 12px rgba(0,0,0,0.06)'). Background --bg-canvas. Heading 'Sign in to Conflict Coach' at 500 weight, --text-primary. Email Input: border --border-strong, 15px Inter. 'Send magic link' = primary --accent filled button (full width). Divider 'or' in --text-tertiary. 'Continue with Google' = secondary outlined button (full width).
- Login error state: inline below triggering input, 14px --danger (#B5594D), role='alert' for screen reader announcement. NOT a toast — per design doc §6.2.
- Login magic-link-sent state: replace form with confirmation message in --text-secondary. 'Check your email…' — no animation celebrating it.
- Fine print: 12px --text-tertiary below both buttons. Links to /terms and /privacy underlined in --accent.
- Accessibility: single <main> landmark on both pages. Login card inputs must have <label> elements. Focus visible ring 2px --accent. Tab order: email → send button → divider → Google button → fine print links. Route change announces h1 to screen readers (tabIndex={-1} on h1, per T33 pattern).
