# Case State Machine Module

The case state machine is the authoritative gatekeeper for every lifecycle transition in Conflict Coach. Every mutation that changes `cases.status` must call one of the named transition functions defined here — no mutation is permitted to set status directly. This module is a pure, side-effect-free TypeScript module: it receives the current status value, validates the transition, and returns the new status (or throws a typed error if the transition is illegal). Centralizing this logic in one file makes it trivially testable and prevents state-skipping bugs from surfacing in individual mutations. T8 also explicitly covers the invitee-decline path (`DRAFT_PRIVATE_COACHING → CLOSED_ABANDONED`), which was added as a planning assumption and must be verified by name in the test suite.

## Acceptance Criteria

- AC1: Each transition is pure: takes state, returns new status or throws CONFLICT
- AC2: DRAFT_PRIVATE_COACHING→CLOSED_ABANDONED: invitee decline, explicitly tested
- AC3: All 9 valid transitions and 8+ illegal transitions tested

## Implementation Notes

The state diagram from the tech spec (§5.1) defines the following valid states. The table below is the canonical mapping from exported function name to transition:

| Function | Source State | Target State |
|---|---|---|
| `inviteeJoins` | `DRAFT_PRIVATE_COACHING` | `BOTH_PRIVATE_COACHING` |
| `inviteeDeclines` | `DRAFT_PRIVATE_COACHING` | `CLOSED_ABANDONED` |
| `bothCompletePrivateCoaching` | `BOTH_PRIVATE_COACHING` | `READY_FOR_JOINT` |
| `enterJoint` | `READY_FOR_JOINT` | `JOINT_ACTIVE` |
| `confirmClosure` | `JOINT_ACTIVE` | `CLOSED_RESOLVED` |
| `unilateralClose` | `JOINT_ACTIVE` | `CLOSED_UNRESOLVED` |
| `abandonByInactivity` | `JOINT_ACTIVE` | `CLOSED_ABANDONED` |
| `hitBudgetCap` | `JOINT_ACTIVE` | `COST_LIMITED` |
| `closeCostLimited` | `COST_LIMITED` | `CLOSED_RESOLVED` |

That is exactly 9 valid transitions. The `COST_LIMITED` status was added in T2 (`planning_assumptions[2]`) and must appear in both the schema and this module. `CLOSED_RESOLVED`, `CLOSED_UNRESOLVED`, and `CLOSED_ABANDONED` are terminal: no further transition is valid from them.

**Error shape**: Each function throws a plain object `{ code: "CONFLICT", message: string }` — not a subclass of `Error`, not `ConvexError`. Because T8 has zero Convex imports, it cannot use `ConvexError` from `convex/values`. Callers (Convex mutations in T9, T10, T16) catch this object and re-throw as a `ConvexError` using the `convex/lib/errors.ts` factory from T4. This pattern isolates the pure state machine from the Convex runtime and keeps it importable by Vitest directly.

**COST_LIMITED → terminal states (bypass cases)**: The state machine defines only one transition _out of_ `COST_LIMITED` — `closeCostLimited` → `CLOSED_RESOLVED`. If a COST_LIMITED case needs to reach `CLOSED_UNRESOLVED` (user unilaterally closes) or `CLOSED_ABANDONED` (inactivity cron), the calling mutation/cron patches the status directly without calling a state-machine function, because those edge paths are not represented by a named transition in the plan. This is intentional and documented; callers must guard on `status === "COST_LIMITED"` before doing so.

Each exported function follows the same contract: it accepts the current `CaseStatus` string, asserts it is the expected source state, and returns the target state. If the source state is wrong it throws `{ code: "CONFLICT", message: "<context>" }`. Because these are pure functions, the caller (the Convex mutation) is responsible for the actual `ctx.db.patch()` call with the returned value.

The module has **zero** imports from `convex/values`, `convex/server`, or the Anthropic SDK — it is a plain TypeScript module that Vitest can import directly without any Convex runtime.

The test file (`convex/lib/__tests__/stateMachine.test.ts`) should be written as a standard Vitest spec. Each valid transition gets its own `it()` block; illegal transitions are grouped. The decline-invite path (`inviteeDeclines`) must have its own named `it('inviteeDeclines: DRAFT_PRIVATE_COACHING → CLOSED_ABANDONED')` block to satisfy AC2's "explicitly tested" requirement.

type CaseStatus =
  | "DRAFT_PRIVATE_COACHING"
  | "BOTH_PRIVATE_COACHING"
  | "READY_FOR_JOINT"
  | "JOINT_ACTIVE"
  | "CLOSED_RESOLVED"
  | "CLOSED_UNRESOLVED"
  | "CLOSED_ABANDONED"
  | "COST_LIMITED";

// Each function: assert source state; return target state; throw on mismatch

function closeCostLimited(status: CaseStatus): CaseStatus;

### Queries Used

None — this is a pure computation module with no external calls.

### Invariants

- Every function returns a value only when the input is the **exact** expected source state; any other input throws `{ code: "CONFLICT", message: string }`.
- Terminal states (`CLOSED_RESOLVED`, `CLOSED_UNRESOLVED`, `CLOSED_ABANDONED`) are never valid inputs to any transition function.
- `BOTH_PRIVATE_COACHING` and `DRAFT_PRIVATE_COACHING` cannot transition directly to `JOINT_ACTIVE`, `CLOSED_RESOLVED`, or `CLOSED_UNRESOLVED`.
- `COST_LIMITED` can only move to `CLOSED_RESOLVED` (via `closeCostLimited`) within this module; COST_LIMITED→CLOSED_UNRESOLVED and COST_LIMITED→CLOSED_ABANDONED are caller-side direct patches (not state-machine functions).
- The thrown error object must carry `code: "CONFLICT"` so callers can match on it and wrap in a `ConvexError` (T4).

### Non-goals

- This module does NOT read from or write to the Convex database (`ctx.db`).
- It does NOT validate business rules beyond status correctness (e.g., it does not check whether both `partyStates` rows have `status === "PRIVATE_COACHING_COMPLETE"` before allowing `bothCompletePrivateCoaching` — that pre-check belongs in the calling mutation).
- It does NOT handle `partyStates` status transitions (per-party phase state is separate from case status).
- No `COST_LIMITED → CLOSED_UNRESOLVED` or `COST_LIMITED → CLOSED_ABANDONED` transition functions are defined here; those paths are handled by mutations/crons patching directly (see Implementation Notes).

## Stack Practices

- Pure TypeScript — zero Convex runtime or SDK imports.
- Export one named function per transition.
- Error shape: throw `{ code: "CONFLICT", message: string }` — a plain object, not an Error subclass. Callers (mutations) wrap this in `ConvexError` using T4's factory.
- Anti-pattern: do NOT read from ctx.db inside state machine functions.
- Anti-pattern: do NOT import `ConvexError` from `convex/values` in this file — it would break Vitest isolation.

## Design Practices

- Pure module: no side effects, no module-level state, no singletons. This module is designed to be imported directly by Vitest without mocking anything.
- YAGNI: define exactly the 9 transition functions that correspond to the 9 valid transitions. Do not add a general-purpose `transition(from, to)` dispatcher or a transitions map — the explicit named functions are the point.
- `depends_on: T2` is an ordering dependency only: T8 needs the canonical `CaseStatus` union from `convex/schema.ts` (T2) to type the parameter and return correctly. At runtime, T8 has no Convex dependency.
