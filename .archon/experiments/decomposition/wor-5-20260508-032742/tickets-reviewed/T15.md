# Draft Coach backend (session lifecycle, AI conversation, readiness detection, send gate)

The Draft Coach is a private, per-user AI assistant embedded inside the joint chat phase. Its job is to help one party craft a message before it ever enters the shared conversation — giving users a low-stakes space to work out tone, intent, and phrasing. This task implements the full server-side lifecycle: creating a draft session, streaming AI replies, detecting when the user is ready for a polished draft, and handing that draft off into the joint chat through the existing `jointChat/sendUserMessage` mutation. Because Draft Coach conversations are private to the drafter, nothing in this module may ever expose the other party's synthesis text or private coaching content.

## Acceptance Criteria

- `draftCoach/startSession`: creates draftSessions row; throws CONFLICT if case not JOINT_ACTIVE
- Readiness detection: produces { 'draft': '...' } JSON; writes to draftSessions.finalDraft
- `draftCoach/sendFinalDraft`: reads finalDraft; calls jointChat/sendUserMessage internally
- Draft Coach conversation invisible to other party

## Implementation Notes

The `draftSessions` table (defined in T2) tracks per-session state: `status` (`ACTIVE` | `SENT` | `DISCARDED`), `finalDraft`, and the `userId` + `caseId` foreign keys. Each user may have at most one active draft session per case at a time — `startSession` should enforce this by checking for an existing `ACTIVE` session and either returning it or throwing `CONFLICT`.

The `draftCoach/session` query (referenced in `api.draftCoach.session`) returns `{ session, messages } | null` for the calling party. It must only return rows owned by `ctx.auth.getUserIdentity()` — never rows belonging to the other party.

The `generateResponse` internalAction follows the same streaming pattern established in T12 and T14: insert a `STREAMING` `draftMessages` row via `ctx.runMutation(internal.draftCoach.writeDraftMessage, { sessionId, status: 'STREAMING', content: '' })`, stream `claude-sonnet-4-5` tokens in, patch via the same mutation every ~50 ms, then set `status=COMPLETE`. The prompt is assembled via `assemblePrompt({ role: 'DRAFT_COACH', ... })` from `convex/lib/prompts.ts` (T6) using the category-specific template if available.

> **Important:** `generateResponse` is an `internalAction` and therefore CANNOT access `ctx.db` directly. All database reads must go through `ctx.runQuery(internal.draftCoach.getDraftCoachContext, { sessionId })` and all writes through `ctx.runMutation(internal.draftCoach.writeDraftMessage, ...)` or `ctx.runMutation(internal.draftCoach.recordDraftTokenUsage, ...)`. Define the necessary `internalQuery` and `internalMutation` helpers in the same file. Violating this causes a Convex runtime error.

**Readiness detection** occurs inside `generateResponse`. After building the AI reply, the action checks whether the returned text parses as `{ "draft": "..." }` JSON (or can be extracted as such). If yes, it writes the cleaned draft string to `draftSessions.finalDraft` via an internal mutation — this is the signal that triggers the `DraftReadyCard` in the UI (T28). The Sonnet prompt (from T6) includes instructions to format the response as `{ "draft": "..." }` JSON when the drafter's message is ready to send; readiness is detected in the same generation call, not via a separate classifier.

**Privacy boundary** (load-bearing): `generateResponse` builds Draft Coach context from: (a) the drafting user's own `jointMessages`, (b) their own `synthesisText` from `partyStates`. It must NOT query the other party's `synthesisText` or `privateMessages`. Use an indexed query filtering by `userId` for `partyStates`; never `.collect()` all party states for a case and pick.

`draftCoach/sendFinalDraft` is a mutation that reads `draftSessions.finalDraft`, calls `ctx.runMutation(internal.jointChat.sendUserMessage, { caseId, content: finalDraft })`, then sets `draftSessions.status = 'SENT'`. This is the only path through which a Draft Coach conversation enters the shared space. Throw `ConvexError(CONFLICT)` if `finalDraft` is null.

`draftCoach/discardSession` simply sets `status = 'DISCARDED'`. Throw `ConvexError(CONFLICT)` if session is already in a terminal state.

**Token accounting:** After each Sonnet call in `generateResponse`, compute approximate token usage and call `ctx.runMutation(internal.draftCoach.recordDraftTokenUsage, { caseId, tokenDelta })`. Before making the Sonnet call, check `cases.tokenCount` against `HARD_CAP` from `convex/lib/pricing.ts` (T4). If `tokenCount >= HARD_CAP`, set `case.status = 'COST_LIMITED'` and return without calling the API. T17 adds the full degradation behavior; the hard gate must be present here.

`CLAUDE_MOCK=true` applies the same guard as T12 and T14: `if (process.env.CLAUDE_MOCK === 'true') { /* return canned stub via ctx.runMutation — same write path as production */ }`.

- `ctx.db.get(sessionId)` — load draft session (inside `getDraftCoachContext` internalQuery and mutations)

- `ctx.db.query("partyStates").withIndex("by_case_user", q => q.eq("caseId", ...).eq("userId", ...)).first()` — caller's own party state only (inside `getDraftCoachContext`)

## Stack Practices

- **`ctx.db` is forbidden inside `generateResponse` (internalAction) handlers.** All database reads must go through `ctx.runQuery(internal.draftCoach.getDraftCoachContext, { sessionId })` and all writes through `ctx.runMutation(internal.draftCoach.writeDraftMessage, ...)` or `ctx.runMutation(internal.draftCoach.recordDraftTokenUsage, ...)`. Violating this causes a Convex runtime error.
- **Streaming row lifecycle:** Insert `STREAMING` `draftMessages` row via `ctx.runMutation(internal.draftCoach.writeDraftMessage, { sessionId, status: 'STREAMING', content: '' })` before the stream loop; patch content on each chunk (pass `messageId`); set `status: 'COMPLETE'` on finish. On any unhandled error, the action must still attempt to flip the row to `COMPLETE` so the UI is never stuck on a spinner.
- `compressIfNeeded` requires the Anthropic client as its third argument — create `const anthropic = new Anthropic()` at the top of `generateResponse` and pass it through to `compressIfNeeded(messages, budgetTokens, anthropic)`.
- `draftCoach/sendFinalDraft` calling `jointChat/sendUserMessage` via `ctx.runMutation(internal.jointChat.sendUserMessage, ...)`.
- Privacy enforcement: `getDraftCoachContext` must NOT read synthesisText for the other party — use `withIndex("by_case_user", q => q.eq("caseId", ...).eq("userId", callerUserId))`, not `.collect()`.
- Anti-pattern: do NOT use `ctx.db.query().collect()` to build context — use index with limit.
- CLAUDE_MOCK guard: same `process.env.CLAUDE_MOCK === 'true'` check as T12 and T14; mock path writes via `ctx.runMutation` (same write path as production — not a shortcut).

## Design Practices

- **Single-stage Sonnet for Draft Coach vs two-stage Haiku+Sonnet for joint coach:** Joint chat receives many user messages and most are `NORMAL_EXCHANGE` — the Haiku gate is the primary cost control. Draft Coach is different: the drafter actively chose to engage, message volume per session is low, and each message warrants a thoughtful response. Adding a Haiku classifier would add latency without meaningful cost savings; Sonnet is called directly.
- **One-shot readiness detection (JSON format in the same call):** Rather than running a separate readiness classifier after every draft coach response, the Sonnet prompt (from T6) instructs the model to respond with coaching advice normally and to switch to `{ "draft": "..." }` JSON when it believes the message is ready to send. Detecting readiness in the same call saves a round-trip and keeps the draft refinement loop at single-Sonnet cost per turn.
- **Privacy enforced at input, not via output filter:** Unlike joint coach responses (which run through `filterResponse` because they are visible to both parties), Draft Coach output is private to the drafter — it never enters the shared space except through `sendFinalDraft`. Privacy is enforced by restricting what goes into the context (`getDraftCoachContext` reads only the caller's own partyState), not by post-generation filtering. The output filter would be redundant and would add unnecessary latency.
