# Authentication — Convex Auth with magic link and Google OAuth

This task wires up the authentication layer that gates every subsequent feature in Conflict Coach. Because the entire privacy and authorization model rests on knowing who the caller is — `ctx.auth.getUserIdentity()` is called at the top of every Convex function — authentication must be solid and correct before any other backend work can proceed. The goal is a working Convex Auth configuration with two providers (email magic link via Resend, Google OAuth) and an automatic `users` table upsert on first login.

## Acceptance Criteria

- User registers and logs in via magic link
- User logs in via Google OAuth
- First login creates a users row with role='USER'
- Subsequent logins upsert the users row
- Session persists across browser reload
- Session expires after 30 days of inactivity
- ctx.auth.getUserIdentity() returns the identity in any Convex function after login

## Implementation Notes

Convex Auth is the chosen provider (PRD Q1 resolved in its favor). The Clerk swap path is documented in README but not implemented. Auth configuration lives in `convex/auth.config.ts`; magic link uses the Resend provider. The Convex client in `main.tsx` must be wrapped in `<ConvexAuthProvider>` around (or wrapping) `<ConvexProvider>` — the exact nesting order is prescribed by `@convex-dev/auth`.

The `users` table (defined in T2) has fields: `email`, `displayName` (optional), `role` (union `"USER" | "ADMIN"`), and `createdAt`. The upsert on first login must key by `email` obtained from `ctx.auth.getUserIdentity()`. Promotion to `ADMIN` is a manual Convex Dashboard action in v1; no self-serve admin claim UI is built.

The tech spec (§4.2) specifies that `ctx.auth.getUserIdentity()` returns the authenticated identity or `null` — and that every query, mutation, and action begins by checking for this identity and then looking up the `users` row. The authorization helper module for that pattern is built in T4 (which depends on T3). T4's `requireAuth` helper queries the `users` table directly (inline, not via `upsertUserOnLogin`) and throws a `NOT_FOUND` `ConvexError` if no row exists — so the upsert in this task must run in the Convex Auth callback **before** any subsequent function can call `requireAuth`. This task's sole responsibility is: make login work and ensure a `users` row exists after login.

`upsertUserOnLogin` is called exclusively from the Convex Auth user-creation/update callback (e.g., the `createOrUpdateUser` option in `@convex-dev/auth`'s provider config). It is **not** called by T4's `requireAuth`.

Environment variables required (also in `.env.example` from T1):
- `GOOGLE_OAUTH_CLIENT_ID` and `GOOGLE_OAUTH_CLIENT_SECRET` — stored as Convex environment variables in the Convex dashboard.
- `AUTH_RESEND_KEY` — the Resend API key, stored as a Convex environment variable. This is the exact env var name expected by `@convex-dev/auth`'s Resend provider.

Session expiry at 30 days of inactivity should be configured in the Convex Auth settings via the `SESSION_MAX_DURATION_MS` option (or equivalent; verify in `@convex-dev/auth` docs for the exact key). 30 days = `30 * 24 * 60 * 60 * 1000` ms.

- `convex/auth.config.ts` — default export: Convex Auth config object (not directly called by app code; consumed by Convex runtime)
- `convex/auth.ts` — if a separate upsert helper is extracted: `export async function upsertUserOnLogin(ctx, identity): Promise<Id<"users">>` (internal; called from the Convex Auth `createOrUpdateUser` callback only — **not** from T4's `requireAuth`)
- `src/main.tsx` — no new exports; provider wiring is a side-effect change

**Signatures**
```ts
// convex/auth.ts (if extracted)
import type { MutationCtx } from "./_generated/server";
import type { UserIdentity } from "convex/server";
import type { Id } from "./_generated/dataModel";

export async function upsertUserOnLogin(
  ctx: MutationCtx,
  identity: UserIdentity

```

**Queries used**
- `ctx.db.query("users").withIndex("by_email", q => q.eq("email", identity.email)).unique()` — check for existing user
- `ctx.db.insert("users", { email, role: "USER", createdAt: Date.now() })` — create on first login
- `ctx.db.patch(existingUser._id, {})` — upsert path (no-op patch; optionally update `displayName` if provided by the identity)

**Invariants**
- `role` defaults to `"USER"` on insert; never set to `"ADMIN"` programmatically during login
- `email` is the stable key — if a user logs in via Google and then magic link with the same email, they get the same `users` row
- A `users` row must exist before any subsequent function calls; T4's `requireAuth` throws `NOT_FOUND` (not `USER_NOT_FOUND`) if no row is found — so the upsert callback in this task is the sole mechanism that guarantees row existence
- `ctx.auth.getUserIdentity()` must return non-null inside any Convex function called by a logged-in user after this task is complete

**Non-goals**
- Admin role assignment UI (manual Convex Dashboard action in v1)
- Clerk integration (swap path documented in README but not implemented)
- Email notification on login (deferred to v1.1)
- Password-based registration (not in v1)
- The authorization helpers (`requireAuth`, `assertPartyToCase`, `assertAdmin`) — those are T4

## Stack Practices

- Install: `npm install @convex-dev/auth @auth/core`.
- Auth config file is `convex/auth.config.ts`.
- Magic link provider: use `Resend` email provider from `@convex-dev/auth/providers/Resend`.
- Google OAuth: requires `GOOGLE_OAUTH_CLIENT_ID` and `GOOGLE_OAUTH_CLIENT_SECRET` in Convex environment variables. `AUTH_RESEND_KEY` is the required env var name for the Resend API key.
- Client side: import `useAuthActions` from `@convex-dev/auth/react`.
- Anti-pattern: do NOT store auth tokens in localStorage manually.
- `ConvexAuthProvider` must wrap `ConvexProvider` in `main.tsx`.
