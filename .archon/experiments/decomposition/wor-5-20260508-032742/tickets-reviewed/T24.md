# Case Detail Router and Phase-Appropriate View Scaffold

`/cases/:caseId` serves as the central routing hub of the Conflict Coach application, directing users to the correct phase-specific sub-route based on the live case status. This task creates the layout route — `CaseDetail.tsx` — that wraps all phase views with a persistent `PhaseHeader`, handles the routing logic that maps each case status to its appropriate sub-path, and provides a shared loading skeleton and friendly error boundary. It is the gateway every feature view depends on; getting the routing model right here prevents status-skipping bugs and ensures consistent layout across all phases.

## Acceptance Criteria

- GET /cases/:caseId redirects to correct sub-route by case.status
- Phase-skip redirects with toasts
- PhaseHeader on all sub-routes
- Loading: skeleton while cases/get in flight
- caseId not found / not party: friendly error page

## Implementation Notes

`CaseDetail.tsx` is a React Router v6 **layout route** that renders `<Outlet />` for child routes. The component runs `useQuery(api.cases.get, { caseId })` once and shares the result downstream via `useOutletContext` — child routes must NOT re-fetch this query independently.

**Status → sub-route mapping** (drive this in a `useEffect` + `navigate(..., { replace: true })` pattern):
- `DRAFT_PRIVATE_COACHING` → `../private`
- `READY_FOR_JOINT` → `../ready`
- `JOINT_ACTIVE` → `../joint`
- `COST_LIMITED` → `../joint` (COST_LIMITED is still joint-chat visible; treated identically to JOINT_ACTIVE for routing purposes)
- `CLOSED_RESOLVED`, `CLOSED_UNRESOLVED`, `CLOSED_ABANDONED` → `../closed` (all three closure statuses share one sub-route)

Phase-skip toasts fire when a user deep-links to a sub-route that doesn't match the live status. Compare the URL segment (the last path component) against the expected segment for the live status. Two toast messages cover all cases:
- Past-phase skip (user is at a sub-route the case has already moved beyond): `"That phase has already passed."`
- Future-phase skip (user is at a sub-route the case has not yet reached): `"That phase hasn't started yet."`

Both toasts redirect to the correct sub-route via `navigate(..., { replace: true })`.

**Loading states**: `useQuery` returns `undefined` while loading and `null` when the document doesn't exist (Convex convention). Treat `undefined` → render `CaseDetailSkeleton`; treat `null` (or a thrown `NOT_FOUND` / `FORBIDDEN` ConvexError) → render the friendly error page.

**Context sharing**: Use `useOutletContext<CaseDetailContext>()` typed hook in child routes. Define the context type in a shared `src/routes/cases/types.ts` file so all sub-routes import from one place.

**PhaseHeader**: rendered directly inside `CaseDetail.tsx` above `<Outlet />`, making it available unconditionally on all case sub-routes. It receives `caseData` as a prop.

The `CaseDetailSkeleton` renders two variants:
- **Chat-phase skeleton** (full-height column): used when URL hint or prior nav state suggests a chat phase (`/private`, `/joint`).
- **Default skeleton** (chat-phase layout): used when no URL hint is available (e.g., bare `/cases/:id`). Defaults to the chat-phase layout as the most common phase.

**Focus management**: Route changes must move focus to the `h1` of the new sub-route (implemented via `useLocation` subscription in `App.tsx` + `h1 tabIndex={-1}` in each sub-route). T24 adds the `tabIndex={-1}` to `CaseDetail`'s own heading only; the `App.tsx` `useLocation` subscription is added in T33 (accessibility pass).

**Router registration**: Register `CaseDetail` as a layout route in `src/App.tsx`:
```tsx
<Route path="/cases/:caseId" element={<CaseDetail />}>
  <Route path="private" element={<PrivateCoachingView />} />
  <Route path="ready" element={<ReadyForJointView />} />
  <Route path="joint" element={<JointChatView />} />
  <Route path="closed" element={<ClosedView />} />
</Route>
```

## Stack Practices

- React Router v6 nested routes: CaseDetail.tsx is a layout route that renders `<Outlet />`.
- Phase routing: `useEffect` + `navigate` with `replace: true` after `cases/get` resolves.
- `useQuery` loading: `undefined` = loading → `CaseDetailSkeleton`; `null` = not found → error page.
- Anti-pattern: do NOT re-fetch `cases/get` in each sub-route — use `useOutletContext`.
- `CaseWithParties` is the return type of `api.cases.get`; always import from `src/routes/cases/types.ts`, never redefine inline.
- RTL tests use `createMemoryRouter` + `RouterProvider`; do NOT use `MemoryRouter` alone (it does not support nested route matching needed for Outlet tests).

## Design Practices

- CaseDetail layout: `PhaseHeader` rendered above `<Outlet />` in the layout — all sub-routes inherit it. Full-height layout with `PhaseHeader` fixed or sticky at top using `bg-surface`, `border-bottom --border-default`.
- Loading skeleton: phase-appropriate placeholder (full-height column for chat views, centered card for ready-for-joint). Use shadcn/ui `Skeleton` with 300ms delay. `--bg-surface-subtle` shimmer.
- Error page (not found / not party): centered on `--bg-canvas`. Friendly `--text-secondary` message, no technical jargon. Single "Back to Dashboard" secondary link. No alarming colors (no `text-destructive`, no red backgrounds).
- Route transition: crossfade 100ms between sub-routes (design doc §2.6). No slide animations.
- Accessibility: route change must move focus to the `h1` of the new sub-route (`useLocation` subscription in `App.tsx` + `h1 tabIndex={-1}` — `App.tsx` subscription is T33 scope). `PhaseHeader` `<nav>` landmark wraps the back link and phase title. Toast announcements use shadcn/ui toast's `aria-live`.
