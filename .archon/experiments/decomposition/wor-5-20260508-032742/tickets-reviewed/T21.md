# Invite Acceptance Page (`/invite/:token`)

The invite acceptance page is Jordan's (the invitee's) first experience with Conflict Coach. She arrives via a link Alex sent, carrying a one-time token in the URL. The page must handle four distinct states gracefully — not-yet-authenticated, ready to accept, already-consumed token, and nonexistent token — because the token is the sole entry point for the second party. Getting this page right is critical to the case lifecycle: a confusing or broken entry flow means no joint session ever happens. The page surfaces just enough about the case (Alex's main topic, the category) to let Jordan decide whether to engage, while holding all private content behind auth and the redeem gate.

## Acceptance Criteria

- Logged-out: explanation card + 'Sign in to continue'; token preserved through auth
- Logged-in unredeemed: shows initiator name, mainTopic (labeled), category; no private content
- Accept calls invites/redeem; routes to /cases/:id/private
- Decline → CLOSED_ABANDONED → redirect with toast
- CONSUMED token: 'This invite has already been used.'
- Non-existent token: NOT_FOUND error message

## Implementation Notes

The route is `/invite/:token`. On mount, run `useQuery(api.invites.preview, { token })` — this is an unauthenticated query that returns only public fields (initiator display name, mainTopic, category). Do NOT call `invites/redeem` on page load; redeem is triggered only by explicit user action.

**Token preservation through auth:** When the user is logged out, save `token` to `sessionStorage` before navigating to `/login`. Use the named constant `PENDING_INVITE_TOKEN_KEY = 'pendingInviteToken'` (defined as a module-level export so T20/App.tsx can import the same string without duplication). After successful auth, the login page (or App.tsx redirect logic) checks `sessionStorage` for a pending invite token and routes back to `/invite/:token`. Clear it from `sessionStorage` once consumed. T21 is only responsible for writing; the read+restore path is T20/App.tsx.

**Auth detection:** Use `useConvexAuth()` from `@convex-dev/auth/react` which returns `{ isAuthenticated, isLoading }`. When `isLoading === true`, render a neutral skeleton — do NOT show the logged-out explanation card, as this would flash-redirect the user before auth state is known. Only branch on `isAuthenticated` after `isLoading === false`.

**State machine context:** `invites/redeem` is a Convex mutation that atomically binds the invitee's userId to the case's `inviteeId` field and marks the invite token as `CONSUMED`. On success it returns the caseId. A CONSUMED token means `redeem` will throw `ConvexError({ code: 'TOKEN_INVALID' })`. The `invites/preview` query does NOT check token validity — it will return data even for a consumed token's case. Token validity is enforced only at redeem time; the preview query exists solely for the display-before-auth UX.

**Preview query loading:** `useQuery(api.invites.preview, { token })` returns `undefined` while the query is in flight, `null` when the token's case cannot be found (NOT_FOUND), and the preview object when valid. Show a skeleton while `undefined`. Show the NOT_FOUND state when `null`.

**Decline path:** Per planning assumption, declining sets the case status to `CLOSED_ABANDONED`. This is implemented via `api.invites.decline` mutation (see Queries below). After decline succeeds, show a shadcn/ui toast ("You've declined the invitation") and redirect to `/` or `/dashboard` depending on whether the user is logged in.

**Error handling:** Wrap redeem and decline mutations in try/catch. Map `ConvexError` codes to inline messages: `TOKEN_INVALID` → "This invite has already been used." `NOT_FOUND` → display the not-found state. Do not show raw error objects. Both Accept and Decline buttons must be disabled while their respective mutation is in-flight (use a local `isPending` boolean from the mutation's return).

**Routing after accept:** On successful redeem, navigate to `/cases/${caseId}/private` using React Router's `useNavigate`.

### Queries used
- `api.invites.preview` — query; args: `{ token: string }`; returns `{ initiatorName: string; mainTopic: string; category: string } | null`
- `api.invites.redeem` — mutation; args: `{ token: string }`; returns `{ caseId: Id<'cases'> }`; throws `ConvexError({ code: 'TOKEN_INVALID' })` on consumed token, `ConvexError({ code: 'NOT_FOUND' })` on nonexistent
- `api.invites.decline` — mutation; args: `{ token: string }`; returns `void`; sets case status to `CLOSED_ABANDONED` via the state machine; throws `ConvexError({ code: 'NOT_FOUND' })` if token invalid

### Invariants
- `invites/redeem` is NEVER called on page load; only on explicit "Accept" button click.
- `invites/decline` is NEVER called on page load; only on explicit "Decline" button click.
- When `useConvexAuth().isLoading === true`, the component renders a neutral skeleton — neither the logged-out card nor the logged-in card is shown.
- When logged out (`isLoading === false && isAuthenticated === false`), token is written to `sessionStorage` under `PENDING_INVITE_TOKEN_KEY` before navigation to `/login`.
- The page never displays private coaching content (synthesis, private messages) — only `mainTopic`, `category`, `initiatorName`.
- Both Accept and Decline buttons are disabled (and show a loading indicator) while their mutation is in-flight (`isPending === true`).
- After successful redeem, the component navigates immediately to `/cases/${caseId}/private` — it does not re-render the acceptance card.
- Decline always results in a redirect away from `/invite/:token` (to `/` or `/dashboard`) after the mutation settles.
- `h1` elements on all states have `tabIndex={-1}` so App.tsx route-change focus logic can call `.focus()` on them.

### Non-goals
- This ticket does NOT implement the restore-from-`sessionStorage` logic inside the login page — that is an integration concern for T20 or a follow-up; T21 only writes to `sessionStorage`.
- Does NOT implement any private coaching content display.
- Does NOT handle multi-party (3+) invite flows.
- Does NOT send email notifications on accept/decline (P2, out of scope).

## Stack Practices

- Route param: `const { token } = useParams<{ token: string }>()`.
- Auth guard: `const { isAuthenticated, isLoading } = useConvexAuth()`. Render skeleton when `isLoading === true`; only branch on `isAuthenticated` after loading is false.
- Token persistence through auth: store in `sessionStorage` under `PENDING_INVITE_TOKEN_KEY` before redirecting to /login.
- Preview query: `const preview = useQuery(api.invites.preview, { token: token! })`.
- Preview state: `undefined` → skeleton; `null` → not-found state; object → logged-in unredeemed state.
- Decline mutation: `useMutation(api.invites.decline)`. Pass `{ token }` on Decline click.
- Toast: use shadcn/ui toast. Anti-pattern: do NOT call invites/redeem on page load.
- In-flight state: track `isPending` via local `useState`; set `true` before mutation call, `false` in finally block. Disable both Accept and Decline during any in-flight call.

## Design Practices

- Page layout (design doc §4.3): centered card on --bg-canvas, max-width ~480px, bg-surface shadow-1. All states use this same card container.
- Logged-out state: heading 'Alex has invited you to work through something together.' (500 weight, --text-primary). Body text in --text-secondary 15px per the design doc's exact copy. Single primary button 'Sign in to continue' (--accent). No decorative imagery.
- Logged-in unredeemed state: show initiator's mainTopic labeled 'Alex wrote this in the shared summary.' Helper text: 'You'll have your own private space to share your perspective.' in --text-secondary 14px. Category pill: --bg-surface-subtle, --text-secondary, 12px. Lock icon (Lucide) adjacent to the privacy callout — privacy must be visible, not implied (design doc §1.3).
- Accept button: primary --accent, full width within the card. Decline: ghost or text-only link in --text-secondary to visually subordinate it — 'One primary action per screen' (design doc §1.4).
- CONSUMED / NOT_FOUND states: --text-secondary body text, no alarming colors. Offer 'Go to dashboard' secondary link on BOTH states.
- Privacy label styling: use --private-tint (#F0E9E0) background on the section that shows the main topic, reinforcing that this is the only non-private data shared. ShieldCheck Lucide icon adjacent.
- Accessibility: single <main>. Heading hierarchy h1 → h2. All h1 elements across all states have `tabIndex={-1}` (per T33 route-change focus pattern: App.tsx calls `document.querySelector('h1')?.focus()` on navigation). Accept/Decline buttons have clear accessible text. Error states use role='alert'. Toast (decline confirmation) announced by screen reader via shadcn/ui toast's built-in aria-live.
