# Invite Flow Backend (redeem mutation, preview query, token validation)

The invite flow is the gate through which a second party enters a Conflict Coach case. Without it, every case is one-sided. This ticket implements the two Convex functions that power the `/invite/:token` route: `invites/preview` (a public, unauthenticated query that shows safe non-private case metadata) and `invites/redeem` (the authenticated mutation that atomically binds the invitee to the case and marks the token consumed). Token lifecycle correctness ‚Äî specifically the guarantee that a single token can only ever be redeemed once ‚Äî is a hard safety requirement, not a nice-to-have.

## Acceptance Criteria

- `invites/redeem`: validates token, atomically binds invitee, marks token CONSUMED
- `invites/redeem` on CONSUMED or REVOKED token: throws TOKEN_INVALID
- `invites/redeem` on missing token: throws TOKEN_INVALID
- `invites/redeem` by initiator (caller === `cases.initiatorUserId`): throws FORBIDDEN
- `invites/redeem` when invitee slot already bound: throws TOKEN_INVALID
- `invites/preview`: returns non-private fields only (status, initiatorDisplayName, mainTopic, category)
- `invites/preview` on CONSUMED token: returns `{ status: "CONSUMED" }`
- `invites/preview` on missing token: returns `null`
- `invites/preview` requires no authentication

## Implementation Notes

The `inviteTokens` table (defined in T2) stores each token as a 32-character URL-safe string with status `ACTIVE | CONSUMED | REVOKED`, indexed by `by_token`. The redemption mutation must use a single transaction to:

1. Look up the token via `ctx.db.query("inviteTokens").withIndex("by_token", q => q.eq("token", token)).unique()`.
2. If not found or `status !== "ACTIVE"`, throw `ConvexError({ code: "TOKEN_INVALID", ... })`.
3. Prevent the caller from redeeming their own invite (i.e., if the caller is the `cases.initiatorUserId`), throw `FORBIDDEN`.
4. Patch `cases.inviteeUserId` to the caller's user id and `cases.status` from `DRAFT_PRIVATE_COACHING` to `BOTH_PRIVATE_COACHING` (only if it was still `DRAFT_PRIVATE_COACHING`; if the invitee slot is already bound, throw `TOKEN_INVALID`).
5. Insert a `partyStates` row for the invitee (role `INVITEE`) with form fields left empty: `{ caseId, userId: callerId, role: "INVITEE", mainTopic: undefined, description: undefined, desiredOutcome: undefined, privateCoachingCompletedAt: undefined, synthesisText: undefined, closureProposalSummary: undefined }`.
6. Patch the `inviteTokens` row: `status: "CONSUMED"`, `consumedAt: Date.now()`, `consumedByUserId: callerId`.

All six steps happen in one Convex mutation transaction ‚Äî if any step throws, none of the writes persist.

`invites/preview` is a Convex **query** (not a mutation) with **no auth requirement**. It exists solely so the `/invite/:token` page can render case metadata before the user logs in. It must return only non-private fields: `initiatorDisplayName` (from `users.displayName`), `mainTopic` (from the initiator's `partyStates` row ‚Äî this is the shared field written by the initiator in the case creation form), and `category` (from `cases`). It must **never** return: description, desiredOutcome, private coaching messages, synthesis, or any invitee data. If the token is not found, return `null`. If the token is `CONSUMED`, return `{ status: "CONSUMED" }` so the client can show the right error state. If the token is `ACTIVE`, return `{ status: "ACTIVE", initiatorDisplayName, mainTopic, category }`.

Authorization for `invites/redeem` uses `requireAuth(ctx)` from `convex/lib/auth.ts` (T4). The token itself is the credential ‚Äî no case-level party check is needed (the caller is not yet a party).

Token generation (in `cases/create`, T9) uses `crypto.randomUUID().replace(/-/g, '').slice(0, 32)`. Tokens are stored in plaintext in v1 (acceptable because consumption is instant and the token has no post-use value ‚Äî noted in tech spec ¬ß11 checklist).

**Return type note:** `redeem` is declared as `Promise<{ caseId: string }>`. Convex IDs are `Id<'cases'>` at runtime; call `.toString()` or type the return as `Id<'cases'>` ‚Äî be consistent with how T9's `cases/create` returns `caseId` so T21 (the invite page) can use it without a cast.

```ts
import { v } from "convex/values";

export type InvitePreview =
  | { status: "CONSUMED" }
  | {
      status: "ACTIVE";
      initiatorDisplayName: string;
      mainTopic: string | undefined;
      category: string;
    };

export const preview = query({
  args: { token: v.string() },
  handler: async (ctx, { token }): Promise<InvitePreview | null> => { ... },
});

export const redeem = mutation({
  args: { token: v.string() },
  handler: async (ctx, { token }): Promise<{ caseId: string }> => { ... },
});
```

### Queries used
- `ctx.db.query("inviteTokens").withIndex("by_token", q => q.eq("token", token)).unique()`
- `ctx.db.get(tokenRow.caseId)` ‚Äî to fetch the case
- `ctx.db.query("partyStates").withIndex("by_case_and_user", q => q.eq("caseId", caseId).eq("userId", initiatorUserId)).unique()` ‚Äî to fetch initiator's `mainTopic` for preview
- `ctx.db.get(caseDoc.initiatorUserId)` ‚Äî to fetch `users` row for `displayName`
- `ctx.db.patch(caseId, { inviteeUserId: callerId, status: "BOTH_PRIVATE_COACHING", updatedAt: Date.now() })` ‚Äî in redeem
- `ctx.db.patch(tokenId, { status: "CONSUMED", consumedAt: Date.now(), consumedByUserId: callerId })` ‚Äî in redeem
- `ctx.db.insert("partyStates", { caseId, userId: callerId, role: "INVITEE", mainTopic: undefined, description: undefined, desiredOutcome: undefined, privateCoachingCompletedAt: undefined, synthesisText: undefined, closureProposalSummary: undefined })` ‚Äî in redeem

### Invariants
- A token in status `CONSUMED` or `REVOKED` can never be redeemed; `redeem` throws `TOKEN_INVALID`.
- A missing token causes `redeem` to throw `TOKEN_INVALID`.
- `redeem` is idempotent only in the sense that a second call will throw `TOKEN_INVALID` ‚Äî it does NOT silently succeed.
- `preview` never exposes the initiator's `description`, `desiredOutcome`, or any private coaching content.
- `preview` never exposes any invitee data.
- The initiator cannot redeem their own invite (`cases.initiatorUserId === caller.userId` ‚Üí `FORBIDDEN`).
- If `cases.inviteeUserId` is already set (invitee slot bound), `redeem` throws `TOKEN_INVALID`.
- All six writes in `redeem` succeed atomically or none are committed.
- `preview` requires no `ctx.auth` ‚Äî calling it with `ctx.auth = null` must not throw.

### Non-goals
- Token revocation UI (admin-only, not built in v1).
- Email delivery of invite links (that's T9's responsibility ‚Äî this ticket only validates and redeems).
- Auth or session creation ‚Äî `redeem` assumes the caller is already authenticated.
- Updating `cases.status` for any transition other than `DRAFT_PRIVATE_COACHING ‚Üí BOTH_PRIVATE_COACHING`.

## Stack Practices

- Atomicity: Convex mutations are fully transactional.
- `invites/preview` is a Convex query ‚Äî no auth required for preview.
- Anti-pattern: do NOT check token validity in `invites/preview` ‚Äî return `null` for missing, `{ status: "CONSUMED" }` for consumed; do not throw.

## Design Practices

- **Privacy boundary is unconditional in `preview`:** the handler must never read `description`, `desiredOutcome`, `synthesisText`, or any field from the invitee's party state row. The simplest safe implementation is to only read the explicit allow-list: `initiatorDisplayName`, `mainTopic`, `category`. Adding any new field requires explicit review.
- **Atomicity is the correctness guarantee:** the six writes in `redeem` are all in one Convex mutation. Do not split into multiple mutations or actions ‚Äî the whole point is that partial state is impossible. This is what makes "a token can only be redeemed once" a true invariant and not a race condition.
- **YAGNI ‚Äî no token revocation in v1:** the `REVOKED` status is in the schema (T2) for future admin tooling, but no `revoke` mutation is built here. Guard the non-`ACTIVE` path with a single `status !== "ACTIVE"` check, which naturally covers both `CONSUMED` and `REVOKED`.
