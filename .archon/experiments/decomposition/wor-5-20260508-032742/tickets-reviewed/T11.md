# Templates backend and seed data

Templates are the mechanism by which admins tune AI behavior per conflict category without shipping code. This task implements the Convex query and mutation functions for template management — including listing non-archived templates for case creation, admin-only CRUD (create, publish new version, archive), and a seed script that bootstraps the system with an admin user and three starter templates. T9 (case CRUD) depends on this task because case creation must resolve a `templateVersionId` from the selected category.

## Acceptance Criteria

- `templates/list`: returns non-archived templates; optionally filtered by `category`
- `admin/templates/listAll`: returns all templates including archived; throws FORBIDDEN for non-ADMIN
- `admin/templates/create`: creates `templates` + initial `templateVersions` row atomically; writes audit entry
- `admin/templates/publishNewVersion`: inserts new immutable version row, updates `currentVersionId`; writes audit entry
- `admin/templates/archive`: sets `archivedAt`; writes audit entry; throws FORBIDDEN for non-ADMIN
- `admin/templateVersions/list`: returns version history ordered by `version` asc; throws FORBIDDEN for non-ADMIN
- admin mutations throw FORBIDDEN for non-ADMIN users
- `convex/seed.ts`: creates admin user and exactly three templates (categories: `"workplace"`, `"family"`, `"personal"`); idempotent on re-run

## Implementation Notes

The data model for templates is a two-table design (tech spec §3.1):

- **`templates`** — one row per template identity. Carries `category`, `name`, `currentVersionId`, `archivedAt`, `createdAt`, `createdByUserId`.
- **`templateVersions`** — immutable snapshots. Each row carries `templateId`, a monotonically incrementing `version` number, `globalGuidance`, `coachInstructions`, `draftCoachInstructions`, `publishedAt`, `publishedByUserId`, and optional `notes`. Once inserted, a templateVersion row is **never mutated**. Editing a template publishes a new version row and updates `templates.currentVersionId`.

**Invariant (tech spec §3.2 #5):** `templateVersions` rows are immutable once created. Publishing a new version means inserting a new row and updating `templates.currentVersionId`. There is no update mutation for `templateVersions` — its absence IS the enforcement.

**Case pinning (tech spec §3.2 #1):** `cases.templateVersionId` is set at case creation and never changes, even if the template is edited or archived later. The query T9 uses to resolve a template must return the current non-archived version for the selected category at creation time; thereafter the case holds its own reference.

**`templates/list` behavior:** Return all templates where `archivedAt` is absent/undefined, optionally filtered by `category`. The result is used by the case-creation form (T23) to populate the template picker and by case-creation mutation (T9) for the implicit category → template resolution. This query requires authentication but no admin role. When `category` is provided, use `ctx.db.query("templates").withIndex("by_category", q => q.eq("category", category))` then `.filter(q => q.eq(q.field("archivedAt"), undefined))`. When `category` is omitted, use `.collect()` with a post-filter for `archivedAt`.

**`admin/templates/listAll`:** Returns all templates including archived ones. Admin only. Used by the admin template management UI (T32).

**`admin/templateVersions/list`:** Returns all `templateVersions` for a given `templateId` ordered by `version` asc. Admin only. Used for version history display in the admin edit UI (T32).

**`admin/templates/create`:** Creates a `templates` row and an initial `templateVersions` row (version = 1) in a single mutation. Updates `templates.currentVersionId` to point at the new version. Writes an `auditLog` entry with `action: "TEMPLATE_CREATED"`, `actorUserId`, `targetId: templateId`, `targetType: "template"`.

**`admin/templates/publishNewVersion`:** Inserts a new `templateVersions` row. The new version number = `ctx.db.query("templateVersions").withIndex("by_template", q => q.eq("templateId", templateId)).order("desc").first()?.version ?? 0) + 1`. Sets `templates.currentVersionId` to the new version id. Writes an `auditLog` entry with `action: "TEMPLATE_PUBLISHED"`, `actorUserId`, `targetId: templateId`, `targetType: "template"`.

**`admin/templates/archive`:** Sets `archivedAt` on the `templates` row using `ctx.db.patch(templateId, { archivedAt: Date.now() })`. Does NOT delete any data. Writes an `auditLog` entry with `action: "TEMPLATE_ARCHIVED"`, `actorUserId`, `targetId: templateId`, `targetType: "template"`. Pinned cases continue to resolve correctly — they hold `templateVersionId` directly and never look up `templates.currentVersionId` after creation.

**Authorization:** All `admin/*` mutations must call `assertAdmin(ctx)` from `convex/lib/auth.ts` (T4) at the top of the function. `templates/list` calls `requireAuth(ctx)` only.

**`convex/seed.ts`:** A standalone script (or Convex mutation callable from the CLI) that:
1. Upserts a user record with `role: "ADMIN"` for a known seed email (e.g., `admin@conflictcoach.app`). The seed should check for existence using `ctx.db.query("users").withIndex("by_email", q => q.eq("email", seedEmail)).first()` and skip if found.
2. Creates three templates covering categories `"workplace"`, `"family"`, `"personal"`. Each template should have meaningful `globalGuidance` covering the AI's general coaching role for that category, and minimal `coachInstructions` that establish tone and facilitation style.
3. Template idempotency: check for existing template by category using `ctx.db.query("templates").withIndex("by_category", q => q.eq("category", cat)).first()` — skip that category if a template already exists.
4. The seed is idempotent — calling it twice does not produce duplicate rows.

The seed function is NOT called automatically; it is invoked manually via the Convex dashboard or CLI during environment setup (per tech spec §11.3 cold-start sequence referenced in T1).

**File layout (tech spec §7.1–7.2 naming conventions):**
- `convex/templates.ts` — `templates/list` query
- `convex/admin/templates.ts` — `admin/templates/listAll`, `admin/templates/create`, `admin/templates/publishNewVersion`, `admin/templates/archive`
- `convex/admin/templateVersions.ts` — `admin/templateVersions/list`
- `convex/seed.ts` — seed mutation

### Queries used
- `ctx.db.query("templates").withIndex("by_category", q => q.eq("category", category)).filter(q => q.eq(q.field("archivedAt"), undefined)).collect()` — `templates/list` with category filter + archived exclusion
- `ctx.db.query("templates").collect()` followed by post-filter for `archivedAt === undefined` — `templates/list` without category
- `ctx.db.query("templates").collect()` — `admin/templates/listAll` (all, including archived)
- `ctx.db.query("templateVersions").withIndex("by_template", q => q.eq("templateId", templateId)).order("asc").collect()` — version history list
- `ctx.db.query("templateVersions").withIndex("by_template", q => q.eq("templateId", templateId)).order("desc").first()` — compute next version number in `publishNewVersion`
- `ctx.db.insert("templates", ...)`, `ctx.db.insert("templateVersions", ...)` — create and publishNewVersion
- `ctx.db.patch(templateId, { currentVersionId: newVersionId })` — update pointer after publishNewVersion
- `ctx.db.patch(templateId, { archivedAt: Date.now() })` — archive
- `ctx.db.query("users").withIndex("by_email", q => q.eq("email", seedEmail)).first()` — seed idempotency check for admin user
- `ctx.db.query("templates").withIndex("by_category", q => q.eq("category", cat)).first()` — seed idempotency check per template category

### Invariants
- `templateVersions` rows are NEVER mutated after insertion. No `patch` or `replace` calls on `templateVersions` rows.
- `templates.currentVersionId` always points to the most-recently published `templateVersions` row for that template.
- `admin/*` mutations throw `ConvexError({ code: "FORBIDDEN", ... })` when caller's `user.role !== "ADMIN"`.
- `templates/list` never returns templates where `archivedAt` is set.
- `publishNewVersion` increments `version` by exactly 1 relative to the current max version for that template, derived from an `order("desc").first()` query.
- `seed` is idempotent: re-running does not create duplicate templates (checked per-category) or duplicate admin user records (checked by email).
- `auditLog` receives an entry for every admin mutation: `create` → `action: "TEMPLATE_CREATED"`, `publishNewVersion` → `action: "TEMPLATE_PUBLISHED"`, `archive` → `action: "TEMPLATE_ARCHIVED"`. Each entry carries `actorUserId`, `targetId` (the templateId), and `targetType: "template"`.

## Stack Practices

- File structure: `convex/templates.ts`, `convex/admin/templates.ts`, `convex/admin/templateVersions.ts`, `convex/seed.ts`.
- Immutability of templateVersions: enforced by absence of an update mutation — do NOT add `ctx.db.patch` or `ctx.db.replace` calls for `templateVersions` rows.
- Anti-pattern: do NOT version templates by mutating the existing row.
- Version computation: always derive the next version number from `order("desc").first()` on `by_template` index, then +1. Never store or increment a counter in the `templates` row itself.
- Test isolation: use convex-test (or equivalent) to create an isolated in-memory Convex database per test; do NOT import `convex/_generated/server` directly in Vitest tests.

## Design Practices

- **Immutability as audit trail:** `templateVersions` rows are append-only by design, not just convention. The absence of an update mutation is the enforcement — this means every published version is permanently inspectable in the admin UI (T32), which is the point. Do not work around this by adding soft-update fields to `templateVersions`.
- **Admin accountability:** Every admin mutation writes an `auditLog` entry. This is not optional logging — it is the accountability trail for content changes that affect all active cases. Omitting it for any of the three mutations breaks T32's audit log UI.
- **Seed as cold-start bootstrap only:** The seed function is not a dev fixture pattern. It is the one-time cold-start action per tech spec §11.3. Keep it simple, idempotent, and self-contained. Do not share seed logic with test fixtures — tests should seed their own minimal data.
- **Non-admin `templates/list` does not expose archived templates:** Category routing in case creation (T9) must never resolve an archived template. The `archivedAt` exclusion in `templates/list` is the sole enforcement — there is no secondary check in T9. Do not relax this filter for any caller.
