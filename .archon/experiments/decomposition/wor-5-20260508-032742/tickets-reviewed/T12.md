# Private coaching backend (send message, AI response, mark complete, crisis detection, token tracking)

This task implements the server-side foundation for the private coaching phase — the first AI-interactive leg of the mediation process. Each party has a fully isolated chat channel with a Private Coach AI; nothing in this module is visible to the other party. The backend must handle the full message lifecycle from user submission through Claude streaming, accumulate token usage toward the per-case cost cap established in planning assumptions, flag crisis-keyword content so the client can surface a CrisisCard, and provide an idempotent mark-complete mutation that triggers synthesis scheduling when both parties are done.

## Acceptance Criteria

- `privateCoaching/myMessages`: returns only caller's messages
- `generateAIResponse`: streams into privateMessages row
- `generateAIResponse`: increments cases.tokenCount
- `privateCoaching/markComplete`: idempotent

## Implementation Notes

**Module location and shape.** The query and mutations live in `convex/privateCoaching.ts`; the action is declared as an `internalAction` in the same file (or a companion `convex/internal/privateCoaching.ts`). Mark it `internalAction` — it must never be callable directly from the client.

**Message flow.** `sendUserMessage` (mutation) inserts the user's message row with `status: "COMPLETE"` and immediately inserts a second `privateMessages` row with `role: "AI"`, `content: ""`, `status: "STREAMING"`. It then calls `ctx.scheduler.runAfter(0, internal.privateCoaching.generateAIResponse, { caseId, userId })`. This two-step insert ensures the reactive client sees the STREAMING placeholder before the action has a chance to write. The mutation returns the AI placeholder's `Id<"privateMessages">` (not the user message ID) so callers can track the streaming row if needed; the reactive `myMessages` query updates automatically regardless.

**Streaming flush pattern.** Inside `generateAIResponse`, the action opens a `stream: true` Claude call. It accumulates incoming delta tokens in a local string buffer and calls an internal `_flushStreamingChunk` mutation approximately every 50ms (using `Date.now()` deltas, not `setInterval` — Convex actions do not have a persistent event loop). On completion, a final `_finalizeMessage` mutation sets `status: "COMPLETE"`, writes the full accumulated `content`, and records the total `tokens` field on the message row. The 50ms cadence is a best-effort heuristic; a single flush per response is acceptable for the initial implementation.

**Private Coach system prompt.** No template is applied. The hardcoded system prompt from tech spec §6.3.1 is injected verbatim:
> "You are a calm, curious, non-judgmental listener helping a person articulate their perspective in an interpersonal conflict. Ask clarifying questions. Reflect what they say. Help them identify what they actually want, what they're feeling, and what the other person might be thinking. Do not take sides. Do not tell them they're right or wrong. Your only goal is to help them prepare to communicate with the other party clearly and calmly."
This comes from `convex/lib/prompts.ts` (`assemblePrompt({ role: "PRIVATE_COACH" })`), which is T6.

**Context assembly.** Before calling Claude, the action calls `requireAuth(ctx)` then `assertPartyToCase(ctx, user, caseId)`. It then reads: (1) the caller's `partyStates` row for `mainTopic`, `description`, `desiredOutcome`; (2) all prior `privateMessages` for `{ caseId, userId }` ordered by `createdAt`. It does NOT read any other party's data. Pass these to `compressIfNeeded` from `convex/lib/transcriptCompression.ts` (T7) before building the final messages array.

**Token counting and cost accumulation.** After the Claude call completes, the action calls `ctx.runMutation(internal.privateCoaching._finalizeMessage, { messageId, content, tokens })`. Inside `_finalizeMessage`, the mutation patches `cases.tokenCount` using `(existingCount ?? 0) + responseTokens`. The `usage` object from the Claude streaming response provides `input_tokens` + `output_tokens`; sum both. **Never overwrite or reset tokenCount to zero.** The cap enforcement logic (soft/hard) is out of scope for this ticket and is handled in T17 — this ticket only accumulates.

**Crisis keyword detection.** After inserting the user's message in `sendUserMessage`, check `content` against a small list of crisis keywords (configurable constant in the module, e.g. `["suicide", "self-harm", "hurt myself", "kill myself", "abuse", "emergency"]`). Because `privateMessages` schema does not include `isCrisis` in T2, the preferred approach for v1 is to detect on the client by inspecting the raw message content — the backend does not need a schema change. The backend's only responsibility is making the raw content available; the client (T25) reads it to conditionally render the `CrisisCard`. Document this decision in a code comment in the module.

**`markComplete` idempotency.** The mutation calls `requireAuth(ctx)` and `assertPartyToCase(ctx, user, caseId)`. It then reads both `partyStates` rows for the case via `ctx.db.query("partyStates").withIndex("by_case", q => q.eq("caseId", caseId)).collect()`. It sets the caller's `partyStates.privateCoachingCompletedAt = Date.now()` only if not already set. It then checks whether the other party's `partyStates.privateCoachingCompletedAt` is also set. If both are now complete, it calls `ctx.scheduler.runAfter(0, internal.synthesis.generate, { caseId })`. Running `markComplete` a second time must be a no-op: no duplicate scheduler call, no error, `privateCoachingCompletedAt` value unchanged.

**CLAUDE_MOCK guard.** At the top of `generateAIResponse`, check `process.env.CLAUDE_MOCK === "true"`. If true, skip the Claude API call entirely, call `ctx.runMutation(internal.privateCoaching._finalizeMessage, { messageId: aiPlaceholderId, content: "This is a mock coach response for testing.", tokens: 0 })`, wait 300ms to simulate latency, then return. This guard is critical for CI and E2E tests.

**Privacy invariant.** `myMessages` must query with the index `by_case_and_user` using `{ caseId, userId: user._id }`. It must never fall through to a query that returns messages by `caseId` alone. The `by_case` index on `privateMessages` is server-side only (context assembly); it is never exposed via any query reachable from the client.

### Invariants
- `myMessages` MUST filter by both `caseId` AND `userId` — never returns another user's messages. The `by_case_and_user` index (defined in T2 schema) enforces this at the query level.
- `sendUserMessage` inserts the STREAMING AI placeholder in the same mutation as the user message — it is never possible for the user message to exist without an accompanying AI placeholder. Both rows are written in one Convex mutation (atomic).
- `markComplete` calling `ctx.scheduler.runAfter` for synthesis is idempotent: the guard `if (!myState.privateCoachingCompletedAt)` prevents double-scheduling. A second call with `privateCoachingCompletedAt` already set is a no-op.
- `generateAIResponse` MUST be `internalAction` — no client callable path should exist.
- Token accumulation uses `(existing ?? 0) + delta`; never overwrites; never sets to zero.
- `_finalizeMessage` always sets `status: "COMPLETE"` unconditionally — the AI placeholder can only move STREAMING → COMPLETE, never backwards.

### Non-goals
- Cap enforcement (soft/hard budget limits) — handled in T17.
- The other party's message visibility — enforced by index choice, no separate filter needed.
- Crisis keyword DB schema changes — client-side detection only in v1.
- Synthesis generation logic — only the scheduler call (`ctx.scheduler.runAfter`) is in scope; T13 owns the action body.

## Stack Practices

- **Streaming pattern:** mutation inserts STREAMING row → action streams → flush via `ctx.runMutation(internal.privateCoaching._flushStreamingChunk, ...)` every ~50ms → finalize via `ctx.runMutation(internal.privateCoaching._finalizeMessage, ...)` on completion.
- **`generateAIResponse` must be `internalAction`.** Never expose it via `httpAction` or `action` — Convex will reject direct client calls, and the function body relies on `userId` being passed from a trusted mutation, not from client args.
- **`requireAuth → assertPartyToCase` call chain.** Every public-facing query and mutation must call `requireAuth(ctx)` first, then `assertPartyToCase(ctx, user, caseId)` before any database reads. Do not inline auth logic.
- **Token accumulation pattern:** Always use `(existing ?? 0) + delta` inside `_finalizeMessage`. Never read `tokenCount`, add locally, and write in the same action — read in the mutation where the patch is atomic.
- **CLAUDE_MOCK=true:** check `process.env.CLAUDE_MOCK === "true"` at the very top of `generateAIResponse` before any Anthropic client instantiation.
- **`by_case` index is server-only.** Only `generateAIResponse` (internalAction context assembly) and `markComplete` (partyStates dual-read) use server-only indexes. All public queries must use `by_case_and_user`.

## Design Practices

- **Why `internalAction` for `generateAIResponse`:** Convex actions that call external APIs (Anthropic) must not be callable by arbitrary clients. Making it `internalAction` ensures only trusted Convex scheduler calls (from `sendUserMessage`) can invoke it. This eliminates a class of prompt-injection and DoS vectors at the platform level.
- **Why two-row atomic insert in `sendUserMessage`:** The mutation writes both the user message and the AI STREAMING placeholder in one transaction. Clients using `useQuery(api.privateCoaching.myMessages)` receive a reactive update that already includes both rows — the user never sees a flash of "no AI response pending." This is preferable to optimistic UI on the client because Convex mutations are serialized and the scheduler call is enqueued atomically.
- **Why streaming flush via internal mutation (not bulk write at end):** The 50ms flush cadence gives the reactive client partial content, producing a live-typing visual effect without requiring WebSocket or SSE infrastructure. Convex's reactive query layer handles propagation. A single flush at the end would work but degrades UX for longer responses.
- **Why crisis detection is client-side (v1):** Adding `isCrisis` to the `privateMessages` schema would require a migration and adds a write to every `sendUserMessage` call. The crisis keyword list is short and deterministic; client-side pattern matching against raw `content` is equivalent in effect and avoids schema coupling. Document with `// v1: crisis detection is client-side (T25). No schema change needed.` comment in module.
- **Privacy invariant rationale:** Private coaching messages are the most sensitive data in the system. The `by_case_and_user` index is the only acceptable query path for public-facing reads. Any future developer adding a query should be aware that `by_case` alone would expose one party's messages to the other — this invariant should be enforced via code review and the unit test for index usage above.
