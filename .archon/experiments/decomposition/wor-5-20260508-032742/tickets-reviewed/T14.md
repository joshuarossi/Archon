# Joint chat entry mutation and Coach facilitation backend

T14 implements the backend machinery that transitions a case into live joint-session mode and keeps it productively facilitated by an AI Coach. It covers the `jointChat/enter` mutation (which advances the case from `READY_FOR_JOINT` to `JOINT_ACTIVE`), the reactive queries that serve joint messages and each party's private synthesis text, the `jointChat/sendUserMessage` mutation that persists user turns and schedules Coach evaluation, and the `jointChat/generateCoachResponse` internalAction that runs the two-stage Haiku-gate → Sonnet facilitation pipeline. This task sits at the heart of the product: without it, neither party can exchange messages in the joint session or receive Coach guidance.

## Acceptance Criteria

- `jointChat/enter`: READY_FOR_JOINT→JOINT_ACTIVE, idempotent
- `generateCoachResponse`: Haiku classifies; Sonnet only for non-NORMAL_EXCHANGE
- T5 filter on Coach Sonnet response; fallback on failure
- Increments tokenCount; enforces HARD_CAP

## Implementation Notes

**Domain file:** All functions for this ticket live in `convex/jointChat.ts`. Follow the one-file-per-domain convention established in T9 and T12.

**`jointChat/enter` mutation:** Reads the current `cases` row and asserts the caller is a party (via `assertPartyToCase` from `convex/lib/auth.ts`). If `case.status === 'JOINT_ACTIVE'` the mutation returns without writing — idempotency is required because both parties call this independently and the second caller must not throw. If `case.status !== 'READY_FOR_JOINT'` throw `ConvexError(CONFLICT)` from the codes defined in T4. On the happy path, call the `transitionToJointActive` pure function from `convex/lib/stateMachine.ts` (T8) to get the new status, then write it back to `cases`.

**Reactive queries:**
- `jointChat/messages({ caseId })`: Index on `jointMessages.by_caseId` (defined in T2 schema). Assert caller is party + case is not in a private-only state (i.e., status must not be `DRAFT_PRIVATE_COACHING`). Return the full `JointMessage[]` array.
- `jointChat/mySynthesis({ caseId })`: Read the caller's `partyStates` row and return `{ text: synthesisText }` if populated, or `null` if synthesis has not yet been written. Assert party membership.

**`jointChat/sendUserMessage` mutation:** Inserts a `jointMessages` row with `authorUserId`, `caseId`, `content`, `status: 'SENT'`, `role: 'USER'`. Then calls `ctx.scheduler.runAfter(0, internal.jointChat.generateCoachResponse, { caseId, triggerType: 'USER_MESSAGE' })`. Assert case is `JOINT_ACTIVE`; throw `CONFLICT` otherwise. Detect `@coach` mention (or similar) inside the mutation body — NOT on the client — per the anti-pattern note in stack practices.

**`generateCoachResponse` internalAction — two-stage pipeline:**

> **Important:** `internalAction` handlers cannot access `ctx.db` directly. All database reads must go through `ctx.runQuery(internal.jointChat.getCoachContext, { caseId })` and all writes through `ctx.runMutation(internal.jointChat.recordCoachTokenUsage, ...)` or `ctx.runMutation(internal.jointChat.writeCoachMessage, ...)`. Define the necessary `internalQuery` and `internalMutation` helpers in the same file.

*Stage 1 — Haiku gate (classification):*
Check `CLAUDE_MOCK` guard first (return a canned `NORMAL_EXCHANGE` result in mock mode). Fetch the most recent user joint message. Call `claude-haiku-4-5-20251001` with a concise single-turn classification prompt requesting one of: `INFLAMMATORY`, `PROGRESS`, `QUESTION_TO_COACH`, `NORMAL_EXCHANGE`. Use `max_tokens: 50` (non-streaming). If the result is `NORMAL_EXCHANGE` and `triggerType` is not `TIMER` and not `AT_MENTION`, return early — no Sonnet call. This gate is the primary cost-control mechanism.

*Stage 2 — Sonnet facilitation:*
Assemble the prompt via `assemblePrompt({ role: 'COACH', ... })` from `convex/lib/prompts.ts` (T6). Build the context window from joint chat history + BOTH parties' `synthesisText` values (never raw private messages). Run the assembled context through `compressIfNeeded(messages, 60_000, anthropic)` from `convex/lib/transcriptCompression.ts` (T7) — pass the Anthropic client instance created at the top of the action handler. Stay within the 60k-token budget. Call `claude-sonnet-4-5` with streaming. **Streaming row lifecycle:** before the streaming loop, insert a `jointMessages` row with `status: 'STREAMING'` and `content: ''` via `ctx.runMutation(internal.jointChat.writeCoachMessage, { caseId, status: 'STREAMING', content: '' })`; on each chunk patch the content; on finish set `status: 'COMPLETE'`.

*Privacy filter:* After generation completes, call `filterResponse(coachOutput, [...initiatorPrivateMessages, ...inviteePrivateMessages])` from `convex/lib/privacyFilter.ts` (T5). `FilterResult` is `{ passed: true } | { passed: false; matchedSubstring: string }`. If `result.passed === false`, retry the full Sonnet generation up to 2 more times (3 total attempts). On final failure after all 3 attempts, write `COACH_FALLBACK_TEXT` (the named constant — see Exports below) to `jointMessages` as a `COMPLETE` row and log a flag for admin review via the `auditLog` table. Never write a raw failing response.

*Token accounting:* After each Sonnet call, compute approximate token usage (input tokens + output tokens, approximated via word-count if exact counts are unavailable). Call `ctx.runMutation(internal.jointChat.recordCoachTokenUsage, { caseId, tokenDelta })`. Before making any Sonnet call, check `cases.tokenCount` against the `HARD_CAP` constant from `convex/lib/pricing.ts` (T4). If `tokenCount >= HARD_CAP`, set `case.status = 'COST_LIMITED'` via `ctx.runMutation` and return without calling the API — token enforcement in T17 will add the full degradation behavior, but the hard gate must be present here so no call is ever made past the cap.

**`CLAUDE_MOCK` guard:** Mirror the pattern from T12. At the top of `generateCoachResponse`, if `process.env.CLAUDE_MOCK === 'true'`, skip all API calls. For the Haiku gate return `QUESTION_TO_COACH` (so Sonnet stage runs in tests). For Sonnet, write a canned `COMPLETE` coach message to `jointMessages` via `ctx.runMutation`.

**@-mention detection:** Inside `jointChat/sendUserMessage`, check `content.toLowerCase().includes('@coach')`. If true, pass `triggerType: 'AT_MENTION'` to the scheduled action so the Haiku gate is bypassed.

**Invariants**
- `jointChat/enter` called twice by the same user must not throw and must not create duplicate `jointMessages` rows
- `generateCoachResponse` MUST NOT be called (or proceed to Sonnet) when `cases.tokenCount >= HARD_CAP`
- Coach output must pass the T5 privacy filter before being written as `COMPLETE`; on failure after 3 total attempts (initial + 2 retries) the `COACH_FALLBACK_TEXT` constant is written — never a raw failing response
- `sendUserMessage` must throw `CONFLICT` if `case.status !== 'JOINT_ACTIVE'`
- @-mention detection happens inside `sendUserMessage`, never on the client
- `generateCoachResponse` never calls `ctx.db` directly — all DB access goes through `ctx.runQuery`/`ctx.runMutation` wrappers
- A `STREAMING` coach message row must always transition to `COMPLETE` (or be overwritten by fallback) — it must never be left in the `STREAMING` state at action exit

## Stack Practices

- Haiku classification: single non-streaming call, max_tokens 50.
- Anti-pattern: do NOT hardcode @-mention as a client-side feature — detection happens inside the mutation body (`content.toLowerCase().includes('@coach')`).
- CLAUDE_MOCK guard: same `process.env.CLAUDE_MOCK === 'true'` check as T12; mock path writes via `ctx.runMutation` (same path as production).
- **`ctx.db` is forbidden inside `internalAction` handlers.** All database reads must go through `ctx.runQuery(internal.jointChat.getCoachContext, ...)` and all writes through `ctx.runMutation(internal.jointChat.writeCoachMessage, ...)` or `ctx.runMutation(internal.jointChat.recordCoachTokenUsage, ...)`. Violating this causes a Convex runtime error.
- **Streaming row lifecycle:** Insert `STREAMING` row via `ctx.runMutation(internal.jointChat.writeCoachMessage, { status: 'STREAMING', content: '' })` before the stream loop; patch content on each chunk; set `status: 'COMPLETE'` on finish. On any unhandled error, the action must still attempt to flip the row to a terminal state (COMPLETE with fallback text) so the UI is never stuck on a spinner.
- `compressIfNeeded` requires the Anthropic client as its third argument — create `const anthropic = new Anthropic()` at the top of the action handler and pass it through.
- Privacy filter retry: `filterResponse` returns `FilterResult = { passed: true } | { passed: false; matchedSubstring: string }` (T5). Check `result.passed`; if false, retry generation (not just re-filtering). Maximum 3 total attempts.

## Design Practices

- **Two-stage pipeline as cost-control:** The Haiku-gate-first design means the majority of messages (classified `NORMAL_EXCHANGE`) never reach Sonnet, keeping AI costs proportional to meaningful facilitation moments rather than message volume. This is the primary cost-control mechanism for the joint session.
- **Non-streaming Haiku, streaming Sonnet:** Haiku is non-streaming (`max_tokens: 50`) because the classification output is a single token — streaming overhead is not justified. Sonnet is streaming to give the UI a real-time typing effect, which is central to the "live coach" UX. These are different decisions made for different reasons.
- **`internalAction` + `internalMutation` split for DB writes:** Convex enforces that actions cannot call `ctx.db` directly. The `writeCoachMessage` and `recordCoachTokenUsage` internalMutations are not boilerplate — they are the correct Convex pattern for an action that needs to persist state. Each mutation call is a serializable transaction, so partial-write scenarios (e.g., action crashes after inserting STREAMING but before COMPLETE) can be detected and cleaned up.
- **`scheduler.runAfter(0)` decouples mutation from action:** `sendUserMessage` commits the user row to the database before the action starts, so the user message is visible in the UI immediately (no waiting for AI). The action is enqueued with `runAfter(0)` and runs in a separate execution context; mutation latency and AI latency are never added together from the user's perspective.
